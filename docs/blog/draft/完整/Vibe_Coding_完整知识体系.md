## 第一章 AI 使用的底层认知：打破信息茧房

### 1.1 单一长对话的隐患

很多人用 AI 的方式是这样的：开一个对话，把所有问题一股脑丢进去，方案让它写，审阅让它做，优化还是让它来。

你会不会觉得，越聊下来，它说的每一句话，都是你想听的话。一个对话框从头聊到尾，上下文越长，AI 就越懂你，效率越高。

但问题在于——AI 的底层逻辑是基于上下文生成最 **"连贯"** 的回答，是最连贯的，不是最正确的。这意味着当你在一个对话里聊了 20 轮之后，AI 已经完全 "锁定" 了你的思路框架。

你问它"你觉得这个方案有没有问题"，它大概率会说"整体不错，有几个小地方可以优化"。因为这个方案就是它在前面帮你一步步搭出来的。你让它自己否定自己？它做不到——不是技术上做不到，是概率上它不会这么做，因为否定前文会导致"不连贯"。

所以我们其实是在跟一面镜子对话。你说什么，它就反射什么，只不过反射得更有条理。

打个比方：这就像你创业，从市场调研、商业计划、财务模型到融资，全部都跟一个人商量。这个人还特别听话，你说什么他都觉得有道理。你觉得这个人是你的军师，其实他是你的狗腿子。

### 1.2 正确使用 AI 的三步核心方法

核心思路：不要试图在一个 AI 对话里解决所有问题，面对一个问题，要多维度、交叉核验。

**第一步：分角色、分场景开独立对话**

同一个问题，至少开启两到三个独立对话窗口，也可切换不同 AI 工具，分配不同角色：

1. **支持者**：全力推演方案优势、利好、可行点，助推方案落地
2. **批评者**：专门挑漏洞、风险、短板、致命问题
3. **小白用户**：站外行视角，提基础疑问、理解门槛、实际使用痛点

**第二步：找差异，不找共识**

汇总三份对话输出内容，重点抓取三方说法矛盾、对立的地方。支持者认可的优势，恰恰是批评者判定的致命缺陷——这类冲突点才是需要深度思考推敲的核心。三方达成共识的内容无需过多纠结。

常见矛盾对照：

| 支持者视角 | 批评者视角 |
|-----------|-----------|
| 用户体验好 | 成本太高 |
| 市场前景大 | 用户需求存疑 |
| 技术可行 | 学习成本高 |
| 传播效果强 | 信任感不足 |

**第三步：由人来做最终判断**

梳理全部矛盾差异点，结合三点核心信息自主拍板决策：

1. 自身对业务的深度理解
2. 对真实用户的认知了解
3. 对行业市场趋势的判断

**本质定位：AI 是素材生成器，绝非决策者。** AI 能帮你在 5 分钟内生成 10 个方案、20 个角度、30 个论据，但它不是决策者。让 AI 帮你把信息摊开、把角度打散、把矛盾暴露出来，然后你站在这些矛盾的中间做最终的判断。

处理同一个复杂问题的时候，AI 对话越长，思维就会越窄。不是因为 AI 变笨了，是因为它太懂你了——而一个太懂你的人，不会让你听到真正需要的信息。

## 第二章 AI 协作方法论：先对齐，再协作

> 作者：郑大可

### 2.1 传统 AI 协作模式的致命缺陷

大众通用的低效交互流程：

```
我有问题 → 我直接提问 AI → AI 给出回答
```

在复杂研发场景下延伸为：

1. 直接把完整需求丢给 AI，AI 一次性输出成品
2. 人工审阅内容，逐处纠错、反复迭代修改
3. 对话上下文越长，修改越混乱，返工量暴增

**问题根源：** 双方信息认知错位——使用者以为 AI 完全理解需求，AI 以为使用者表述清晰。二者在错误方向持续推进，等到发现产出偏离目标时，沉没成本已经堆积很高。更严重的是，很多时候使用者自身都没有完全理清整件事的完整逻辑，直接让 AI 输出，等同于把自己未思考完善的工作全权交由 AI 兜底。

### 2.2 优化协作微习惯：先建立全局视野，确认后再开工

#### 前置第一步：生成项目指导文档

**标准启动提示词模板：**

```
我要用你帮我完成X（填入具体任务），最终目标是Y（填入最终达成效果）
在正式开工之前，先输出这件事的完整整体路径，梳理任务全程潜在风险问题、核心关键节点，
整理一份标准化项目指导MD文档。
我们先针对这份文档逐条补充细节，等我确认无误之后，再正式开始落地执行。
```

**项目指导文档必备 4 大模块：**

- ✅ 整体执行路径
- ⚠️ 过程潜在问题与风险
- ○ 任务关键时间/步骤节点
- ➡️ 分步骤行动建议

**交互逻辑：** 先和 AI 对齐全局文档 → 双向补充细节、确认共识 → 确认完成才进入正式内容产出。

#### 第二步：多模型交叉验证纠错

1. 将 AI 生成的项目指导 MD 文档分发至多款不同大模型（常用：Claude、ChatGPT、DeepSeek，也可使用 AI 聚合平台）
2. 统一指令：让模型全面审阅文档，挑剔排查所有会影响最终产出质量的漏洞、逻辑缺陷、缺失项
3. 汇总所有模型反馈，整理问题清单 + 配套优化建议，回头修正原始指导文档

> 交叉验证适配绝大多数 AI 协作场景，通用性极强，推荐所有人使用。

### 2.3 「先对齐、再协作」模式带来的改变

1. **大幅减少反复返工**：全局指导文档划定方向，AI 不会跑偏
2. **协作流程标准化**：需求梳理 → 方案设计 → 落地实现 → 测试验收，全程有锚点把控
3. **适用范围广泛**：开发、写作、策划、设计等项目类工作，以及知识学习场景

### 2.4 实操落地建议

立刻拿当下正在处理的任意一件事，新建 AI 对话窗口，套用这套先出全局文档、确认再执行的流程，对比传统提问方式的效率差异。

> **核心理念：** 不存在速成暴富套路。这套方法本质是学会正确驾驭 AI 工具，放大自身综合做事能力。

## 第三章 Dan Koe 学习法与行动方法论

> Dan Koe 的方法论核心可以概括为一条主线：**定义目标 → 规划项目 → 直接动手 → 边做边学 → 公开输出**。

### 3.1 AI 提示工程：新时代的基础技能

在进入具体方法之前，Dan Koe 强调提示工程是当下最重要的基础设施能力。

1. **提示工程的定位转变**：几年前谷歌搜索是一项生存技能；如今 AI 能输出更精准匹配的信息，提示工程成为新的必备技能。
2. **对 AI 的定位**：不要把 AI 当成单纯替你干活的机器，而是 **创造性陪练伙伴**。简单短句提示可解决基础问题；创意类工作里，AI 输出质量完全由你的提问、认知水平决定。
3. **实操举例（Photoshop 去背景）**：
   - **第一步**：直接向 AI 提问精准问题——"如何在 Photoshop 中去除照片的背景"，获取分步操作步骤
   - **第二步**：延伸追问不懂的专业名词，比如"选区是什么""蒙版是什么"，边实操边即时学习
4. **工具辅助**：Kortex 桌面软件可按 Alt/Option+C 唤起悬浮 AI 聊天窗口，不用切换网页、离开当前操作界面随时问询 AI。

### 3.2 前置核心步骤：为自己的学习设立专属目标

#### 目标逻辑

目标由 **自我定义**，而非社会、他人分配。能清晰感知自我进度，是获得生活愉悦感的关键。

#### 落地填写「生活重置清单」

分两大板块逐条书写，每一项至少罗列 10 个要点，来回对照梳理：

- ① **生活里不想要的事物、状态**
- ② **生活里想要的事物、状态**

这份清单就是你的 **学习过滤器**，筛选所有值得投入时间的内容。不写下来的弊端：大脑容易分心混乱，缺少识别机会、匹配目标的参考框架，自动忽略能帮你达成目标的机会与知识。

#### AI 长期对话日记用法

- 把整理好的清单背景输入 AI，让 AI 给出直白、客观、不留情面的真实建议
- 保存该对话窗口，后续遇到人生抉择、困惑时持续在此沟通，当作长期成长日记迭代

### 3.3 规划一个落地项目

#### 核心学习理念

学习效率最高的模式：**先定真实项目，遇到缺口再针对性查资料**。你的吸收程度和项目推进进度强绑定。

- **反面误区**：无脑刷海量教程。海量信息会给大脑制造噪音、混乱，绝大多数内容无效浪费，只会催生焦虑、迷茫，大幅拖慢学习速度。
- **核心结论**：想学得更快，直接跳过通篇刷教程的环节，优先规划项目。

#### "项目"范围无限制

任何目标都可作为项目载体：自我健康管理计划、个人小生意/自媒体业务、PS 修图/绘画单张作品、编程小程序、短视频、写作书稿等。

#### 完整启动项目步骤

1. **筛选项目方向**：这件事必须能推动你靠近前面清单里「想要的生活」
2. **新建笔记/文档**，自由头脑风暴所有相关想法
3. **收集 3–5 个对标灵感范本**（例：做 YouTube 视频就找 3–5 个同赛道优质对标账号视频）
4. **拆解对标范本**的结构、亮点、核心特征
5. **拆分项目**：划分阶段、里程碑、素材需求、待学习知识点
6. **搭建想法收纳库**：固定一处记录零散灵感，避免遗忘丢失

> **关键提醒：** 项目规划完毕后，不要立刻开始批量预习学习内容。直接动手。

### 3.4 克服启动障碍：蔡加尼克效应

#### 蔡加尼克效应

**蔡加尼克效应**是一种心理现象：人们更容易记住 **未完成** 的任务，而不是已完成的任务。这意味着，如果我们没有完成项目的任务，就更容易找到开始的动力。

#### 蔡加尼克²效应：启动仪式

从 Justin Sung 那里学到的进阶技巧。在开始之前做一些简单的任务，以模拟有未完成任务的感觉。这可以是一个工作仪式，比如煮咖啡、清理桌子，或者早上去散步并在手机上记笔记。

#### 最小启动法

把你需要开始的任务分解成你能采取的 **最简单的可能行动**。例如对于写作：坐在你的桌子前，打开你的大纲，简单地通读一遍。这只需要很少的动力。

### 3.5 从你当下已知的内容直接动手

#### 底层认知

**学习源于实操挣扎，而非被动死记硬背。**

启动方式：直接上手做项目，杜绝提前刷几十套教程铺垫。动手过程会直白暴露你的知识盲区，此时主动解决缺口，大脑记忆留存效果最强。

**零基础兜底方案：** 哪怕完全不懂，也要踏出第一步——下载软件、点开工具随便创作一点内容。先让大脑产生求知渴望，不然就算搜到教程、知识点，也很难消化吸收。

#### 标准循环执行流程

1. 动手后发现自己不知道怎么操作
2. 自行尝试、实操试错、经历失败
3. 搜索资料 / 向 AI 提问获取解法
4. 把答案落地实操验证
5. 循环重复，直到项目整体完成
6. 自行搜索无法解决难题时，直接请教行业专家

### 3.6 创建一个学习计划

在有了参考框架、项目大纲，并且已经开始取得进展之后，就可以开始用一般性学习和教程来补充了。

创建一个新的聊天，试试这个提示：
> 对于 **[主题]**，你可以输入任何东西，从 Photoshop 到讲故事到内容创作。

**3 种时间块：**

| 时间块 | 时长 | 内容 |
|--------|------|------|
| **构建** | 30–90 分钟 | 构建你的项目，并在需要时搜索信息 |
| **学习** | 30–60 分钟 | 遵循你的学习计划，并记下你学到的东西。可通过要求 AI 总结笔记来复习 |
| **散步** | 30 分钟 | 听 YouTube 视频、有声书或讲座，在想法出现时用手机记下来 |

这些是你的 **关键习惯**。没有人会给你时间去学习和构建，你必须自己安排。

### 3.7 通过写作系统地反思所学

#### 为什么需要公开输出？

Dan Koe 提出一个逻辑链条：

1. 你学习和构建，最终想把它变成可持续的生活
2. 要持续，你需要钱
3. 要赚钱，你需要人们关心你所构建的东西
4. 要让人关心，你需要 **公开展示** 你正在做的事情
5. 最佳方式就是 **写作**——在你构建的过程中，教授你所知道的和正在学习的东西

**为什么是写作？**
- **媒体的基础**：写作是一切内容形式的根基
- **低门槛**：任何人都可以立即开始，不需要视频编辑技能
- **力量更大**：比单纯建立粉丝受众拥有更大的影响力

#### 费曼技巧

写作是你系统地反思所学内容的方式。当你教授你学到的东西时，你会暴露更多的知识空白。

**费曼技巧**核心步骤：
1. **选择一个概念**——选择你想理解的主题
2. **教授它**——用简单的语言解释这个概念，就像你在教一个孩子
3. **识别空白**——当你难以清晰地解释某事时，找出你理解薄弱的领域
4. **复习和简化**——回到原始材料，重新学习概念，然后尝试用更简单的术语再次解释

#### 导师效应

**老师比学生学得更多。** 教授你学到的东西鼓励你以自己的方式理解经验，暴露更多知识空白，增加模式识别的效果。

### 3.8 综合落地执行

**通过在互联网上写作。** 把社交媒体当作你的 **公开日记**。

推荐落地步骤：
- **每周写一份通讯**，总结你学到的东西——是要 **教给他们**，而不是堆砌信息
- **在 X、Threads 或 LinkedIn 上发帖**（优先选易上手的写作平台）
- **谈论你的观点、信念、个人经历**，以及你正在学习和构建的东西
- 将"社交媒体"作为一项需要学习的技能
- **每天早上增加一个 30–60 分钟的写作时间块**
- **善用 AI 辅助写作**，例如向 AI 提问："你是一位制作吸引人的社交帖子的专家。关于 **[主题]** 的帖子最好的开头方式是什么？"

> 本篇解决 **"如何高效地用 AI 编程"** 的问题。从 10 条实战准则到一线实践者的真实反思，提供可直接落地的操作指南。

## 第四章 AI 编程高效协作 10 条实战准则

### 准则 ① 先调研再动手

别急着让 AI 写代码。第一句话永远是：先帮我调研全网竞品，生成一份傻子都能看懂的报告。磨刀不误砍柴工，方向错了，写得再快也是废代码。

### 准则 ② 吃透需求，方案过审才开发

让 AI 先完整分析你的资料，确认理解了再动手。先出方案，用 Mermaid 图画出来，你确认了才开始写。AI 最大的问题不是写得慢，是写得快但方向跑偏。

### 准则 ③ 不确定就问，绝不瞎猜

告诉 AI：你有任何不确定的地方，必须提问找我确认，不要自己瞎猜。一个 5 秒钟的确认，省得你 5 小时的 debug。

### 准则 ④ 查最新文档，别用过时写法

AI 的训练数据有截止日期。让它用联网搜索获取最新技术文档，别用过时的写法。AI 给的 API 调用方式可能是废弃版本，跑起来全是报错。

### 准则 ⑤ 简单优先，只加不改

先用最简单直接的方式实现核心功能，避免过度设计。一定不要影响现有功能，只新增代码，遵循开闭原则。能跑通的烂代码 > 跑不通的好架构。

### 准则 ⑥ 把我当新手，保姆级引导

第一次做某个功能？告诉 AI：把我当傻子，给我充分的操作引导和步骤说明。别装懂，AI 不会笑话你，但上线后的 bug 会。

### 准则 ⑦ 前端要独特，别千篇一律

禁止蓝紫渐变！禁止模板感！让 AI 做出让人眼前一亮的页面，不是又一个 SaaS 后台。用户第一眼看的是界面，不是你的代码架构。

### 准则 ⑧ 多需求并行，规划优先级

同时给了多个需求？让 AI 合理规划优先级和依赖关系，能并行的并行，用最快速度完成，注意代码别冲突。

### 准则 ⑨ 自测自修，跑通再交付

让 AI 自主写测试用例、执行验证、出了问题自己修复。先测通了再找你验收。你验收的应该是成品，不是半成品。

### 准则 ⑩ 文档同步，Git 规范，封装复用

需求分析、方案设计沉淀为 Markdown，每次改完代码同步更新文档。提交 Git 写清楚 commit message。好的能力封装成 Skill 复用。今天偷的懒，明天都是技术债。

### 十条准则的核心本质

10 条的本质就一句话：**给 AI 搭好脚手架。** 先想再做、遇疑则问、测通再交、沉淀复用。

这在行业里有个专业叫法——**Harness Engineering（脚手架工程）**。不是 AI 不行，是你没给它搭好框架。

## 第六章 Vibe Coding 立项：选择技术栈

如果你让 AI 上来就直接写代码，那这个项目大概率已经开始跑偏了。刚开始用 AI 做项目的人，最缺的往往不是代码能力，而是不知道该让 AI 按什么技术路线、什么框架规范去写。

### 6.1 先确定产品形态

问自己一个问题：你的产品最终是以什么形式呈现给用户的？

- **网站** → Web 前端技术栈
- **小程序** → 小程序平台开发规范
- **App** → 原生开发或跨平台方案
- **纯后端接口** → 接口设计、数据库和服务部署为核心

如果你不确定，直接让 AI 根据立项文档来分析，让它结合产品目标、用户场景、核心功能和后期规划，判断最适合的产品形态并说明理由。

> 对没有代码基础的人来说，最怕的不是没有选择，而是选择太多。你需要的是一个 **当前最适合执行的方向**。

### 6.2 技术栈不是越高级越好

很多新手容易被各种技术名词带偏。但做项目不是比谁用的技术更新，尤其是 Vibe Coding。

**技术栈最重要的不是炫，而是适合。** 它要适合你的产品形态，适合你的项目复杂度，适合 AI 理解和维护，也适合你后期继续迭代。选择技术栈时一定要 **克制**。

### 6.3 优先选择成熟框架和 SDK

成熟轮子不只是帮你更快把产品做出来，更重要的是能帮你省掉大量试错、重写和踩坑的时间。

成熟框架的价值不只是它提供了现成能力，还在于它通常有自己的项目结构、开发规范、最佳实践和社区经验。**它不仅告诉你能怎么写，也告诉你应该怎么写。**

你可以直接告诉 AI：**"这个项目必须严格遵守当前框架的官方规范，不允许为了图省事绕开框架设计。"** 仅仅这一句话，就能在一定程度上减少 **AI 漂移**。

### 6.4 判断技术是否靠谱

让 AI 帮你查几个关键点：

- 这个技术是不是有足够多的人在用？
- 项目是否还在维护？
- 文档是否完整？
- 社区里有没有大量讨论？
- 最近更新时间是不是太久远？
- 授权协议是否允许商业使用？

如果是开源项目，可以让 AI 帮你查看 GitHub 的 Star 数量、活跃度、最近提交时间、发布记录和使用案例。**授权协议**不能忽略——有些开源项目对商业使用有限制。

### 6.5 让 AI 给你唯一推荐方案

不要只问"这个项目用什么技术栈"。你应该把立项文档、产品形态、功能范围、目标用户、部署要求和后期规划交给 AI，然后让它基于这些内容，给出当前项目的 **唯一推荐技术栈**，并说明：

- 为什么选它？
- 为什么其他方案不适合？
- 后期可能会遇到哪些风险？

> **用 AI 做项目，最怕技术路线不稳。** 今天用这个，明天换那个，后天又觉得另一个更高级——那样写到最后，项目很容易变成一堆拼起来的代码。

### 6.6 技术栈要写进项目文档

技术栈讨论完之后，一定要写进项目文档里。里面至少要说明：

- 产品形态是什么？
- 前端用什么？后端用什么？数据库用什么？
- 主要框架和 SDK 是什么？
- 为什么这么选？哪些技术不用？
- 什么情况下才需要重新评估技术路线？

这份文档后面会成为 AI 开发项目时的重要依据。每次 AI 想引入新框架、新库、新架构时，都应该先回到这份文档里看一眼。

### 6.7 选择技术栈的本质

选择技术栈本质上是在给项目选地基。在正式写代码之前，你至少要先确认三件事：

1. **产品形态**是什么？
2. **技术栈**是什么？
3. **框架规范**是什么？

产品形态决定项目往哪里走，技术栈决定项目用什么工具做，框架规范决定 AI 不能乱写。

## 第七章 Vibe Coding 前端骨架搭建

如果你用 AI 写出来的前端项目，每个页面都像一个独立产品，那大概率不是 AI 不会写页面，而是你一开始没有做好前端骨架。

### 7.1 先确定整体设计风格

设计风格不是某一个页面好不好看，而是整个 UI 呈现出来的统一视觉语言。

常见风格参考：
- **玻璃拟态/液态玻璃**：半透明、模糊背景
- **Material Design**：层级清楚，组件规范完整
- **企业级管理台风格**：信息密度高，表格多，操作效率优先
- **现代极简风格**：卡片清楚，按钮克制，留白适中
- **科技感风格**：深色背景、渐变发光元素、动效更多
- **内容社区风格**：阅读舒服，内容层级清楚

如果你不知道自己想要什么风格，可以找几个喜欢的网站把链接发给 AI；如果是 App，就找几张参考截图发给 AI。让 AI 分析这些界面的视觉风格、布局方式、组件特点，以及这种风格适不适合你的项目。

### 7.2 确定前端技术方案和 UI 组件库

两个东西不要混在一起：

- **前端技术方案**解决项目怎么组织、怎么运行：页面怎么放、路由怎么写、目录怎么分、项目怎么启动、构建和部署。常见方案有 React、Vue、Angular、Next.js、Nuxt.js、Taro、uni-app 等。
- **UI 组件库**解决界面元素怎么统一：按钮、输入框、弹窗、表格、菜单、卡片等。常见组件库有 Ant Design、Element Plus、Naive UI、MUI、Radix UI、Shadcn UI 等。

> **一句话：前端框架管项目结构，UI 组件库管界面统一。**

如果你不懂技术，主要靠 AI 写前端，强烈建议优先选成熟方案。你可以直接告诉 AI：**"这个项目必须遵守当前前端框架的官方规范，优先复用选定的 UI 组件库，不允许到处手写重复组件，也不允许绕开框架最佳实践。"** 这句话就是在给 AI 画边界。

### 7.3 目录、模块和组件

**第一，目录边界要清楚。** 页面放哪里、组件放哪里、请求接口放哪里、工具函数放哪里、样式文件放哪里，都要有规则。

**第二，模块边界要清楚。** 登录、用户、订单、内容、设置这些功能区域不要全部混在一个地方。

**第三，组件复用规则要清楚。** 按钮、输入框、卡片、表格、弹窗会反复出现。能用 UI 组件库就优先用；组件库不够用，再基于组件库封装项目自己的业务组件。

简单判断标准：只要 UI 相似度比较高，并且在项目里出现超过两次，就应该考虑封装成组件。你可以把这条写进 agent 的宪法里：**"同类 UI 结构出现超过两次时，必须优先抽象为可复用组件，不允许复制、粘贴、重复实现。"**

### 7.4 提前准备样式系统

**第一，设计 Token。** 颜色、字号、行高、间距、圆角、阴影、边框，这些东西必须在写代码前定好。告诉 AI：所有样式都必须从统一设计 Token 里取，不允许到处硬编码。

**第二，多语言规范。** 如果项目未来可能支持中英文，前端文案就不要到处硬编码。中文两个字能放下，英文可能变成一整句。如果一开始不考虑，后面切语言时，很容易出现按钮撑爆、标题换行、布局错位。

**第三，主题规则。** 先确定一个默认主色调，再预留主题扩展方式。主题色不能散落在每个页面里手写，否则后面想换主色调就要全项目到处改。

### 7.5 写进实施计划，再让 AI 搭骨架

实施计划里至少要写清楚：

- 整体设计风格是什么？
- 前端技术方案是什么？
- UI 组件库用什么？
- 目录结构怎么组织？
- 模块边界怎么划分？
- 哪些组件先封装？
- 设计 Token 怎么定义？
- 多语言和主题规则怎么预留？
- 第一阶段只做哪些内容？
- 验收标准是什么？

这一轮目标只有一个：**项目能跑起来，基础结构是对的。** 项目能启动、首页能打开、路由结构清楚、目录规范正确、UI 组件库接入完成、设计 Token 准备好——这就够了。

前端骨架阶段重点不是做多少页面，而是让后面的页面都基于同一套规则来构建。骨架跑通之后，先提交一次 Git，然后再一个模块一个模块往里加功能。

### 7.6 统一规则才是核心

做前端最怕的不是页面丑，最怕的是没有统一规则。

- 没有提前定设计风格 → AI 容易把页面写得不像一个整体
- 没有目录规范 → AI 容易乱建文件夹
- 没有组件规范 → AI 容易到处手写按钮、弹窗、表格
- 没有组件复用规则 → AI 容易复制一堆相似代码
- 没有设计 Token → AI 容易到处硬编码样式

所以，让 AI 写前端之前先打骨架：先定设计风格，再选技术方案和 UI 组件库，然后定目录、模块、组件复用、设计 Token、多语言和主题规则，接着把这些内容写进实施计划，最后再让 AI 搭一个能跑起来的前端基础结构。

## 第八章 Vibe Coding 项目崩溃根源与前置性能容量规划

### 8.1 问题总览

#### 为什么 AI 搓出来的 Demo 一放量就崩？

1. **Vibe Coding 天然运行环境局限**：AI 快速编码默认适配单人、网络无损耗、极小数据量的理想调试场景。完全没有带入多用户真实生产环境的负载压力，开发者容易误以为 Demo 流畅 = 上线稳定。

2. **误区：归咎 AI 编码能力，实则缺失容量规划**：崩服务的核心原因不是 AI 写的代码质量差，而是开发全程跳过了 **容量规划（Capacity Planning）** 与前置性能设计。真实产品需要把模糊的"很多人用"翻译成可落地的数字指标、架构约束、容错方案。

### 8.2 容量规划 6 大核心难题

#### 难题 1：并发与峰值——同时有多少人在用

**概念区分：**
- **总量用户**：平台全部注册/访问用户总数（总量大 ≠ 并发高）
- **并发**：同一时刻、同一功能发起请求的活跃人数
- **峰值**：流量冲高瞬间的最大并发量（压垮服务的致命点）

**AI 场景典型风险：** AI 生成类任务耗时远高于普通接口（如头像生成单次需 10s），少量峰值并发就会占满模型算力、形成请求排队雪崩。

#### 难题 2：流量分布——流量倾斜到哪个功能

**行业通用规律：二八流量法则。** 20% 高频核心功能承载 80% 整体访问流量，其余 80% 功能只有少量低频访问。资源不能平均分配，必须优先倾斜核心接口。

#### 难题 3：数据规模——单次大小 & 长期总量

拆两层维度评估：

1. **单次请求数据体量**：传输/处理文件大小直接决定带宽、内存、IO 压力
2. **长期数据总量膨胀**：小数据量下数据库查询丝滑，数据量级暴涨后查询延迟会显著升高
3. **AI 独有隐藏体积**：模型文件、运行镜像、依赖代码包本身占用空间，直接影响容器冷启动速度

#### 难题 4：基准任务时长——核心操作合格响应时间

- 普通检索类操作：0.5 秒内为流畅体验，超过 5 秒用户耐心完全耗尽
- **AI 产品特殊指标：首字时间（TTFT）**——多久吐出第一个文字比整体生成总时长更影响体感

#### 难题 5：错误率——高压下失败比例与兜底方案

必须回答两个核心问题：
1. 峰值高压场景下，可容忍的请求失败比例（行业常规参考：≤5% 为健康区间）
2. 请求失败后的用户体验兜底逻辑，禁止无限加载转圈

#### 难题 6：极限保护——系统触顶后的阈值与兜底策略

4 种行业成熟保护手段：

1. **限流**：拦截超额请求，优先保障一部分用户正常使用
2. **降级**：负载过高时临时关闭非核心功能，死守基础核心功能可用
3. **排队**：请求积压时展示清晰排队序号、预估等待时长
4. **缓存**：高频重复请求结果持久缓存，避免重复调用大模型

> **核心原则：** 系统可以变慢、部分功能临时下线，但整体服务不能彻底宕机。

### 8.3 可直接投喂 AI 的标准化设计模板

> 使用规则：AI 写业务代码前，先填充以下所有空白项，强制 AI 基于真实业务指标输出带性能容错的工程代码。

```plaintext
在写代码之前，请基于以下场景做完整性能架构设计：
1. 并发与峰值：
   预计同时在线___人使用，业务峰值最高可达___人
2. 业务流量占比：
   使用率最高的核心功能是___，优先保障该接口性能与资源配额
3. 数据规模预估：
   单次请求处理数据体量约___；业务长期数据总量预估增长至___
4. 基准响应时长：
   核心操作___整体响应需控制在___秒内；AI流式首字输出时间(TTFT)≤___秒
5. 错误处理机制：
   当___（外部API超时/网络波动/模型过载等）发生请求失败时，执行兜底逻辑：___
6. 极限过载保护策略：
   系统负载超出承载上限时，启用 [限流/降级/排队/缓存] 中的___组合方案
```

### 8.4 补充落地认知

- **边界区分**：Vibe Coding 适合快速产出原型 Demo，但正式上线只靠 AI 编码不足以支撑稳定运行。容量规划覆盖代码、存储、服务器算力、运维监控全链路。
- **性能设计本质**：并非高深晦涩的底层工程黑盒，而是一套严谨、量化、前置化的产品开发思维。先量化压力指标，再匹配技术方案，从根源避免上线翻车。

## 第九章 AI 设计能力六大阶段

吃透 Token 机制、Latent Space 隐空间、Diffusion 扩散模型底层逻辑。明白 Sampling 采样算法、CFG Scale 引导权重、Resolution 分辨率、Aspect Ratio 画幅比例的实际影响。

### 第四阶段：设计功底 & 跨岗位技术协同

精通 PS 图层、Alpha 通道、PNG 透明底、CMYK/RGB 色域、Vector 矢量图、Raster 位图、File Format 文件格式基础技能。

**实操要点：** 多模型搭配使用，实现批量生图、风格统一复刻、细节瑕疵修复、批量迭代优化；快速产出定稿商用图；把控出图效率与画质精度，做到可复用、可量产、可落地交付。

## 第十章 AI Agent 工程师成长指南

### 10.1 背景与现状

"我是后端开发，想成为 AI Agent 工程师，公司内部有转岗机会。6 年经验，卷后端有点卷不过别人，面 P7 不够，面 P6 被挑战。"

6 年经验后端 → 转 AI Agent 是正确方向。现在懂 System Design 又懂 AI 应用落地的工程师（AI Engineer）确实是稀缺物种。后端转 Agent 优势非常大——很多算法工程师写出来的工程代码没法看。Agent 落地到最后，拼的是工程能力、系统稳定性、并发处理、数据流转，这些都是后端的看家本领。

> **警惕**：千万别信网上那些三天速成课。那些教程大部分是拿个 LangChain 的 Demo 跑一下，连个数据库都不挂，跑通了 Hello World 就告诉你学会了。真到了生产环境，这种代码上线就是事故。

### 10.2 打破认知误区：Agent 并非简单调 API

很多后端兄弟觉得，Agent 不就是调 API 吗？

这是初级阶段。现在的 Agent 工程，本质上是在写 **概率性代码**。以前写后端，输入 A 必然得到 B，这是确定性的。现在写 Agent，输入 A，可能得到 B，也可能得到 C，还可能给你胡说八道。你要做的是用工程手段，把这个概率性的过程约束在一个可控的范围内。

### 10.3 需要恶补的内容

#### （一）Prompt Engineering 进阶学习

- **推荐资料**：OpenAI 官方 Cookbook（最好教材）
- **重点内容**：Chain of Thought 思维链、ReAct 推理加行动模式
- **必读博客**：Lilian Weng《LLM Powered Autonomous Agents》——Agent 领域圣经，把 Agent 拆解为规划、记忆、工具使用三个核心组件

#### （二）开源模型与私有化部署

2026 年的标准答案是 **DeepSeek 系列**和 **Qwen 3 系列**：

- **DeepSeek 系列**：DeepSeek-V3.2 和 R1 版本，推理能力和 Agentic 任务处理能力已能和闭源顶尖模型掰手腕，性价比极高
- **Qwen 3 系列**：企业内部超一半私有化部署在用 Qwen 3，特别是 Qwen3-Thinking 版本

动手去搞：下载 Ollama，在本地把 DeepSeek-V3.2 的量化版跑起来，感受一下模型如何理解复杂指令。这种体感非常重要。

#### （三）RAG 技术深度掌握

面试必考题。2026 年的主流是 **GraphRAG**（结合知识图谱的检索增强生成），需要了解怎么把非结构化文本提取成实体和关系，存到图数据库里。

除了 GraphRAG，还要了解 **Adaptive RAG**——让 Agent 自己判断是直接回答、查数据库还是查互联网。

**框架选择：**
- **Haystack**：生产级 RAG 首选，Pipeline 设计清晰，监控和可视化做得好
- **LlamaIndex**：在复杂索引和 GraphRAG 支持上最强
- 别死磕 LangChain，封装太厚，调试像黑盒

#### （四）LangGraph 与 Agent 状态机实现

Agent 本质上就是一个有状态的循环系统：

```
Start → 思考 → 决定调用工具 → 执行工具 → 拿到结果 → 再次思考 → End
```

自学时不要光看，要动手写：
1. 不要用 LangChain，尝试用原生 Python 代码手写一个 ReAct 模式的 Agent
2. 定义一个 While 循环，在循环里维护一个 Messages 列表模拟内存
3. 定义几个 Python 函数作为工具，传给模型的 Function Calling 接口

当你亲手把这个循环跑通，看到模型自己决定调用你的函数，拿到结果后再组织语言回答的时候，你就入门了。

### 10.4 实战项目：打造 P7 水平的 SQL Agent

推荐实战方向：做一个 **SQL Agent**——业务运营人员不懂 SQL，直接问机器人"上周北京地区的销售额是多少？"

**难点：**
1. 数据库表结构怎么告诉模型？→ Schema Linking
2. 模型生成的 SQL 要是错的怎么办？→ Self-Correction
3. 怎么防止模型执行删库跑路的操作？→ 权限控制

**技术组合：** RAG 检索表结构说明 + Few-shot 优秀 SQL 示例 + Code Execution Environment 安全执行。

2026 年还可以加 Multi-Agent：设计 Writer（写 SQL）+ Reviewer（检查语法错误）两个 Agent 互相交互，直到产出正确的 SQL。多智能体协作是现在最火的方向。

### 10.5 工程化能力：Agent 上线关键问题

面试官问：Agent 上线了，响应太慢怎么办？幻觉严重怎么办？

从工程角度回答：

1. **流式输出 Streaming**：不用等整个答案生成完，一个字一个字往回吐
2. **缓存 Caching**：相似问题用语义缓存返回结果
3. **评测 Evaluation（重点中的重点）**：
   - 用更强模型给弱模型打分 → LLM-as-a-Judge
   - Ragas 框架评测 RAG 系统
   - Langfuse / Arize Phoenix——主流 Trace 和评测平台

如果面试时能拿出一套自己设计的自动化评测方案，证明你的 Agent 在准确率上提升了 10%，这比说精通 Python 有用一万倍。

### 10.6 学习路径与资源汇总

1. **理论基础**：Lilian Weng 博客、OpenAI Cookbook；关注 Andrej Karpathy 动态
2. **框架工具**：LlamaIndex（GraphRAG）、Haystack（生产 Pipeline）、LangGraph（状态机）
3. **模型知识**：Hugging Face、Ollama，重点研究 DeepSeek 和 Qwen 3 系列
4. **实战练习**：DeepLearning.AI 课程
5. **前沿资讯**：Arxiv 论文，关注 Self-Correction、Multi-Agent 协作方向

### 10.7 心态调整与职业发展

当你把 GraphRAG 的各种策略研究透了，当你把 Agent 的规划模式手写过一遍了，当你能聊出如何解决幻觉、如何做自动化评测这些实际问题时，你就是专家。

Agent 这个领域太新了，2026 年正是从 Demo 走向大规模生产的元年。你只要能展示出有系统的思考、有工程落地的能力，这就够了。

**不要怕卷。** 后端的卷是存量竞争，比谁的 CRUD 写得更花哨。Agent 的卷是增量竞争，大家都在抢着定义未来的交互方式。6 年的后端经验是你最大的资产——结构化思维、系统稳定性、API 设计，这些都是写好 Agent 的基石。

> 先把手弄脏，去写代码，去报错，去看着模型胡说八道然后想办法修好它。别想那么多，今晚就开始——哪怕只是把 Ollama 装上，跑个 DeepSeek，跟它聊聊你的焦虑，也是第一步。
