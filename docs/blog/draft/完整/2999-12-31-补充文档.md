## 一、算法与数据结构

### 海量 QQ 去重

**【腾讯】面试题**
40 亿个 QQ 号如何去重？仅 1GB 内存。

**解析**：一个整数占 4 个字节，那么 40 亿个 QQ 号需要 16GB 内存，远远超过了 1GB 的限制。我们可以使用位图法来解决，每一位代表一个数字，也即 40 亿 ÷ 8 = 500MB，这样就可以在 1GB 内存内完成去重。

### N 的阶乘末尾有几个0？

**解析**：判断这个阶乘中会出现多少个5（包括把所有5的倍数拆解得到的5），就有多少个0。

**代码实现**：
```java
public int trailingZeroes(int n) {
    int count = 0;
    while (n > 0) {
        n = n / 5;
        count += n;
    }
    return count;
}
```

### 设计快速插入和查找的数据结构，你会用什么结构？

**解析**：

如果需要同时满足快速插入和快速查找，根据不同场景可以选择：

1. **平衡二叉搜索树（如红黑树、AVL树）**：
   - 插入和查找都是 O(log n)
   - 支持有序遍历

2. **哈希表（散列表）**：
   - 平均情况下插入和查找都是 O(1)
   - 不支持有序性，存在哈希冲突问题

3. **跳表**：
   - 插入和查找都是 O(log n)
   - 实现相对简单，支持有序遍历，在 Redis 中广泛使用

4. **B树/B+树**：
   - 适合磁盘存储，多路平衡查找树
   - 减少磁盘IO次数，数据库索引用B+树

**场景选择**：
- 内存中存储：跳表或哈希表
- 需要有序性：跳表或红黑树
- 磁盘存储：B+树

### 进程是分配资源的基本单位，那么这个资源指的是什么？

**解析**：进程是操作系统进行资源分配和调度的基本单位，这里的资源主要包括：

1. **内存空间**：每个进程有独立的地址空间
2. **文件描述符表**：打开的文件、套接字等资源
3. **CPU上下文**：程序计数器、寄存器等
4. **I/O设备**：分配的I/O设备
5. **页表**：用于虚拟地址到物理地址的映射

关键理解：进程之间的资源是相互隔离的，一个进程崩溃不会影响其他进程。线程共享进程的资源，线程是CPU调度的基本单位。

### 百万级别数据的 Excel 如何快速导入到数据库中

**【阿里】面试题**

**解析**：从 Excel 中读取百万级数据并将其插入数据库时可能遇到的问题：

#### 数据正确性问题
读取和导入过程中会有各种潜在的问题，我们需要妥善处理各类异常情况，在插入数据前，我们需要对数据进行校验，确保数据的正确性。然后在插入过程中处理异常情况，如果单条数据插入失败，我们可以二次尝试，二次尝试仍然失败的话，将数据记录到日志中，方便后续对数据进行二次插入。导入数据过程中还需要注意数据可能出现重复的情况，我们需要采取设定唯一约束的方式来处理，同时记录日志。

#### 内存溢出问题
我们不能一次性将数据读取到内存，需要分批处理数据。我们可以使用 Apache POI 的事件模型来处理 Excel，这种方式适合处理大文件，因为它不会将整个文件加载到内存中，它会按行读取数据，就大大释放了内存的压力。

#### 性能瓶颈问题
针对百万级数据的处理，单线程显然效率很低，所以我们采取多线程处理数据。我们可以将多线程分为两种场景：一种是多线程读取文件操作，另一个是多线程插入数据。当然在数据插入的时候，除了使用多线程技术外，还可以使用批量插入数据的功能更进一步提升速度，可以通过 `values(),(),()` 的方式一次性插入多条数据。

### Hive 中的排序函数

**解析**：Hive 中的排序分为四种：

- `ORDER BY`：全局排序，只有一个 Reducer
- `SORT BY`：分区内有序，不是全局排序
- `DISTRIBUTE BY`：按照指定字段分组，配合 SORT BY 使用
- `CLUSTER BY`：当分区字段和排序字段相同时，可以替代 DISTRIBUTE BY + SORT BY

**ORDER BY 和 SORT BY 的区别**：
- ORDER BY：全局排序，数据量大会很慢
- SORT BY：每个 Reducer 内排序，只保证区内有序

### left join 和 right join 的区别

**解析**：
- `LEFT JOIN`（左连接）：保留左表所有记录，右表匹配不到的字段显示 NULL
- `RIGHT JOIN`（右连接）：保留右表所有记录，左表匹配不到的字段显示 NULL

## 三、架构设计

### 为什么复杂的架构一定要做分层设计？

**【腾讯】面试题**

**解析**：分层思想在软件工程中是一种常见的设计方式，它是将整体系统拆分成 N 个层次，每个层次都有自己的职责，多个层次协同提供完整的功能。软件架构整体上会被分为三层：UI 层、业务逻辑层、数据访问层。

- **UI 层**：进行数据展示以及和用户进行交互
- **业务逻辑层**：负责业务逻辑的实现，可能会综合调度多个数据访问层
- **数据访问层**：主要和数据库进行通信

**分层架构的优点**：
1. **职能分离，隔离关注点**，每个人关注自己所在分层的任务即可
2. **解耦合，方便系统扩展**。当项目分层后，每层都仅能和自己相关的层做数据交互，不能跨层调度，这样就做到了解耦，也方便系统的扩展。我们可以将数据访问和业务逻辑层独立部署。

### 设计一个分布式缓存系统需考虑哪些点?

设计一个分布式缓存系统需要考虑以下几个方面：

1. **数据分区和一致性**
   - 如何将数据均匀分布到多个节点上，确保每个节点负载均衡
   - 采用何种一致性模型（如强一致性、最终一致性）来保证数据的一致性

2. **高可用性和容错性**
   - 如何设计系统的冗余机制，确保在部分节点故障时，系统仍能提供服务
   - 采用何种故障检测和恢复机制，确保系统的可用性

3. **数据过期和淘汰策略**
   - 如何设置数据的过期时间，确保缓存数据的时效性
   - 采用何种淘汰策略（如LRU、LFU）来管理缓存空间，确保缓存的高效利用

4. **并发控制和锁机制**
   - 如何设计并发控制机制，确保在高并发场景下，数据的正确性和一致性
   - 采用何种锁机制（如分布式锁）来避免并发访问导致的冲突

5. **性能优化**
   - 如何优化数据访问路径，减少网络延迟和IO开销，提高系统的响应速度
   - 采用何种数据结构算法，优化数据的存储和检索效率

6. **监控和管理**
   - 如何设计系统的监控机制，实时监控系统的运行状态和性能指标
   - 提供何种管理接口，方便管理员对系统进行配置、维护和故障排查

7. **安全性**
   - 如何设计系统的安全机制，确保数据的机密性和完整性
   - 采用何种认证和授权机制，控制用户对系统的访问权限

总结来说，设计一个分布式缓存系统需要综合考虑数据分区、一致性、高可用性、容错性、数据过期、淘汰策略、并发控制、性能优化、监控管理和安全性等多个方面，确保系统的高效、可靠和安全运行。

### 怎么设计一个短链地址，要考虑跨机房部署问题

设计一个短链地址系统，需要考虑以下几个方面，特别是跨机房部署的问题：

1. **短链生成算法**
   - 设计一个高效的短链生成算法，将长链接转换为短链接。常见的算法包括哈希算法、自增序列等
   - 确保生成的短链接具有唯一性，避免冲突

2. **数据存储和一致性**
   - 选择合适的数据存储方案，如关系型数据库、NOSQL数据库或分布式缓存系统，存储短链接和长链接的映射关系
   - 在跨机房部署的情况下，确保数据的一致性和高可用性。可以采用分布式数据库或数据同步机制，确保数据在多个机房之间实时同步

3. **负载均衡和流量分发**
   - 设计负载均衡策略，将用户请求均匀分发到多个机房的节点上，提高系统的整体性能和可用性
   - 可以使用DNS负载均衡、硬件负载均衡器或软件负载均衡器实现流量分发

4. **高可用性和容错性**
   - 设计系统的冗余机制，确保在部分机房或节点故障时，系统仍能提供服务
   - 采用故障检测和恢复机制，确保系统的可用性

5. **性能优化**
   - 优化短链接的解析和跳转过程，减少网络延迟和IO开销，提高系统的响应速度
   - 采用缓存机制，缓存热点短链接，减少数据库访问次数

6. **监控和管理**
   - 设计系统的监控机制，实时监控系统的运行状态和性能指标，包括短链接的生成量、访问量、机房负载等
   - 提供管理接口，方便管理员对系统进行配置、维护和故障排查

7. **安全性**
   - 设计系统的安全机制，确保短链接的机密性和完整性，防止恶意攻击
   - 采用HTTPS协议，加密用户请求和响应，保护用户数据的安全

总结来说，设计一个短链地址系统需要考虑短链生成算法、数据存储和一致性、负载均衡、高可用性、性能优化、监控管理和安全性等多个方面，特别是在跨机房部署的情况下，需要确保数据的一致性、系统的可用性和高性能。

### 分布式ID生成器怎么实现？

**解析**：常见的分布式ID生成方案有几种：

1.** UUID **
   - 优点：简单，本地生成，无网络延迟
   - 缺点：无序，占空间大，信息泄露风险

2.** 数据库自增ID **
   - 基于数据库的自增主键，不同节点设置不同步长
   - 优点：有序，实现简单
   - 缺点：依赖数据库，存在单点问题，扩容麻烦

3.** 雪花算法（Snowflake）**
   - 时间戳 + 工作机器ID + 序列号，64位
   - 优点：高性能，低延迟，有序
   - 缺点：依赖时钟同步，回拨问题

4.** Redis 生成 **
   - 利用 Redis 的 INCR 命令生成递增ID
   - 优点：性能好，有序
   - 缺点：依赖 Redis，需要持久化保证不重复

5.** 美团 Leaf **、** 百度 UIDGenerator **等开源方案**
   - 分段缓存策略，解决性能和扩展性问题

**选型建议**：
- 对性能要求高：雪花算法
- 需要有序：数据库或Redis
- 简单场景：UUID

### 怎么理解幂等性，有遇到过实际场景么？

**解析**：**幂等性**就是多次执行同一个操作和执行一次产生的效果是相同的。

**常见场景**：

1. **支付场景**：用户支付一次不能扣两次钱
2. **订单创建**：不能重复创建同一个订单
3. **接口重试**：网络超时重试时不能产生重复数据
4. **MQ消息消费**：消息重复消费不能产生副作用

**实现方式**：

1. **唯一索引**：防止新增重复数据
2. **token机制**：操作前获取token，操作时验证token
3. **乐观锁**：基于版本号实现
4. **悲观锁**：操作前加锁
5. **状态机**：根据状态判断，只有特定状态才能流转

**实际例子**：
- 支付系统：同一笔订单号，多次支付只扣一次钱
- 创建订单：前端防重 + 后端唯一索引，避免重复创建

### UML类图理解的怎么样？会画类图吗？

**解析**：UML类图用于描述系统中类的静态结构，主要包含以下几个元素：

**基本元素**：
- **类**：分为三层，类名、属性、方法
- **接口**：使用棒棒糖表示
- **关系**：
  - **继承（泛化）**：空心三角箭头指向父类
  - **实现**：空心三角虚线箭头指向接口
  - **关联**：实线箭头，一个类知道另一个类
  - **聚合**：空心菱形，整体和部分，部分可以脱离整体存在
  - **组合**：实心菱形，整体和部分，部分不能脱离整体存在
  - **依赖**：虚线箭头，一个类使用另一个类

**举例（一个简单的例子）**：
```
动物 ◇——> 心脏（组合）
汽车 ◇——> 轮胎（聚合）
Student ——> Teacher（关联）
Animal ←-- （空心三角）Dog（继承）
Shape ←-- （空心三角虚线）Circle（实现）
```

## 四、网络

### 什么是网络四元组

**【京东】面试题**

网络四元组是 TCP 协议中用于唯一标识一个连接的核心概念，它由源 IP 地址、目标 IP 地址、源端口号、目标端口号四个要素组成。

1. **源 IP 地址**：发起连接的客户端 IP 地址
2. **目标 IP 地址**：服务端的 IP 地址
3. **源端口号**：客户端随机分配的端口（范围：1024-65535）
4. **目标端口号**：服务端监听的端口

五元组是在四元组基础上增加了传输层协议（TCP/UDP）。

### HTTP 1.1 -> 2.0 -> 3.0 演进

**HTTP/1.1**：
- 基于TCP，文本协议
- 队头阻塞问题，同一连接只能处理一个请求
- 支持长连接（Keep-Alive）
- 缓存机制（Cache-Control、ETag）

**HTTP/2**：
- 二进制分帧，多路复用
- 头部压缩（HPACK）
- 服务器推送
- 解决了队头阻塞问题（应用层）

**HTTP/3**：
- 基于QUIC协议，UDP之上
- 解决了TCP队头阻塞问题
- 更好的拥塞控制
- 连接迁移，0-RTT握手

### HTTPS 状态码

常见状态码分类：
- **1xx**：信息性状态码
- **2xx**：成功（200 OK，201 Created，204 No Content）
- **3xx**：重定向（301永久重定向，302临时重定向，304未修改）
- **4xx**：客户端错误（400错误请求，401未授权，403禁止访问，404未找到，405方法不允许）
- **5xx**：服务器错误（500服务器错误，502网关错误，503服务不可用，504网关超时）

## 五、Java 相关

### IOC、AOP、@Autowired 和 @Resource 注解的区别

**IOC（控制反转）**：将对象创建和依赖关系管理交给Spring容器，不是开发者自己new对象。

**AOP（面向切面）**：面向切面编程，不修改原有代码的基础上增加额外功能，如日志、事务、权限校验。

**@Autowired vs @Resource**：
- `@Autowired`：Spring提供的注解，默认按类型装配，结合`@Qualifier`按名称装配
- `@Resource`：JSR-250标准注解，默认按名称装配，找不到名称再按类型装配
- `@Autowired` 依赖注入是通过AutowiredAnnotationBeanPostProcessor处理的，而`@Resource`是通过CommonAnnotationBeanPostProcessor处理的

### Spring Boot 自定义 Starter

自定义Starter需要包含两个部分：
1. **autoconfigure模块**：包含自动配置代码
2. **starter模块**：只用来依赖管理，引入autoconfigure

关键步骤：
- 定义配置属性类
- 编写自动配置类
- 在`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`中注册自动配置
- 提供默认配置条件（`@ConditionalOnClass`、`@ConditionalOnProperty`等）

## 六、大数据

### 数据倾斜 —— 如何发现、判断、解决？

**阿里判断规则**：
- 通过 Spark UI 观察各个Task执行时间和处理数据量
- 如果某个Task处理的数据量远远大于其他Task，就是数据倾斜

**发现方法**：
- 看日志：某个Task执行特别慢，一直卡住
- 看监控：大部分Task很快完成，少数几个跑很久
- 看Shuffle数据分布：某个分区数据量过大

**解决方法**：
1. **加盐优化**：对key增加随机前缀，分散到不同Reduce
2. **两阶段聚合**：先局部聚合，再全局聚合
3. **过滤空key**：如果是null值导致倾斜，提前过滤
4. **增大并行度**：增加Reduce数量，每个分区数据量变少
5. **优化Join**：将小表广播（broadcast join），避免大表Shuffle

### maxwell 在项目里是怎么运行的？

Maxwell是一个CDC工具，用于监听MySQL的binlog，将数据变更输出到Kafka等消息队列。

**运行流程**：
1. 连接MySQL，开启binlog监听
2. 捕获数据的增删改操作
3. 将变更数据转换成JSON格式
4. 发送到Kafka
5. 下游Flink消费Kafka消息，进行实时数据处理

### 了解实时数仓吗，Flink 大概是做什么的？

实时数仓是对数据进行实时处理和分析的数据仓库，满足低延迟的数据分析需求。

**Flink**：是一个分布式流处理引擎，支持高吞吐量、低延迟的流数据处理，可以做：
- 实时ETL
- 实时聚合统计
- 事件驱动应用
- 窗口计算

### 如果用户量激增导致数据库应付不过来如何处理？

### 有了解大模型是怎么训练出来的吗？

大模型训练大致分为几个阶段：
1. **数据预处理**：大规模文本数据清洗、去重、过滤
2. **预训练**：在大规模数据上进行自监督学习，学习语言知识，预测下一个token
3. **微调**：基于人类指令数据进行有监督微调（SFT）
4. **RLHF**：人类反馈强化学习，训练奖励模型，优化模型输出符合人类偏好

### 了解激活函数吗，用过什么激活函数？

激活函数给神经网络引入非线性：
- **Sigmoid**：输出0-1，容易梯度消失
- **ReLU**：max(0, x)，解决梯度消失，现在最常用
- **Leaky ReLU**：解决Dead ReLU问题
- **Tanh**：输出-1到1，中心化
- **Swish/GELU**：Transformer中常用，性能更好

### 了解随机森林吗？

随机森林是集成学习算法，由多棵决策树组成：
- 随机选择样本（bootstrap抽样）
- 随机选择特征
- 每棵树独立训练
- 预测时投票分类或平均回归
- 优点：抗过拟合，泛化能力强，并行训练

### 了解推荐系统吗？

推荐系统常见算法：
- **协同过滤**：基于用户的CF、基于物品的CF
- **矩阵分解**：隐语义模型
- **内容推荐**：基于物品内容特征推荐
- **深度学习**：NCF、Wide&Deep、DIN等深度学习模型
- **召回+排序**：两阶段架构，召回快速筛出候选，排序精排
