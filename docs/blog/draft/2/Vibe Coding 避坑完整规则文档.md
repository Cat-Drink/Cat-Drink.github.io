# Vibe Coding 避坑完整规则文档

## 写在前面

Vibe coding 真的很爽。你只要说一句 “帮我做一个博客系统”，AI 就开始刷刷刷地写文件、搭路由、建数据库。十分钟后，一个能跑的产品雏形就出来了。

但这种爽，往往只持续到第一次需求变更。

当你发现某个功能要改、某个 bug 要修，你打开代码 ——
天哪，service 里混着 UI 逻辑，model 里塞着 API 请求，连状态管理都分散在五个文件里。
你想让 AI 改，它改了一处，另一处莫名其妙挂了。
你开始怀疑人生：这个项目到底是怎么长的？

我过去七个月，每天花六七个小时泡在 vibe coding 里，从零做了好几个项目。
经历了无数次 “推倒重来” 之后，我终于总结出了一套能让你不翻车的规则。

---

## 铁律一：不要一上来就让 AI 写代码

是的，你没有看错。
Vibe coding 最大的陷阱就是太容易开始了。

很多人打开 Claude Code 或 Cursor，第一句话就是：“帮我写一个电商后台。”
AI 很听话，刷刷刷给你生成几十个文件。看起来效率爆炸，但问题在于 ——
AI 是在没有上下文、没有结构、没有规范的情况下自由发挥的。

它猜你想怎么做。
猜对了还好，猜错了，后面每一步都是在错误的基础上继续叠东西。
到最后，你想改一个按钮的颜色，都能引发连锁报错。

---

## 铁律二：把项目结构当房子的梁柱来设计

很多人觉得结构是小事 —— 等代码多了再重构嘛。
但等你代码多了，你根本重构不动。因为每个模块都跟其他模块纠缠在一起，动一处就塌一片。

### 标准项目结构（后端 Node\.js 示例）

```Plain Text
backend/
├── routers/        # 只负责路由和请求分发
├── controllers/    # 只负责接收参数、调用service、返回响应
├── services/       # 只负责业务逻辑
├── models/         # 只负责数据结构定义和数据库操作
├── utils/          # 工具函数
└── config/         # 配置项
```

## 铁律三：架构思维 —— 让每一次调用都可预期

Vibe coding 做得最多的事情是 “加功能”。
今天加个登录，明天加个评论，后天加个搜索。
如果一开始没有架构思维，这些功能就是一堆乱七八糟的补丁。

### 架构思维核心三句话

1. **每一层都有边界**
Controller 不该做的事，打死也不做。
Service 不该碰的数据库字段，碰了就算 bug。

2. **每个模块都有清晰的职责**
一个模块只做一件事。比如 “发送邮件” 就是一个模块，“生成报告” 是另一个模块。
不要搞出一个 “发送邮件并且生成报告并且更新数据库” 的巨无霸函数。

3. **每一次调用都是可预期的**
你调用一个函数，传入 A，就应该得到 B。
不会今天返回 B，明天返回 C，后天直接报错。
如果你不确定一个函数会干什么，说明设计有问题。

这三句话听起来简单，但你在 vibe coding 里一不小心就会违反。
因为 AI 最喜欢做的事就是 “方便”—— 哪里方便写哪里，哪个函数顺手就往里塞逻辑。
作为人，你的职责就是盯着 AI，不让它越界。

> 架构不是限制，而是给你的未来留后路。

## 写在最后：Vibe coding 教我的，远不止代码

做了一年 vibe coding 之后，我发现我学到的最重要的东西不是怎么用 AI 写代码，而是：

- 如何把一个大问题拆成小问题（架构思维）

- 如何在不确定性中建立规则（文档先行）

- 如何快速试错并及时回头（Git 和最佳实践）

- 如何让一个聪明的帮手按照你的意图做事（清晰的边界和职责）

有人说 vibe coding 会让程序员变懒、变菜。
我觉得恰恰相反 —— 它会逼你成为更好的架构师和产品经理。

因为 AI 已经把执行层的活干完了。
你最值钱的能力，变成了做对决策的能力。
