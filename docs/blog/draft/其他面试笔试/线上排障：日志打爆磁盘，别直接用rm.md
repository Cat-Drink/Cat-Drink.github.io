# 线上排障：日志打爆磁盘，别直接用rm

## 问题现象

线上服务器触发磁盘报警，日志目录使用率达到100%。很多人的第一反应是：

```bash
rm 大型日志文件
```

删掉文件后，用 `ls` 查看确实文件不见了，但用 `df` 查看系统磁盘空间 **依旧是100%**。

报警没有解除，新日志写不进去，业务持续报错。

## 为什么会这样？

在Linux系统中，真正释放磁盘空间必须同时满足两个条件：

1. 文件的硬链接计数为0
2. **没有任何存活进程持有该文件的文件描述符**

当业务进程正在持续向日志文件写入时，使用 `rm` 仅仅是移除了文件在目录树中的访问路径，但：

> 后台进程持有的文件描述符还紧紧抓着它，物理磁盘上的数据块并没有被内核回收，这部分空间陷入了被僵尸进程占用的状态。

## 问题定位方法

使用 `lsof` 命令可以过滤出系统里所有**已经被删除，但依然被活动进程持有**的文件：

```bash
lsof | grep "deleted"
```

在输出结果中能够直接看到是哪一个业务进程占用了这个巨大的日志文件。

## 解决方法

### 方法一：重启服务（非生产环境推荐）

定位到进程后，如果业务允许，直接重启该服务。进程退出后，文件描述符被释放，磁盘空间马上就会恢复。

但在生产环境中，通常不能为了清理日志去重启核心业务服务。

### 方法二：文件截断（生产环境推荐）

**正确的做法是文件截断，不要直接删除。**

直接在终端执行：

```bash
: > /path/to/your/big.log
# 或者
> /path/to/your/big.log
```

这种截断操作：
- **不改变文件的inode节点**
- **不破坏正在运行的进程链路**
- 只负责清空数据块的内容

执行完回车，十几个G的空间立刻被释放，**并且业务进程完全没有感知**，不需要重启服务。

## 总结

线上日志打爆磁盘时：
- ❌ 不要直接 `rm` 删除正在被写入的日志文件
- ✅ 正确做法：使用 `> filename` 进行文件截断

这样既能立即释放磁盘空间，又不影响正在运行的业务进程。
