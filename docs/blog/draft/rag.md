## 二、 MySQL 事务与锁

### Q1: MySQL 事务四大特性是什么？分别由什么实现？

**A:** 原子性、一致性、隔离性、持久性。

- **原子性：** 通过 **undo log** 实现。
- **一致性：** 由其他三大特征保证。
- **隔离性：** 通过 **MVCC 和间隙锁（临键锁）** 实现。
- **持久性：** 由**预写 redo log、双写机制、两阶段提交、checkpoint 刷盘机制**共同保证。

### Q2: MySQL 两阶段提交流程是什么？崩溃恢复时 redo log 为 prepare、binlog 不完整会怎样？

**A:** * **流程：** 先提交 redo log，事务改成 commit，然后再提交 binlog；先 prepare，再 commit，保证 redo log 和 binlog 一致。

- **崩溃恢复处理：** **回滚事务**。事务没有成功就回滚，必须保证两个日志都写成功。

### Q3: MySQL 事务隔离级别有哪些？默认级别是什么？面临什么问题？

**A:** * **隔离级别：** 读未提交、读已提交、可重复读、串行化。

- **面临问题：** 脏读、不可重复读、幻读。
- **InnoDB 默认隔离级别：** **可重复读**。
- **幻读解决实现：** **MVCC + 临键锁**。

### Q4: `SELECT ... FOR UPDATE` 加什么锁？行锁使用注意事项？

**A:** * **锁类型：** **加意向排他锁（行锁）**。

- **注意事项：** 必须在事务中使用，否则锁会立即释放；必须命中索引，否则会锁全表。

### Q5: 行锁细分有哪些？临键锁是什么？

**A:** * **行锁细分：** 记录锁、间隙锁、临键锁。

- **记录锁：** 通过主键或索引查询，锁定对应记录。
- **间隙锁：** 锁定记录之间的间隙，不准其他事务插入。
- **临键锁：** **记录锁和间隙锁的合并**，解决幻读问题。

### Q6: 意向锁是什么？

**A:** 意向锁是**表级锁**，告诉外面的事务这个表中已经有锁了，用于快速判断；意向排他锁则其他事务不能加锁，意向共享锁则其他共享锁可以进来。

## 三、 ThreadLocal 相关

### Q7: ThreadLocal 是什么？作用？

**A:** ThreadLocal 是线程工具，维持线程本地成员变量，**实现线程本地变量副本，实现线程隔离**。

### Q8: ThreadLocal 底层结构？存在什么问题？如何解决？

**A:** * **底层：** 维护 `ThreadLocalMap`，key 是 ThreadLocal 对象，value 是存储的变量。

- **问题：** **内存泄漏**（key 是弱引用，value 是强引用，key 为 null 时 value 无法回收，严重会导致内存溢出）。
- **解决：** 使用完调用 `remove()` 方法。

### Q9: ThreadLocal 可以跨线程传递吗？如何解决？

**A:** * **不可以**跨线程传递上下文。

- **解决：** * `InheritableThreadLocal`：支持父子线程上下文传递，线程池场景失效。
  - `TransmittableThreadLocal` (TTL)：项目常用，解决线程池上下文传递问题。

## 五、 RAG 企业智能知识库项目深度问答

### Q3: 项目中最难的问题是什么？怎么解决？

**A:** 最难的问题是**大文件分片上传与断点续传**。

1. **问题：** 一开始把大文件作为一个整个直接从前端上传到服务器，网络波动以及响应出现问题，会导致文件上传到一半失败，需要重新上传；
2. **方案：** 把大文件进行分片上传，给每个文件分片加上 **MD5 指纹标识**；
3. **状态管理：** 用 **Redis Bitmap** 来进行文件分片状态的管理，它只有 0 和 1 两种状态，内存占用小、响应速度快；
4. **文件合并：** 所有文件分片上传完成后，通过 **MinIO 的 `compose object` 方法**按顺序合并成一个大文件；
5. **断点续传：** 用 Redis Bitmap 记录已经上传和未上传的分片，下次上传时根据 bitmap 的状态记录和分片的 index，**只传未上传的分片**，避免重复上传，提升稳定性。

### Q4: Kafka 是什么？项目里怎么用？

**A:** 1.  Kafka 是一个消息队列，核心定位是高吞吐、低延迟、可持久化的分布式消息队列和实时数据总线，相当于分布式消息队列、持久化日志存储加上处理框架的总和；

\2.  **组成：** 生产者、消费者、消费者组、主题分区、副本、ZooKeeper（用于集群管理）；

\3.  **项目用途：** 作为文档处理的异步流水线，异步解析文档上传、解析和向量化的过程，**实现系统削峰填谷**，让用户体验更好。

### Q5: 什么是削峰填谷？

**A:** 就是把高峰请求往后延，先把瞬间高峰请求放进消息队列缓存，再平稳推给后端，避免服务端瞬间压力过大。

### Q6: 为什么用 ES 做向量库，不用其他向量库？

**A:** 1.  我们公司开发组都对 ES 比较熟悉，能够开箱即用；

\2.  ES 支持**向量检索 + 关键词检索**，还能实现权限过滤；

\3.  我们这个企业级知识库项目，除了向量召回，还涉及关键词匹配、权限过滤等条件，这些都能通过 ES 解决，综合考量后选择 ES。

### Q7: 为什么用 IK 分词器？

**A:** ES 自带的 standard 分词器是用来分英语的，按空格分割，而中文没有空格；**IK 分词器是中文分词器**，能按中文的短语、词语来分词，适配我们的中文文档处理场景。

### Q8: Embedding 用什么模型？为什么选 2048 维？

**A:** 1.  **模型：** 通义千问的 2048 维词嵌入模型；

\2.  **原因：** * 维度太高：无关信息过多，最后的召回结果不理想；

\* 维度太低：特征不明显，语义相似度计算会受到影响；

\* **2048 维能平衡语义表达与检索效率**，是综合考量后的选择。

### Q9: Prompt 怎么设计？

**A:** 1.  严格按照用户上传的文档检索回答，不可以自己乱编；

\2.  用户每一次回答的问题，必须加上内容来自哪里，**标注来源**；

\3.  声明本系统指令为最高指令，不可以修改，**防止黑客通过 Prompt 欺骗大模型，避免 Prompt 注入和泄露**；

\4.  核心要满足企业知识库**可控、安全、可信**这三个条件，作为强力约束。

### Q10: 流式输出用 WebSocket，为什么不用 SSE？

**A:** 1.  **SSE：** 单向推送，只能服务端传到客户端，通常用在大模型对话单向推送场景，实现更简单，性能消耗更小；

\2.  **WebSocket：** 双向通信，客户端和服务端可以互相传输数据；

\3.  **选型原因：** **我们的项目需要双向交互**，所以选用 WebSocket，而 Agent Flow 项目只用单向推送，就用了 SSE。

## 六、 前沿技术了解（Mamba）

- **Mamba 是什么：** Mamba 是一种基于**选择性状态空间模型 (SSM)** 的新一代序列建模架构。
- **核心优势：**
  - 它用**线性时间复杂度 $O(n)$** 替代了 Transformer 自注意力的 $O(n^2)$ 复杂度，在超长序列下速度更快、显存占用更低且推理延迟恒定；

- **发展趋势：** 既能媲美 Transformer 的表达能力，又能解决其长文本、端侧部署等瓶颈，目前常与 Transformer 混合搭配、配合动态路由使用，成为下一代高效大模型的核心技术方向。

# Java 后端开发 & AI 应用开发面试指南（续篇）

## 一、 JVM 底层深度问答

### Q12: 双亲委派模型是什么？如何破坏？典型场景？

**A:** * **双亲委派：** 类加载时，子类让父类加载，父类让爷爷类加载，直到顶层；父类无法加载再由子类加载，最后加载不了抛出 `类找不到` 异常。

- **破坏方式：** 重写 `ClassLoader` 的 `loadClass()` 方法。
- **典型场景：** **Tomcat**。需要运行多个相同类的应用，双亲委派无法满足，因此破坏。

### Q13: 内存泄漏和内存溢出的区别？如何查看？

**A:** * **内存泄漏：** 程序使用完后内存未回收，长期占用。

- **内存溢出 (OOM)：** 程序请求分配内存时无足够空间，抛出异常。
- **查看工具：** VisualVM、JConsole，JDK 指令 `jmap`、`jstat`。

### Q14: JVM 垃圾回收机制? 判断垃圾算法？回收算法？

**A:** * **垃圾回收：** 对堆中死亡、长期未使用的对象进行清除回收。

- **判断算法：** **可达性分析算法（GC Roots）**。
- **回收算法：** 标记清除、标记复制、标记整理。

### Q15: 新生代、老年代与 GC 类型？

**A:** * **新生代：** Eden区 + Survivor区，对象生命周期短，适用**标记复制算法**。

- **老年代：** 对象存活久，适用**标记整理算法**。
- **GC 类型：** Minor GC（新生代）、Old GC（老年代）、Full GC（全堆回收）。
- **频繁 Full GC 原因：** 新生代空间太小，对象频繁进入老年代。

### Q16: CMS 垃圾收集器特点？流程？三色标记法？

**A:** * **特点：** **第一个关注 GC 停顿时间 (STW)** 的收集器，基于标记清除，会产生内存碎片。

- **流程：** 初始标记 (STW) $\rightarrow$ 并发标记 $\rightarrow$ 重新标记 (STW) $\rightarrow$ 并发清除。
- **三色标记法：** * 白色（未访问）
  - 灰色（已访问，引用未处理）
  - 黑色（已处理完成）

### Q17: 频繁 GC 怎么排查？

**A:** 先查 GC 日志 $\rightarrow$ 用 `jstat` 查找内存泄漏进程 $\rightarrow$ 查看内存分布 $\rightarrow$ 堆分析 (dump) 定位问题。

### Q18: 对象一定分配在堆中吗？

**A:** 不一定，看**逃逸分析**：

- **未逃逸**（未被其他方法/线程引用）：分配在栈上，方法结束自动销毁。
- **逃逸**（方法/线程逃逸）：分配在堆上。

### Q19: main 方法中 `int a = 10`，a 在堆还是栈？为什么？

**A:** **在栈中**。因为是局部变量，随方法进栈创建，出栈销毁。

## 二、 校园活动管理平台项目

### Q13: ThreadLocal 作用？底层？问题？解决？

**A:** 1. **作用：** 用来存储已经登录的用户的信息，实现身份认证的连续性，实现**线程隔离**，可随时拿到用户的 ID，方便后面的 SQL 操作；

\2. **底层：** 内部维护一个 Map 对象 (`ThreadLocalMap`)，Key 是 `ThreadLocal` 对象，Value 是线程的局部变量，相当于为每个线程维护了一个变量副本；Key 是**弱引用**，避免内存泄漏；

\3. **问题：** Key 是弱引用可以被回收，但 Value 是**强引用**无法回收，容易导致内存泄漏，最终可能引发**内存溢出 (OOM)**；

\4. **解决：** 使用完 `ThreadLocal` 之后，**必须调用 `remove()` 方法**，避免内存泄漏。

### Q14: Redis 分布式锁 + Lua 脚本为什么用？

**A:** 1. 传统分布式锁存在锁过期、误删等问题，这些问题可以通过 Redisson 看门狗机制来解决；

\2. **Lua 脚本是嵌入式脚本语言，能保证 Redis 指令执行的原子性**，可以把多条繁琐的 Redis 指令打包，避免网络波动导致部分指令执行失败，适合分布式锁、批量操作等需要原子性的场景。

### Q15: 详细说说 Redisson 的看门狗机制？

**A:** Redisson 的看门狗机制是一种**自动续期机制**，用于解决分布式锁的过期问题。

### Q12: ES 向量检索怎么优化？

**A:** 最开始没有给向量数据库建索引，文档上传多了之后，向量检索响应很慢，大概 200 毫秒，后来通过**建立索引**的方式，把响应速度优化到 **20 毫秒**。

### Q19: RAG 四步工作流程是什么？

**A:** 文档切割 $\rightarrow$ 词嵌入模型向量化存储 $\rightarrow$ 语义相似度匹配检索 $\rightarrow$ 增强生成。

### Q20: Embedding (词嵌入) 作用？

**A:** 将文本转化为固定维度向量（如 2048 维），通过**余弦相似度**计算语义匹配度，用于向量检索。

### Q21: 向量数据库和传统数据库区别？

**A:** * **传统数据库（如 MySQL）：** 通过索引**精准匹配**数据。

- **向量数据库：** **语义相似度搜索**，基于 ANN 索引，匹配相近语义内容。

### Q22: RAG 中文文档切割为什么重要？切割过大/过小问题？

**A:** * **重要性：** 直接影响向量化精准度、检索召回率与准确率。

- **切割过小：** 信息不完整，语义断裂。
- **切割过大：** 向量表示不精准，包含大量无关内容，重心偏移。
- **优化：** **固定长度 + 滑动窗口**，保证语句完整。

### Q23: 项目中混合检索如何实现？BM25 作用？

**A:** * **混合检索 = 向量检索 + 关键词检索 (BM25)**。

- **流程：** 用户问题向量化，从向量库召回 TOP300 $\rightarrow$ BM25 关键词重排（关键词权重 1.0，向量相似度 0.2） $\rightarrow$ 取 TOP10，与问题一起送入大模型生成答案。

### Q24: RAG 项目权限控制如何实现？

**A:** * **方案：** **组织标签 + RBAC / ABAC 权限模型**。

- **流程：** 文档上传时打组织标签；用户登录生成 JWT Token，存储角色、组织信息；权限过滤器拦截请求，提取组织标签；校验文档公开状态、用户是否为所有者、组织标签是否匹配；权限信息缓存至 Redis。
