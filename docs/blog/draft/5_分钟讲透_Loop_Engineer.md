好，那一个真正能跑的 loop 到底需要什么？

Google Cloud AI 总监 [姓名 unclear，原稿为"adowmoney"] 拆得最清楚：五个零件加一个记忆层——automations、work trees、skills、plugins、sub-agents，再加一个 memory。Codex 和 Claude Code 现在都齐了。

- **Automations**：按计划自动跑，自己发现问题，自己分类。
- **Work trees**：每个 agent 一个独立工作树，并行，不踩脚。
- **Skills**：把项目知识写成文档，不写 agent 就只能猜。
- **Plugins**：把 agent 接入 GitHub、Slack、Linear，让它能碰到你真实的工作环境。
- **Sub-agents**：一个负责想，一个负责验，提案和审核不能是同一个 agent。

然后是第六个东西：**记忆**。一个 Markdown 文件也行，一个 Linear board 也行。记着什么做完了，什么还没做。听起来太简单了，但所有长跑 agent 全靠它——模型每次跑完就忘。所以记忆必须在磁盘上，不能在上下文里。Agent 会忘，[report 不会跟 是什么关系，此句 unclear]。

2023 年，所有人都爱讲 prompt engineering：怎么写 prompt，怎么调温度，怎么 few-shot。核心假设是你说的越清楚，模型就干得越好。

2025 年，[姓名 unclear，原稿为"and"] 说了一句话：context engineering 才是正事。不是你怎么说，而是你把什么文档、代码、历史对话、工具输出打包成一个上下文喂进去。从怎么问变成了怎么喂。

再往后，harness engineering：不止喂，还要编排。哪个 agent 负责规划、哪个负责实现、哪个负责验收。三个角色可以是三个不同的模型，规则、工具、权限、验收标准全配齐。从怎么喂变成了怎么管。

然后就是今天讲的 loop engineering。Harness 搞定了一次任务的编排，loop 解决的是让它反复跑：自动触发、自动恢复、状态持久化，跑完一轮自己决定下一轮。从怎么管变成了怎么让它自己转。

四个阶段一条线：**prompt、context、harness、loop**。你操心的东西越来越少，但你要设计的系统越来越重。
