# 其实作为面试官，我不在乎你的后端项目

## 面试感悟

这段时间参与复盘了几场后端开发/高并发系统的面试，有个感受特别强烈：大多数候选人不是项目不够好，而是讲项目的方式完全跑偏了。

### 候选人的共同点：
- 简历上项目写得很丰富
- 技术栈列得很全面
- 业务流程也能讲清楚
- 但只要追问三轮，专业短板立刻暴露

### 面试官到底在听什么？

很多同学以为面试官在评判"你写了多少功能、用了多少框架"。

但实际上面试官更关心三件事：
1. 你能不能把一个真实技术问题讲清楚
2. 你有没有系统层面的架构设计思维
3. 你是不是具备可培养性和故障交付能力

这三点，往往才决定你能不能过。

## 后端项目讲法的正确打开方式

### 第一步：先把"后端项目"分清层级

所有后端项目讲述，本质也就三层：

#### 第一层：功能层
你写了什么接口，实现了什么业务，功能是否跑通

#### 第二层：设计层
你怎么做技术选型、数据库设计、接口设计，为什么这么设计

#### 第三层：工程层
高并发怎么扛、数据一致性怎么保证、线上故障怎么排查、性能优化怎么做

很多候选人只停留在第一层。于是面试官一追问，就发现你只是"写过"，不是"设计过"。

### 第二步：每一个技术点，准备"讲法结构"，而不是背稿

用面试里最常见的"高并发接口优化"举例：

#### ❌ 常见讲法：
"我用 Redis 做了缓存，用消息队列削峰填谷，优化后 QPS 提升了，数据库压力降了"

这句话听起来很忙，但信息量很低。因为面试官听不到你做技术决策的过程。

#### ✅ 面试官更想听的是：
- 业务核心痛点是什么，性能瓶颈如何定位
- 你怎么定义优化目标（QPS、P99 延迟、成功率）
- 缓存方案怎么设计（穿透、雪崩、击穿如何应对）
- 为什么选 Redis 而非本地缓存，为什么选 RocketMQ 而非 Kafka
- 压测怎么做的，发现了什么瓶颈，怎么解决的
- 上线策略是什么（灰度、回滚），线上怎么监控和告警

### 第三步：会做技术对比的人，基本都会加分

很多同学讲技术选型只会一句话："这个框架比较常用、社区活跃"。

站在面试官角度，这属于没讲。面试官更想听你怎么对比、怎么取舍。

## 大厂高频场景题

### 01、开篇-面试那些事儿

现在的面试，在问到项目经验的时候，往往会问到你在项目中用到了哪些编程技术和核心的业务思路，以及你在项目中遇到的问题是如何解决的。通过你回答这些问题，可以全面了解你的技术深度和解决问题的能力。

那么如何讲解一个项目呢？我们可以从以下几个要点去讲解：
1. 讲解项目的整体架构
2. 讲解项目的业务流程和亮点业务
3. 讲解在项目实现过程中遇到的难点和解决方案
4. 自己在项目中的贡献和积极态度
