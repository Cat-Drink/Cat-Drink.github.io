# 什么是 CI/CD？为什么现在发布代码越来越快？

简单说，**CI/CD 是一套让代码从写完到上线尽可能自动化、标准化的工程流程**。

它解决的不是某一行代码怎么写，而是解决**团队如何更快、更稳、更少出错地把代码交付给用户**。

## CI：持续集成（Continuous Integration）

这里的"集成"指的是：把不同开发者写的代码**频繁合并到同一个主代码库**。

### 过去的问题
过去很多团队几个人各写各的，过一两周甚至一个月才合并一次，结果一合并就出问题：
- 这个人改了接口
- 那个人的功能依赖旧接口
- 这个模块能跑，另一个模块直接崩了

### CI 的核心思想
不要等到最后才合并，而是**每天甚至每次提交代码后都尽快合并，并且立刻自动检查**。

自动检查通常包括：
1. **代码能不能编译/构建成功**：Java 打包、前端生成静态文件、Go/ Rust 编译通过
2. **自动化测试能不能通过**：单元测试检查函数逻辑、接口测试检查服务调用
3. **额外检查**：代码规范扫描、安全漏洞扫描、依赖检查

只要其中一步失败，系统就会告诉开发者：这次提交有问题，先别往下走。

### CI 的价值
CI 的价值不是让程序自动变好，而是**把问题暴露得更早**。越早发现问题，修复成本越低：
- 一个bug刚提交就发现：开发者还记得改了什么，可能 10 分钟就能修掉
- 等到月底集成才发现：大家都忘了上下文，还要排查一堆互相影响的改动，成本非常高

CI 就像在代码进入主干之前，设置了一道**自动安检门**。

## CD：持续交付 vs 持续部署

CD 有两个常见含义：
- **持续交付（Continuous Delivery）**：代码经过自动构建、自动测试后，随时都处于可以发布的状态，但最后是否上线，通常还需要人工点一下确认
- **持续部署（Continuous Deployment）**：更进一步，只要代码通过所有检查，就**自动发布到生产环境**，不再需要人工批准

两者的区别在于最后一步是否自动上线，但目标都是**减少发布过程中的手工操作和不确定性**。

## 为什么 CD 能让发布越来越快？

CD 从三个方面改变了发布流程：

### 1. 把流程固定下来

传统发布流程：
开发写完代码 → 测试手动部署环境 → 测试 → 运维手动打包上传服务器 → 改配置 → 重启服务

这个过程有很多**等待**（等人、等环境、等审批、等窗口期），也有很多**手工风险**（命令敲错、配置复制错、版本包拿错、服务器漏更新）。

CI/CD 把流程变成**流水线**：一组按顺序执行的步骤，拉取代码 → 安装依赖 → 构建 → 测试 → 生成制品 → 部署到测试环境 → 部署到预发布/生产。

每一步都有明确输入输出，规则写好后，机器每次都按同样方式执行，不会漏掉步骤。

### 2. 把发布拆小

以前发布慢，很大原因是**一次发布包含太多改动**：几十个功能、上百个 bug 修复堆在一起上线，任何一个地方出问题都很难定位。

现在团队倾向于**小提交、小发布**：每次改动范围更小，自动测试更容易覆盖，出问题也更容易回滚。发布从"大型活动"变成"日常动作"，速度自然就上来了。

### 3. 环境变得更一致

很多事故不是代码本身错，而是"我电脑上能跑，服务器上不能跑"。

现代 CI/CD 通常配合**容器**、**基础设施即代码**等技术：
- 容器把应用和运行依赖打包在一起
- 基础设施即代码用配置文件描述服务器、网络、数据库等资源

这样开发环境、测试环境、生产环境的差异会更小，部署结果也更可预测。

## 一个典型的 CI/CD 流程

1. 开发者完成功能，把代码提交到 Git 仓库
2. 检测到提交，**自动触发 CI 流水线**：检查代码格式 → 执行构建 → 跑测试
   - 如果失败：在代码合并请求里标红，开发者修复后重新提交
   - 如果通过：代码可以合并到主分支
3. 合并后，**CD 流水线继续**：部署到测试环境 → 验证功能 → 确认没问题后发布到生产环境
4. 生产发布通常会采用**渐进式发布策略**降低风险：
   - **蓝绿部署**：准备两套环境，一套跑旧版本，一套跑新版本，验证通过后切流量
   - **金丝雀发布**：先让一小部分用户用新版本，确认错误率、响应时间正常，再逐步扩大流量

这样即使新版本有问题，也只影响少量用户，并且可以快速切回旧版本。

这就是为什么今天很多互联网产品可以**一天发布多次**——不是开发者写代码突然快了几倍，而是代码交付链路被自动化了。以前很多时间浪费在沟通等待、手工部署、反复排查环境问题上，现在这些步骤被工具和流程接管。

## CI/CD 的前提条件

CI/CD 并不等于随便写代码也能安全上线，它依赖几个前提：

1. **足够可靠的自动化测试**：否则流水线只是自动把 bug 送到线上
2. **清晰的分支管理和代码评审**：避免低质量代码直接进入主干
3. **监控日志和告警**：上线后能及时知道系统是否异常
4. **回滚方案**：出现问题时能快速恢复，而不是现场手忙脚乱

## 总结

CI/CD 的本质是**软件交付方式的升级**：

- **CI**：让代码频繁集成，并在早期自动发现问题
- **CD**：让通过验证的代码能够稳定快速地交付或部署

它把发布从**依赖个人经验的手工流程**，变成**可重复、可追踪、可自动执行的流水线**。

现在发布代码越来越快，不只是因为工具更先进，更是因为团队把**大批量、低频率、高风险**的发布，变成了**小批量、高频率、可验证、可回滚**的持续交付过程。

真正的速度来自：流程标准化、反馈自动化、风险被持续拆小。
