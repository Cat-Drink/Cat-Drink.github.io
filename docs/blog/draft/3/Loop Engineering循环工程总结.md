# Loop Engineering（循环工程）- 总结

## 什么是循环工程

**Loop Engineering（循环工程）** 是2026年6月诞生的AI工程化第四代范式，核心转变是：

> 从「手动逐条提示Agent的操作员」升级为「设计自主驱动Agent循环系统的架构师」。

人类不再是逐句指挥者，而是系统设计者。人类定义目标，AI系统自主迭代。

---

## AI工程化的四次范式跃迁

AI开发的核心关注点逐层升维，四层范式为叠加关系，并非互相替代：

| 阶段 | 工程名称 | 核心问题 | 核心工作逻辑 | 局限性 |
|------|----------|----------|--------------|--------|
| **一代** | Prompt Engineering<br>提示词工程 | 怎么问 | 打磨单次指令，追求模型单次输出合格结果 | 完全依赖人工逐轮输入，人力成本极高 |
| **二代** | Context Engineering<br>上下文工程 | 给AI什么信息 | 构建知识库、历史记录、约束条件，补齐项目完整背景 | 仅解决信息缺失，仍需人工持续引导步骤 |
| **三代** | Harness Engineering<br>编排工程 | 如何调用能力 | 串联模型、工具、数据库，搭建多步骤工作流 | 仅实现工具串联，无法自主循环迭代任务 |
| **四代** | Loop Engineering<br>循环工程 | 如何让AI持续自主完成目标 | 构建闭环循环系统，自动调度、执行、校验、重试、记录状态 | 系统设计门槛更高，需配套校验机制规避风险 |

**核心关系**：循环工程位于上层，是更高维度的系统设计；提示词质量会直接影响循环产出效率。

---

## 循环工程解决的核心问题

传统人工模式中，人在中间传话触发。循环工程彻底拆掉人工中转，解决三个核心问题：

1. **谁来触发下一次运行**
2. **谁来评估结果质量**
3. **谁来决定停止循环**

---

## 六大核心构成要素

一套可稳定无人值守运行的Loop，由 **5个功能组件 + 1个持久记忆系统** 组成：

### 1. 自动触发器 —— Loop的心跳
- 替代人工手动唤醒Agent
- 支持定时触发、事件触发（CI失败、工单新增等）
- 实现无人值守持续运行

### 2. Worktree并行隔离层 —— 多Agent安全并行底座
- 为多个并行Agent分配独立代码工作区
- 自动隔离文件修改冲突，支撑多任务同时处理
- 关键瓶颈：人工代码审查带宽才是并行Agent数量的真实上限

### 3. 技能文件（Skills）—— Agent长期记忆卡片
- 项目内`SKILL.md`文件夹，固化全局项目知识
- 技术框架规范、目录结构、构建部署流程、历史事故约束等
- 解决**意图债务**：避免每次会话AI从零猜测项目规则，一次编写全会话复用

### 4. 连接器（基于MCP）—— 打通真实业务环境
- 通过MCP统一协议打通外部业务系统，赋予Loop实操能力
- **研发侧**：读取Issue工单、自动创建PR、查询数据库、调用第三方API
- **协作侧**：联动Slack/企业微信推送通知、对接CI/CD流水线
- 核心价值：让Agent不止"描述修复方案"，而是在真实环境完成端到端交付

### 5. 子Agent —— 制作者与检查者分离（Maker-Checker制衡架构）
- **痛点**：同一模型同时写代码+自检，会天然自我宽容，Prompt优化无法根治
- **二元分工**：制作Agent（实现代码） + 检查Agent（独立校验缺陷）
- **三角分工**：探索Agent（定位问题） + 实现Agent（开发修复） + 验证Agent（按规范验收）
- 独立校验是系统可信的基础，能捕捉单一模型自圆其说而忽略的漏洞

### 6. 持久记忆 —— 跨对话状态存储（Loop的大脑皮层）
- 大模型单次对话结束后会完全遗忘上下文
- 将任务状态写入代码仓库文件（`TODO.md`）或第三方看板
- 永久记录：已完成事项、合并记录、未关闭任务、执行失败日志

### 六要素一句话总结
> 自动触发让循环转起来，Worktree让并行安全，技能让Agent不迷路，连接器让Agent能动手，子Agent让质量有保障，记忆让循环能持续。

## 双层循环核心架构

区分两层循环是理解Loop Engineering的关键：

### 内循环（Inner Loop）
- Agent内置原生逻辑，无需开发者额外设计
- 固定遵循「感知→推理→行动→观察」闭环
- 例如：读取文件 → 修改代码 → 运行测试 → 读取报错 → 迭代修改
- **Prompt Engineering优化的就是内循环每一步输出质量**

### 外循环（Outer Loop）
- Loop Engineering的核心设计对象，相当于调度多个工程师的项目经理
- 负责全生命周期管理：按计划发现任务 → 分派子Agent执行 → 汇总验证结果 → 持久存储状态 → 开启下一轮循环
- **Loop Engineering就是设计外循环**

### 核心区别
- Prompt工程：优化单次执行的内容质量，作用于**内循环**
- Loop工程：设计整体调度、并行、校验、持久化系统，作用于**外循环**

## 经典运行流程

```
执行 → 观察 → 评估 → 修正 → 回到执行 → ... 直到目标达成
```

这就是Agent自主迭代、自我修正的闭环。

## 风险与三大债务

循环工程会放大三类风险，必须警惕：

| 风险类型 | 说明 | 应对措施 |
|----------|------|----------|
| **成本失控** | 无限迭代可能消耗大量资源 | 设置预算/迭代上限 |
| **自我认同偏差** | AI自我验证容易放过问题 | 外部验证/多Agent分离校验 |
| **可观测性缺失** | 自主运行黑盒化，问题难追溯 | 输出完整运行日志 |

除此之外，还有三类需要警惕的问题：

1. **理解债**：Loop产出代码越快，开发者通读理解比例越低，要坚持阅读所有生成代码
2. **认知投降**：人类形成认知惰性，直接全盘接受结果，需要开发者保持独立判断
3. **最终验证责任仍归人类**：无论自动化程度多高，代码合并前人工终审不可省略

## 落地实践指南

### 起步原则
不追求一步全自动，新手优先选择**只读、低风险**的轻量化Loop，逐步提升权限。

### 最小可用入门Loop（零代码修改风险）
1. 每日定时读取CI失败日志、项目Issue工单列表
2. 按业务优先级整理任务清单，写入`TODO.md`
3. 不修改代码、不发起PR
4. 开发者每日查看清单，自主安排执行

### 自动化程度迭代路线
1. **一阶（只读）**：任务分类、优先级梳理（入门）
2. **二阶（轻度写）**：自动起草问题修复方案
3. **三阶（中度操作）**：自动创建Draft草稿PR
4. **四阶（高度自动化）**：双/三子Agent校验 + 符合规则后自动合并

## 核心总结

**Loop Engineering的核心转变**：
> 从「我来驾驶AI」变成「我来设计驾驶AI的规则」。

循环工程将工程师的工作从「逐轮手动对话AI」升维到「自主循环系统架构设计」。系统自动化程度越高，对人类判断力、业务理解力的要求反而越高。

> 我不再提示AI了，我在写让AI自己跑起来的系统。
