# 二、大模型推理能耗核心理论与计算公式

## 2\.1 底层浮点运算逻辑

基于Transformer架构的大模型，推理算力消耗具备明确的数理依据：每处理/生成1个token，约执行2倍模型参数量的浮点运算。

核心基础公式：

总FLOPs ≈ 2 × 模型参数量\(N\) × 总token数

能耗基础定义：Energy\(焦耳\) = 总FLOPs × 单次FLOP能耗

## 2\.2 工程实战简化公式

行业通用可落地的经验公式（ML Energy Leaderboard）：

**能耗 ≈ k × 参数量\(B\) × token数**

参数说明：

- k为综合系数，整合了硬件性能、数值精度、推理框架优化效果；

- 实测参考：H100显卡运行9B模型，k取值区间为0\.012～0\.016 J/\(B·token\)。

## 2\.3 公式适用边界（关键避坑点）

该公式仅用于快速量级估算，无法精准测算生产环境成本，核心限制条件：

- 非线性偏差：超大模型会产生额外内存带宽开销，实际能耗高于线性公式预期；

- 阶段差异：Prefill输入编码、Decode输出生成两个阶段能耗特性天差地别；

- 隐性开销：未包含GPU空载功耗、机房散热、数据中心PUE电力损耗；

# 三、推理能耗核心影响因素（全维度解析）

## 3\.1 模型参数量（基础核心因子）

同等架构、精度、硬件条件下，模型参数量与能耗基本呈正相关，参数翻倍，能耗接近线性上涨甚至小幅超线性增长。但量化、MoE稀疏架构、新一代GPU会打破这一线性关系，实现降本增效。

## 3\.2 推理阶段差异（定价核心依据）

大模型推理分为两个核心阶段，能耗差距极其悬殊，是云API输入输出差异化定价的根本原因：

- Prefill（输入编码）：仅占总能耗3%左右，能耗极低；

- Decode（输出生成）：占总能耗96%以上，是推理能耗的绝对主力。

业务结论：输出token的能耗成本远高于输入，因此所有云厂商均采用“输出定价＞输入定价”的策略。

## 3\.3 量化压缩技术

数值精度优化是最直接的降能耗手段，以FP16为基准：

- INT8/FP8量化：降低40%推理能耗；

- 4\-bit量化：降低60%推理能耗；

绝大多数通用业务场景下，量化带来的精度损失可控，能效收益极高。

## 3\.4 MoE稀疏架构

稀疏模型仅激活部分参数参与计算，能效优势显著。以Qwen3实测为例：30B总参数的MoE模型，单次推理仅激活3B参数，单token能耗比同规格32B稠密模型低3\.5倍。

优势边界：MoE仅优化计算量能耗，无法大幅降低内存带宽压力，在长上下文、小批量推理场景下，能效优势会明显收窄。

## 3\.5 软硬件栈迭代优化

- 硬件层面：Blackwell等新一代GPU架构，相比老旧显卡能效提升数百倍；

- 软件层面：vLLM、FlashAttention、TensorRT\-LLM等推理框架，可在硬件基础上进一步大幅提升推理能效。

## 3\.6 工程调度与业务场景

- 批量推理（大Batch）：拉高GPU利用率，摊薄单token平均能耗，大幅降低边际成本；

- 超长上下文：KV Cache占用大量显存，内存带宽压力陡增，推理效率下降、单token能耗显著上升。
