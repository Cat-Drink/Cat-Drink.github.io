# AI Agent项目实战踩坑总结（3个项目经验复盘）

## 一、项目前期：不要上手直接写代码

做 Agent 的第一步不是编写代码，而是完整设计任务思考流程。
在开发前必须梳理清楚整套逻辑闭环，核心自问问题：

1. Agent 接收到任务后，第一步执行什么动作？

2. 如何自主判断自身缺少哪些关键信息？

3. 依据什么规则决定调用哪一个工具（tool）？

4. 如何校验当前执行结果是否正确？

5. 判定任务结束、停止迭代的标准是什么？

逻辑流程、决策规则梳理清晰后，代码实现是最简单的环节；如果思考流程没有设计完整，哪怕代码实现再工整，最终 Agent 也无法稳定运行。

## 二、核心关键：工具（Tool）设计是 Agent 的灵魂

很多教程会误导开发者，认为 LLM 大模型选型是 Agent 核心，但实际落地后会发现：工具设计的优劣，对最终效果的影响远大于模型选择。

### 工具设计落地准则

1. **工具拆分原则**

    - 如果需要两三句话才能讲清一个工具的功能，说明工具职责冗余，需要拆分；

    - 工具总量超过 10 个时，需要对工具分组、合并归类，降低 LLM 选择难度。

2. **工具描述优先级高于代码实现**
Agent 完全依靠工具描述理解工具用途，描述模糊、歧义大，会直接导致频繁错选工具；同一段工具逻辑，更换描述文案后，整体运行效果会天差地别。

## 三、重难点：记忆与上下文管理

Agent 多次迭代运行后，对话上下文会持续膨胀，上下文超限是必须提前解决的核心问题，两种主流解决方案：

### 方案 1：简易摘要压缩（低成本，存在细节损耗）

每隔固定执行步数，调用 LLM 对全部历史对话生成精简摘要，用摘要替换原始完整对话。
优势：实现简单、控制上下文长度；
缺陷：会丢失历史对话里的细碎关键信息，复杂任务不适用。

### 方案 2：分层记忆架构（稳定生产级方案）

三层记忆分离，各司其职，避免长文本信息丢失：

1. **短期记忆**：存储当前对话最近几步完整原始信息；

2. **工作记忆**：结构化数据存储当前任务核心状态、中间结果；

3. **长期记忆**：向量数据库持久存储历史任务经验、过往案例。

### 底层核心思路

不要让 LLM 做它不擅长的事：从超长文本里精准提取结构化数据是 LLM 短板，人工维护好结构化任务状态，让 LLM 只专注推理、决策这类自身优势场景。

## 四、开发验收：评估环节绝对不能偷懒

### 落地方案：搭建标准化测试用例集

无需海量数据，二三十条覆盖各类业务场景的测试用例即可，每条用例固定包含三部分：

1. 用户输入指令；

2. 预期最终输出结果；

3. 合理标准中间执行步骤。

每次修改 Prompt、调整工具逻辑后，必须完整跑一遍全部测试集，无需自动化评估，人工核验即可。

- 正常成功用例快速浏览核对；

- 失败用例深度根因分析，定位四类典型故障：

    1. 工具选择错误

    2. 推理链条断裂、逻辑跳步

    3. 模型幻觉、编造不存在信息

    4. 上下文丢失关键任务信息

### 故障对应修复策略

|失败根因|修复优化方案|
|---|---|
|工具用错|修改工具描述 / 拆分冗余工具|
|推理链断裂|优化 System Prompt / 增加 Few\-shot 示例|
|模型幻觉|新增事实校验、信息核对步骤|
|上下文丢信息|调整分层记忆 / 摘要压缩管理策略|

## 五、主流开发框架实际使用体感

### 1\. LangChain

- 优点：生态最完善，各类工具、模型、数据库均可快速对接；

- 缺点：多层抽象封装，排错困难，故障难以定位层级；

- 适用场景：快速搭建 Demo 原型，不适合需要精细控制 Agent 行为的正式项目。

### 2\. LangGraph

- 对比 LangChain 体验更好，以图结构定义 Agent 状态流转；

- 优势：对任务执行流程、下一步动作拥有直观可控能力；

- 适用场景：目前新项目优先选择该框架。

### 3\. 自研底层架构

适合对 Agent 执行行为、流转逻辑有极致精细化管控需求的场景，完全自主实现状态、工具、记忆调度。

## 整体总结

三个 Agent 项目踩坑核心要点梳理：

1. 先设计思考链路，再写代码；

2. 工具设计＞模型选型，工具描述是重中之重；

3. 分层记忆解决上下文溢出问题，规避 LLM 短板；

4. 建立测试用例集做回归评估，按故障类型定向优化；

5. 框架选型：原型用 LangChain，可控需求优先 LangGraph，极致精细控制自主搭建。
