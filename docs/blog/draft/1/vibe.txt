Vibe Coding in Prod (Responsibly)
Vibe Coding 的定义  
Vibe Coding 并非简单使用 AI 生成代码（如 Copilot/Cursor），其核心定义参考 Andrej Karpathy：完全融入氛围，接受指数级增长，忘记代码的存在。与传统 AI 辅助编码的区别在于，Vibe Coding 中人与模型的反馈循环更松散，用户不再紧密参与每一行代码的生成与审查。
Vibe Coding 的优缺点  
优点：降低编码门槛，让非技术人员能独立构建应用（如视频游戏、副业项目等低风险场景），即使存在错误影响也较小。
缺点：新手缺乏技术认知，易导致问题（如 API 密钥使用超限、订阅绕过、数据库异常操作等）。
为什么需要关心 Vibe Coding  
核心原因是AI 能力的指数级增长：AI 可完成的任务长度每 7 个月翻一番，当前约 1 小时，未来将能生成一整天甚至一周的工作量。若仍依赖人工逐行审查，将无法跟上效率需求。类比编译器发展：早期开发者需验证汇编代码，而现在通过信任编译器实现规模化开发，Vibe Coding 将推动软件行业进入类似阶段。
负责任的 Vibe Coding 实践  
聚焦代码库的叶子节点
优先在无其他部分依赖的 “叶子节点”（最终功能、修饰性代码）使用 Vibe Coding，技术债务影响可控；核心架构与底层分支需人工深入理解，确保可扩展性与灵活性。
担任 AI 的 “产品经理”
提供充分指导：如同指导新员工，需明确需求、规格、约束及代码库背景（如相关文件、设计模式）。
投入前期准备：通过与 AI 对话探索代码库、制定计划，将关键信息整合为提示，提升 AI 生成质量。
强调可验证性
设计稳定性压力测试及易于人工验证的输入输出，即使不阅读全部代码，也能通过检查点确保正确性。
接受指数效应
当前不使用 Vibe Coding 影响有限，但未来 AI 将能生成更大工作量，若不适应将成为瓶颈。
问答环节要点  
学习方式的转变
担忧：减少手动编码可能降低基础能力；乐观：AI 可作为 “结对程序员”，通过审查代码、提问（如库选择原因）加速学习，且能快速尝试更多项目，积累经验。
测试驱动开发（TDD）
建议：要求 AI 编写通用的端到端测试（覆盖正常路径、错误情况），避免过于特定于实现；优先审查测试用例，确保测试通过即可信任代码。
指数增长的理解
指数不仅是 “持续变好”，而是 “加速变好”：如计算机内存从 KB 级到 TB 级的飞跃，未来 AI 能力可能呈数百万倍提升，需提前适应这种变化。
工作流程建议
结合 Claude Code 与 Cursor：Claude Code 用于整体规划，Cursor 用于具体修复；在关键节点压缩会话（如完成计划后），减少冗余上下文。
安全与效率平衡
生产环境需技术背景：非技术人员适合低风险场景，生产系统需具备判断危险与安全的能力；未来可通过框架隔离核心功能（如支付系统），仅允许 Vibe Coding 填充 UI 层。









Harness Engineering 驾驭工程
Vibe Coding 的局限性  
适用场景：简单任务（如生成带登录功能的财务仪表盘）可自动写代码并部署。
局限性：处理大型项目（几十万行代码）时 AI 易失控，表现为忘记指令、忽略规范，修复 bug 成本高于重写项目。
原因：AI 注意力有限，面对事无巨细的指令时无法区分重点，导致所有要求都被视为非重点。
Harness Engineering 的核心思路  
定义：通过 “制度约束” 控制 AI 行为，让 AI 在规则内发挥最大能力，而非依赖更优 AI 或提示词。
类比：类似智能汽车需实时索引交通规则，而非一次性学习所有规则，避免注意力混乱。
Harness Engineering 的实施方法  
架构约束  
形式：项目根目录下的 AGENTS.md 文件，作为 AI 的第一份指南。
特点：非详细操作手册，而是 “目录”，指引 AI 按需检索具体操作指南，避免信息过载。
依赖关系  
内容：明确项目文件夹结构、各文件夹作用及相互依赖关系。
作用：AI 可自我理解违规原因并纠偏，避免误删文件夹、重复新增文件等问题，无需人类介入。
反馈循环  
机制：AI 智能体本地审核自身修改，通过运行自生成测试用例反复审查，直至通过。
修复：检测到代码无法通过测试用例时，AI 自动根据报错修复 bug。
垃圾回收  
问题：软件系统随时间混乱，技术文档过时。
解决：定时运行 AI agent 检测文档与代码的不一致，自动修复过时内容。



Harness Engineering 详解
Harness Engineering 的定义与核心价值  
Agent 可靠性的瓶颈不在模型，而在模型周围的系统。模型是引擎，Harness 是方向盘。以 LangChain Coding Agent 为例，在 Terminal Bench 排行榜中，通过仅优化 Harness（系统提示、工具配置、中间件钩子），模型未更换的情况下，排名从 30 名开外提升至前五。Harness 源自马具（缰绳、马鞍等）的比喻：模型如马（强大但无方向），人类工程师如骑手（提供方向），Harness 则是连接两者的控制系统。其核心定义为：通过设计解决方案避免 Agent 重复犯错，公式为 Agent=Model+Harness，模型提供智能，Harness 让智能变得有用。
Harness Engineering 与 Context Engineering 的区别  
Context Engineering 关注 “给 Agent 看什么信息、何时看”，管理上下文窗口；Harness Engineering 范围更广，关注系统 “预防了什么、测量了什么、修复了什么”，包括架构约束、自验证循环、熵治理、系统可演进性等，两者互相包含。
Harness 的六个核心组件  
上下文架构  
Agent 性能在上下文利用率超过 40% 后会下降，关键是提供 “地图” 而非 “百科全书”。例如 OpenAI 将 AGENTS.MD 控制在约 100 行作为目录，指向深层文档；Anthropic 采用 Skill 渐进式加载理念，避免信息过载。
架构约束  
仅靠 Prompt 约束 Agent 行为不可靠（模型可不听从），前沿团队用确定性工具（如自定义 linter、结构化测试）机械执行约束。例如 Vercel 移除 80% 冗余工具后，Agent 因解空间受限反而更快更可靠。
自验证循环  
解决 Agent 死循环（反复编辑同一文件）和跳过验证问题。通过中间件钩子：跟踪文件编辑次数（超阈值提醒重新审视）、退出前强制执行验证。“推理三明治” 策略：规划阶段最高推理强度理解问题，执行阶段降为高等保证速度，验证阶段拉回最高捕获错误，避免全程高推理强度导致超时。
上下文隔离  
多 Agent 协作时，父 Agent 仅看到给子 Agent 的指令和最终结果，中间工具调用和产物被隔离，保持各执行单元上下文干净，避免无关信息污染。
熵治理  
Agent 运行时间越长，系统混乱度越高（文档过时、架构漂移等）。OpenAI 引入后台文档梳理 Agent，定期扫描并自动修复过时文档，形成自维护闭环。
可拆卸性  
Harness 需模块化、可拆卸，以适应模型演进。如 LangChain 中间件架构，各组件独立添加 / 移除，不影响其他部分，避免未来模型进步使现有组件成为瓶颈。
Harness Engineering 的投资回报与实践原则  
Harness 投入呈复利效应：添加 linter 规则可预防所有会话中的同类错误，验证中间件提升所有任务交付质量，投入越早累积收益越大。但需避免过度工程化，原则是 “只在 Agent 确实犯过的错误上投入 Harness，不解决未出现的问题”。
总结  
Harness Engineering 核心主张：Agent 可靠性瓶颈不在模型，而在模型周围的系统。模型是引擎，Agent 是整辆车，Harness 是方向盘和刹车，没有 Harness，再强的引擎也无法到达目的地。





AI 编程的 Agentic Engineering 方法论
Simon Willison 与 Agentic Engineering  
Simon Willison 是 Django 创始人，被称为全球 AI 编程实践最深的人之一。其提出的 Agentic Engineering 方法论强调结构化、测试驱动的 AI 编程模式，区别于仅凭感觉的 Vibe Coding。他发起了新项目 “Agentic Engineering Patterns”，旨在探索 AI 编程时代的有效实践方法。
Vibe Coding vs Agentic Engineering  
Vibe Coding 指依赖 AI 生成代码但缺乏验证的编程方式，而 Agentic Engineering 的核心是 “测试优先”—— 无论 AI 生成的代码多么复杂，无法通过测试则不被接受。
TDD 成为必须的原因  
先写测试的本质是明确 “我要什么”，即通过测试用例定义功能的验收标准。Red/Green TDD 三步流程是其核心：先编写测试（红灯，预期失败），再编写代码使测试通过（绿灯），最后优化代码。
First Run the Tests 的三大好处  
让 AI 明确知道需要通过测试，促使其自动考虑测试兼容性；
测试数量能帮助 AI 判断项目规模，调整生成策略；
引导 AI 主动生成测试用例，形成 “测试 - 代码” 闭环。
AI 编程时代的核心竞争力  
AI 能在几分钟内生成数千行代码（如 1 人 + AI 一周重写 Next.js 框架，成本 1100 美元），代码产量不再是瓶颈，质量保障成为关键。此时 “定义什么是对” 的能力（即编写好的测试、明确验收标准）成为稀缺资源，Red/Green TDD 正是这一能力的核心实践。
总结  
AI 编程时代，写代码的能力不再值钱，能定义标准的人才更有价值。Red/Green TDD 通过 “先写测试 - 红灯确认 - 绿灯通过” 的简单流程，确保 AI 产出高质量成果，是代码免费时代的核心竞争力。




AI 大模型核心知识笔记
AIGC 和生成式 AI  
AIGC（AI Generated Content）：指 AI 生成的内容，包括文字、图片、音频、视频等，如 DeepSeek 生成文章、TRAE 生成代码、即梦生成图片等。
生成式 AI（Generative AI）：海外更流行的术语，生成式 AI 生成的内容即为 AIGC。国内语境下 AIGC 常被用来指代生成式 AI。
概念关系：AI（人工智能）是计算机科学学科；机器学习是 AI 的子集，核心是让计算机通过算法自行学习；机器学习包含监督学习、无监督学习、强化学习；深度学习是机器学习的方法，使用人工神经网络；生成式 AI 是深度学习的应用，大语言模型（LLM）是深度学习在自然语言处理的应用。
大语言模型  
定义：LLM（Large Language Model），用于自然语言相关任务的深度学习模型，可完成生成、分类、总结、改写等任务。
“大” 的含义：不仅训练数据量大（如 GPT-3 训练数据含 3000 亿 TOKEN），更指参数数量巨大（GPT-1 有 1.17 亿参数，GPT-3 达 1750 亿参数）。
能力特点：参数增加使模型具备广泛能力，可替代多个单项任务模型，如豆包、DeepSeek、ChatGPT 等聊天助手均基于大语言模型。
Transformer 架构  
提出背景：2017 年谷歌论文《Attention Is All You Need》提出，改变自然语言处理方向，GPT、BERT 等模型均基于此。
核心创新：
自注意力机制：处理每个词时关注输入序列中所有词，通过计算相关性确定注意力权重，解决长距离语义依赖问题。
位置编码：将词的位置信息转化为向量并与词向量相加，使模型理解词序，支持并行计算，提升训练效率。
结构组成：包含编码器（Encoder）和解码器（Decoder）。编码器将输入转换为抽象表示，解码器基于此生成输出。变种包括仅编码器模型（如 BERT，擅长语言理解）、仅解码器模型（如 GPT，擅长文本生成）、编码器 - 解码器模型（如 T5，适用于翻译、总结等序列转换任务）。
大语言模型的训练  
预训练：使用海量无监督文本数据（如书籍、新闻、论文等）训练基座模型，模型通过预测下一个 TOKEN 学习语言规律，耗时耗力（如 GPT-3 训练数月，使用上千 GPU）。
监督微调（SFT）：用人类撰写的高质量对话数据进行监督学习，使模型具备对话能力，成本低于预训练。
强化学习（RLHF）：训练奖励模型（基于人类对回答的质量排序），通过奖励模型对 SFT 模型生成的回答评分，以评分作为反馈优化模型，提升回答质量。
提示工程  
小样本提示：提供示例引导 AI，利用上下文学习能力让 AI 模仿示范回应，无需额外要求即可遵循风格。
思维链（Chain of Thought）：在提示中展示推理步骤，使 AI 生成中间过程，提升复杂推理（如数学计算）能力，简单方法可在问题后加 “Let's think step by step”。
推理模型：如 OpenAI 的 O1/O3、DeepSeek 的 RE 等，训练阶段强化多步推理能力，可自行进行内部链式推理后输出结果。
外挂  
RAG（检索增强生成）：将外部文档分段转为向量存入数据库，提问时匹配相关段落作为上下文，使 AI 结合外部知识生成准确回答，适用于企业 / 个人知识库。
PAL（程序辅助语言模型）：让 AI 生成计算所需代码，通过代码解释器执行得到结果，解决 AI 计算能力不足问题。
ReAct 框架：结合推理（Reason）与行动（Action），通过 “推理 - 行动 - 观察” 循环与外部工具（如搜索引擎）交互，获取实时信息，动态解决问题。















nthropic 战略转向：从 Agent 到 Skills 的范式转变
行业痛点：智能体动物园的形成原因  
Agent 终局：巨头博弈与管理陷阱  
行业共识是当前 agent 的核心问题已从 “能否实现” 转向 “能否规模化交付”。OpenAI 的管理模式类似 “带初级员工”，导致管理成本转移到人类；Google 则将 agentic 能力深度集成到产品（如 Gemini agent），使其成为工作流核心组件。agent 越接近员工角色，越需要组织级流程规范与质检标准，否则会导致 “返工自动化”。
一致性执行 > 随机灵感  
企业场景中，agent 普遍存在 “聪明但不专业” 的问题，缺乏 expertise 且易 miss context。通用模型擅长通用推理，但无法精准匹配企业特定的字段定义、审批流程、合规口径等专业规则。真实业务中，一致性执行比灵感更重要，需要的是 “专业流程训练过的执行者” 而非 “300IQ 的天才”。
维护噩梦：独立 Agent 导致的熵  
多 Agent 系统会引发 “智能体动物园” 现象：每个 Agent 有独立的 prompt、工具链、权限、评测和失败兜底机制，导致知识碎片化（同规则多版本）、上下文窗口占用（长期 TOKEN 成本上升）、调试困难。核心问题在于通过复制粘贴扩展能力会导致维护成本指数级增长。
Anthropic 的 Skills 范式：从造人到沉淀技能资产  
Skills 的核心定义与价值  
Claude skills 是 “为 agent 打包的可组合流程性知识”，具备三个关键特征：流程性（强调 “怎么做”，如财务报表的拉数、校验、汇总、套模板、出具步骤）、可组合（如数据分析 + 合规审查 + 报表格式化技能叠加）、可执行（模型负责规划 / 选择 / 解释，脚本负责计算 / 转换 / 生成文件）。其本质是将组织隐性知识转化为可版本化、可复用、可执行的显性资产。
Skills 的工程化结构  
技能核心入口为 “skill.md”，包含三件套：元信息（名称、描述，启动时预加载）、执行指南（步骤、边界条件、输入输出，触发时加载）、配套资源（脚本、参考文档、模板文件，按需读取 / 执行）。支持 Git 版本管理、审批和回滚，实现 “像管代码一样管业务流程”。
隐喻：从 “专岗机器人” 到 “入职手册”  
旧范式：为每个岗位造专属 Agent（如财务 Agent、法务 Agent），导致维护爆炸；新范式：通用 Agent + 不同 “入职手册”（Skills），流程变更只需更新手册，实现知识同步。核心价值在于沉淀高质量 “入职手册”（skill library）。
Skills 的架构优势与工程实现  
智能与专业经验的解耦  
多 Agent 问题根源在于系统缺乏软件工程化设计，知识复制粘贴导致维护困难。Skills 范式通过 “通用 Agent 内核 + 技能包” 实现解耦：通用 Agent 负责理解、规划、调用；Skills 负责流程、规范、脚本和资料。扩展能力的单位从 “新增 Agent” 变为 “新增 Skill 包”，类似 “操作系统 + 插件生态”。
渐进式上下文加载策略  
解决上下文窗口限制的核心是 “加载策略” 而非容量：第一层加载元数据（技能名、描述），使 Agent 知晓可用技能；第二层触发时加载 skill.md（具体流程和指令）；第三层按需加载额外文件（如 reference.md、forms.md）。实现技能库无限扩展与上下文低占用的平衡，适配企业海量、分散、动态更新的知识场景。
代码执行：模型与 Runtime 的分工  
Skills 允许将脚本嵌入技能目录，Agent 按需运行：脚本负责计算、排序、抽取、格式化等确定性任务，模型负责理解、决策、编排、解释。优势包括降低成本（避免模型生成代码）、提升确定性（企业需可重复执行而非灵感），实现 “模型做决策，Runtime 做执行” 的高效分工。
MCP 与 Skills 的分层协作  
MCP（模型连接协议）负责工具和数据源连接（如 CRM、网盘、数据库），Skills 负责工作流和规范（步骤、边界、检查点、输出格式）。典型任务流程为：MCP 拉取数据→Skills 处理（分析口径、建模流程、报表模板）→MCP 写回结果。架构上需严格分离连接能力（MCP）与业务方法（Skills）。
Skills 范式下的组织分工重构  
AI 产品 PM：从功能列表到技能资产组合  
核心转变是从 “部门级 Agent 需求” 转向 “关键流程 Skill 拆分”，关注三个指标：覆盖面（覆盖关键 SOP 数量）、复用率（跨部门 / 跨工作流复用能力）、质量标准（输入 / 输出 / 检查点 / 失败兜底的可执行验收口径）。AI 产品护城河正从模型能力转向 “技能资产密度”。
架构师：企业治理能力  
Skills 可插拔特性要求解决三个治理问题：谁能安装 / 使用技能、谁能修改 / 审批 / 回滚技能、如何规避代码执行风险。平台需补齐四类能力：skill registry（目录、版本、依赖）、权限模型（团队 / 角色 / 环境分级）、发布流水线（测试、灰度、回滚）、可观测性（触发率、失败率、成本、风险）。
开发者：知识工程的崛起  
工作重心从调模型 / 堆 prompt 转向知识工程，具体包括：编写可执行 SOP（skill.md，含步骤、边界、输入输出）、开发确定性脚本（抽取、校验、格式化）、上下文编排（分层设计常驻元数据与按需加载细节）。需与领域专家深度协作，将口头经验转化为技能资产。
业务专家：技能开发的参与者  
非技术角色（会计、法务、招聘）可直接参与技能编写，实现 “懂流程的人沉淀流程”。工程团队提供模板、规范、工具、评测、治理支持，双方共创实现技能 “快、稳、可审计”。Skills 类比 “AI 的 App Store 时刻”，核心在于开放供给侧。
Skills 的挑战与未来生态  
标准化瓶颈  
跨平台通用的关键是格式统一，否则技能会成为 “新一代 prompt 方言”（字段不兼容、触发策略差异、工具权限冲突）。判断标准包括：技能格式能否被其他 Agent 读懂、能否兼容下一代模型。需优先治理字段规范、目录规范、版本兼容策略。
企业级安全体系  
技能含脚本时风险从内容升级为执行风险，需构建 “审核 - 权限 - 审计” 三件套：审核（上线前代码 Review，重点检查依赖与外部资源）、权限（按环境分级，开发 / 生产环境隔离）、审计（记录触发者、时间、动作，确保可追溯）。核心原则是 “只从可信来源安装”。
能力边界  
主要限制包括：跨任务迁移难（流程情境化，如财务技能换企业后口径 / 系统变化）、技能选择与误用风险（错误流程会执行到底）、底座模型推理局限（复杂任务依赖模型规划与异常处理能力）。建议设计阶段明确适用范围与不适用场景，通过评测压测误触发率。
未来生态：标准化与市场化  
趋势是 “标准化→市场化”：Linux 基金会成立 AAI-F 推动互操作标准，避免重复造生态；标准收敛后将出现技能目录、市场、应用商店。企业技能分化为通用型（如通用报表）与私有型（内部 SOP / 合规），从业者竞争力转向 “领域经验→可迁移 / 可审计 / 可组合技能包” 的转化能力。
结论：技能资产是新的核心竞争力  
未来 AI 竞争的关键不是 Agent 数量，而是 “组织经验→可复用 / 可治理 / 可分发 Skills” 的沉淀能力。模型通用性提升后，支持组织知识的结构化方式将成为核心壁垒。nthropic 战略转向：从 Agent 到 Skills 的范式转变
行业痛点：智能体动物园的形成原因  
Agent 终局：巨头博弈与管理陷阱  
行业共识是当前 agent 的核心问题已从 “能否实现” 转向 “能否规模化交付”。OpenAI 的管理模式类似 “带初级员工”，导致管理成本转移到人类；Google 则将 agentic 能力深度集成到产品（如 Gemini agent），使其成为工作流核心组件。agent 越接近员工角色，越需要组织级流程规范与质检标准，否则会导致 “返工自动化”。
一致性执行 > 随机灵感  
企业场景中，agent 普遍存在 “聪明但不专业” 的问题，缺乏 expertise 且易 miss context。通用模型擅长通用推理，但无法精准匹配企业特定的字段定义、审批流程、合规口径等专业规则。真实业务中，一致性执行比灵感更重要，需要的是 “专业流程训练过的执行者” 而非 “300IQ 的天才”。
维护噩梦：独立 Agent 导致的熵  
多 Agent 系统会引发 “智能体动物园” 现象：每个 Agent 有独立的 prompt、工具链、权限、评测和失败兜底机制，导致知识碎片化（同规则多版本）、上下文窗口占用（长期 TOKEN 成本上升）、调试困难。核心问题在于通过复制粘贴扩展能力会导致维护成本指数级增长。
Anthropic 的 Skills 范式：从造人到沉淀技能资产  
Skills 的核心定义与价值  
Claude skills 是 “为 agent 打包的可组合流程性知识”，具备三个关键特征：流程性（强调 “怎么做”，如财务报表的拉数、校验、汇总、套模板、出具步骤）、可组合（如数据分析 + 合规审查 + 报表格式化技能叠加）、可执行（模型负责规划 / 选择 / 解释，脚本负责计算 / 转换 / 生成文件）。其本质是将组织隐性知识转化为可版本化、可复用、可执行的显性资产。
Skills 的工程化结构  
技能核心入口为 “skill.md”，包含三件套：元信息（名称、描述，启动时预加载）、执行指南（步骤、边界条件、输入输出，触发时加载）、配套资源（脚本、参考文档、模板文件，按需读取 / 执行）。支持 Git 版本管理、审批和回滚，实现 “像管代码一样管业务流程”。
隐喻：从 “专岗机器人” 到 “入职手册”  
旧范式：为每个岗位造专属 Agent（如财务 Agent、法务 Agent），导致维护爆炸；新范式：通用 Agent + 不同 “入职手册”（Skills），流程变更只需更新手册，实现知识同步。核心价值在于沉淀高质量 “入职手册”（skill library）。
Skills 的架构优势与工程实现  
智能与专业经验的解耦  
多 Agent 问题根源在于系统缺乏软件工程化设计，知识复制粘贴导致维护困难。Skills 范式通过 “通用 Agent 内核 + 技能包” 实现解耦：通用 Agent 负责理解、规划、调用；Skills 负责流程、规范、脚本和资料。扩展能力的单位从 “新增 Agent” 变为 “新增 Skill 包”，类似 “操作系统 + 插件生态”。
渐进式上下文加载策略  
解决上下文窗口限制的核心是 “加载策略” 而非容量：第一层加载元数据（技能名、描述），使 Agent 知晓可用技能；第二层触发时加载 skill.md（具体流程和指令）；第三层按需加载额外文件（如 reference.md、forms.md）。实现技能库无限扩展与上下文低占用的平衡，适配企业海量、分散、动态更新的知识场景。

非技术角色（会计、法务、招聘）可直接参与技能编写，实现 “懂流程的人沉淀流程”。工程团队提供模板、规范、工具、评测、治理支持，双方共创实现技能 “快、稳、可审计”。Skills 类比 “AI 的 App Store 时刻”，核心在于开放供给侧。
Skills 的挑战与未来生态  
标准化瓶颈  
跨平台通用的关键是格式统一，否则技能会成为 “新一代 prompt 方言”（字段不兼容、触发策略差异、工具权限冲突）。判断标准包括：技能格式能否被其他 Agent 读懂、能否兼容下一代模型。需优先治理字段规范、目录规范、版本兼容策略。
企业级安全体系  
技能含脚本时风险从内容升级为执行风险，需构建 “审核 - 权限 - 审计” 三件套：审核（上线前代码 Review，重点检查依赖与外部资源）、权限（按环境分级，开发 / 生产环境隔离）、审计（记录触发者、时间、动作，确保可追溯）。核心原则是 “只从可信来源安装”。
能力边界  
主要限制包括：跨任务迁移难（流程情境化，如财务技能换企业后口径 / 系统变化）、技能选择与误用风险（错误流程会执行到底）、底座模型推理局限（复杂任务依赖模型规划与异常处理能力）。建议设计阶段明确适用范围与不适用场景，通过评测压测误触发率。
未来生态：标准化与市场化  
趋势是 “标准化→市场化”：Linux 基金会成立 AAI-F 推动互操作标准，避免重复造生态；标准收敛后将出现技能目录、市场、应用商店。企业技能分化为通用型（如通用报表）与私有型（内部 SOP / 合规），从业者竞争力转向 “领域经验→可迁移 / 可审计 / 可组合技能包” 的转化能力。
结论：技能资产是新的核心竞争力  
未来 AI 竞争的关键不是 Agent 数量，而是 “组织经验→可复用 / 可治理 / 可分发 Skills” 的沉淀能力。模型通用性提升后，支持组织知识的结构化方式将成为核心壁垒。




规格驱动开发与 AI 协同编程实践
flag 术语翻译  
flag 最初译为 “规范”，后因强调与 AI 协同改为 “规格”。
AI 作为开发助手的定位  
将 AI 视为能力强但需明确指令的 “实习生”，需通过充分上下文实现有效协同，而非依赖其自主完成任务。
AI 编程常见问题及解决思路  
问题：AI 生成代码常出现 “偏离需求”（如 “开到绿化带”），反复修复仍无法解决
解决思路：提供验证机制（类比开源社区 PR 测试流程），通过明确上下文和交互对齐目标
规格驱动开发（SDD）核心概念  
定义：以规格（spec）为事实唯一来源，而非代码
价值：解决文档与代码不一致问题，支持 AI 迭代生成代码，便于团队协作和需求追溯
关键实践：将 prompt、plan 等交互记录视为核心资产，通过版本控制工具（如 git）管理
SDD 与传统开发流程对比  
传统流程：需求→评审→设计→编码→集成→测试
SDD 流程：在设计阶段引入 AI 协同，通过 spec 定义完整上下文，覆盖 PRD、接口文档、数据库设计等，实现全流程 AI 辅助
SPEC KIT 工具介绍  
功能：从 0 到 1 生成项目接口描述、测试用例，支持前后端协同
核心命令：
定义规则（宪法 /constitution）
提需求（spec）
制定技术实现计划（plan）
分解任务（task）
执行生成（implement）
原理：通过模板和脚本适配不同开发平台，将 AI 规则编码化
OPEN SPEC 工具介绍  
定位：轻量级规格管理工具，适用于小功能迭代（1 到 N 场景）
核心流程：
提案（proposal）：描述需求和修改范围
应用（apply）：生成执行计划并调用 AI 实现
归档：记录变更历史，支持团队协作追溯
优势：目录结构简单，支持自然语言交互修改任务，集成测试验证
SDD 实施段位划分  
基础段位：人机协作编写 spec 生成代码，易出现文档丢失
黄金段位：保留 spec 资产，人工介入验证，类似 L2 自动驾驶
理想段位：专注 spec 设计，多模型竞争生成代码，通过测试自动验证（L3 自动驾驶）
AI 协同开发最佳实践  
上下文管理：提供清晰的项目背景、技术栈和验收标准
任务分解：将需求拆分为明确、可验证的小任务
测试驱动：通过测试用例定义成功标准，确保 AI 输出符合预期
资产沉淀：使用 git 等工具管理 spec、prompt 和交互历史，支持复用和迭代









Agent 上下文管理：信息取舍与工程实践
Agent≠调 API 的核心区别  
面试中常被质疑 “Agent 只是调大模型 API”，但实际落地面临关键挑战：当对话轮次增加（如 50 轮），上下文 token 超过模型窗口（如 128K）时，需解决信息取舍问题，而非简单删除或全量发送。
上下文窗口限制与落地痛点  
大语言模型上下文窗口存在硬限制（常见 128K-20 万，Gemini 达 100 万），但实际交互中：
固定开销：System Prompt + 工具说明约 4000 tokens
动态开销：每轮对话（输入 + 输出）500-2000 tokens、Agent 思考过程 200-1000 tokens、工具调用记录 200-500 tokens
50 轮交互后易突破窗口限制，导致三大痛点：API 成本高、首 token 延迟慢、模型 Lost in the Middle（中间信息丢失）问题，最终体现为 “贵、慢、不准”。
历史消息分类与处理策略  
根据信息复用概率，历史消息分为四类，处理方式不同：
用户指令（如 “帮我写支持代理的爬虫”）：原文保留，不压缩。
关键状态（如 “已完成登录模块，下一步大数据解析”）：结构化摘要。
中间推理（如 “先试方案 a，不行换方案 b”）：激进摘要，仅保留结论。
失败记录（如 “用 request 库报错 403”）：保留失败原因，删除过程。
信息价值评估与打分机制  
核心思路是对每条消息打分，按分数决定处理方式：
打分规则：用户指令（0.4 分）、包含关键决策（0.3 分）、任务状态变更（0.2 分）、时间衰减（0.1 分）。
处理逻辑：当上下文 token 超过阈值（如窗口的 80%）触发摘要：
高分（>0.6）：保留原文；
中分（0.3-0.6）：生成摘要；
低分（<0.3）：激进压缩或丢弃。
摘要时机与成本优化  
摘要时机：触发阈值设为窗口的 80%，预留 20% token 用于摘要本身（摘要 prompt 约 2000 tokens，输出 1-2000 tokens），避免摘要过程撑爆窗口。
成本对比：100 轮对话下，不做摘要累计 500 万 tokens，做摘要累计 305 万 tokens（含 10 次摘要调用），节省 40%；若用小模型（如 GPT-4o-mini）做摘要，成本更低。
Agent 的技术含量：四大核心问题  
信息价值评估：定义 “有价值信息”，非简单删除。
Token 预算分配：有限资源下的最优分配策略。
Token 质量控制：多轮摘要可能导致信息失真，需校验机制。
触发时机优化：平衡成本与效果。
类比：上下文管理类似 OS 的虚拟内存管理，是复杂的工程系统难题。













Vibe Coding 实践指南
Vibe Coding 定义  
Vibe Coding 可简单理解为将模糊想法转化为 AI 能理解的描述，进而让 AI 协助实现的过程。
Vibe Coding 工具选择  
入门工具：ChatGPT、Gemini 的 canvas，特点是通过聊天窗口 + 画布，用 prompt 将想法转化为画面，门槛低。
进阶工具：Google 的 AI studio、Kimi 的 visual coding，优势在于免费，能帮助快速了解完整网站构建，更考验 Prompt 输出能力，推荐作为练习起点。
复杂项目工具：Coser、Clark 等 AI 编程工具，可导入下载的项目，能记住项目架构，处理复杂编程问题，操作不复杂。
Vibe Coding 工作流建立  
工作流概念：避免直接让 AI 生成代码导致后期 bug，需建立包含角色代入、需求探索、方案制定、执行阶段的工作流。
工作流文件管理：创建 commands 文件夹存放工作流文件，将工作流封装成 skill，便于 AI 自动搜索和应用，例如开发 YouTube 视频下载工具时，AI 可基于 skill 自动执行功能开发工作流。
约束 AI 发挥的方法  
重要性：防止 AI 为实现功能篡改原有代码，需先建立约束文档。
文档建立：生成产品雏形后，创建 Theodor.md 或 Agent.md 文档，包含技术栈、数据结构、页面布局等内容，AI 基于该文档开发，可避免 bug 和技术性问题。
文档使用：让 AI 生成文档后检查并补充遗漏内容。
代码检查技巧  
无编程经验者方案：准备 code review 文档，完成功能开发后交给 AI 进行代码检查，可发现可维护性等小问题，避免改乱代码。











Agent Skills 核心知识笔记
Agent Skills 定义  
Agent Skills 是 Anthropic 推出的开放标准，目的是让 AI 学习使用各种专业技能，无需每次重复输入提示词。本质是给 AI 装备的技能包，包含精心设计的提示词、代码脚本及各类资源文件，能让 AI 快速掌握特定领域能力（如文档处理、代码规范等）。
Agent Skills 使用方法  
以 Claude Code 为例：
添加官方技能市场：输入命令开通技能商店，获取各类技能。
安装技能包：如 example-skills（含前端设计、网页测试、动图制作等示例技能）。
使用效果：安装 frontend-design 技能后，生成网站时 AI 会主动询问是否使用该技能，生成独特风格页面，告别蓝紫渐变；安装文档处理技能包（含 PPT 制作、Excel 数据分析等），可直接生成排版好的文件，节省时间。
技能包结构  
核心是包含 SKILL.md 文件的文件夹，可含脚本、资源和参考文档，结构因复杂度而异：
SKILL.md 是核心，包含两部分：元数据（YAML 格式，含 name、description，描述越清晰 AI 越易调用）和指令内容（精心设计的提示词，指导 AI 具体操作，如前端设计技能包含设计思考、美学指南等）。
示例：PPT 制作技能含 SKILL.md、脚本、参考文档和资源文件；frontend-design 技能仅含 SKILL.md。
核心机制：渐进式披露  
AI 执行任务时，先扫描技能目录，仅读取各技能元数据，匹配任务后加载完整技能说明文档及所需资源，实现精准匹配且节省上下文空间，避免占用过多 tokens。
支持工具与社区  
支持工具：Claude Code、Cursor、VS Code、Codex 等。
技能获取：Claude Skills Hub 市场、开源的 Awesome Claude Skills 等，如 UI UX Pro MAX 技能可提升 AI 设计能力。
使用方式：安装官方命令行工具，进入项目目录执行对应命令，AI 可自动识别或通过斜杠命令手动触发技能。
开发方法  
基础方法：复制官方技能包，修改目录名、SKILL.md 元数据和指令内容，添加公司资源（Logo、模板等）。
便捷工具：使用 example-skills 中的 Skill Creator 技能，通过回答问题（周报主要部分、输出格式等）自动生成技能包，生成.skill 压缩文件，可解压到个人或项目技能目录，支持 Git 同步和社区分享。
与 MCP、斜杠命令的区别  
MCP：类似 AI 的 “手和眼睛”，连接外部工具和数据源（如搜索网页、读取代码仓库），适合获取数据或操作外部系统。
Agent Skills：类似 “工作手册”，打包专业知识和工作流程，教 AI 在特定领域操作。
斜杠命令：手动输入的快捷键，触发固定操作；Skills 可由 AI 自动识别调用。
结合使用：如发周报时，MCP 获取任务数据，Skills 整理成指定格式。
Agent Skills 火爆原因  
开放标准：封装一次可在多 AI 工具复用，支持社区共享。
降低门槛：无需学习提示词工程和工具链配置，安装技能包即可让 AI 变专业，让普通用户无感享受技术价值，体现技术走向大众的关键是降低使用门槛。








Agent Skill 详解
Agent Skill 概念  
Agent Skill 可理解为大模型的说明文档，能定义任务规则，避免重复输入要求。例如智能客服场景中，可在 Skill 里规定遇到投诉需先安抚用户情绪且不得随意承诺；会议总结场景中，可要求按参会人员、议题、决定的格式输出。实际功能远不止于此，具有高级特性。
基本使用方法  
以 Claude Code 演示会议总结场景的 Agent Skill 使用。首先在用户目录下的.claude/skill 文件夹创建 Agent Skill，进入该文件夹后执行 mkdir 会议总结助手创建文件夹，此文件夹名即 Agent Skill 名称。接着用 VSCode 打开文件夹，创建 skill.md 文件并填写内容，该文件用于描述 Agent Skill 的名称、功能及操作方式，包含元数据（name 和 description，name 需与文件夹名相同，description 说明用途）和指令（详细描述模型需遵循的规则，如规定总结参会人员、议题、决定等内容，还可举例说明输入输出格式）。创建完成后，在 Claude Code 中可查看已有的 Agent Skill，输入总结会议内容的请求并粘贴会议录音文本，Claude Code 会询问是否使用对应的 Agent Skill，同意后便会按规则生成总结。
高级用法：Reference  
Reference 实现按需中的按需加载，即条件触发加载相关文件。比如会议总结助手中，当会议内容涉及钱、预算、采购、费用等时，才加载财务规定文件。操作时，先添加集团财务手册.md 文件，写明费用报销标准等内容，再在 skill.md 文件中新增财务提醒规则，规定触发条件及读取该文件后需执行的操作（如指出金额是否超标、明确审批人）。使用时，若会议内容涉及相关触发条件，Claude Code 会请求读取对应 Reference 文件，同意后便会根据文件内容在总结中添加相应提醒。
高级用法：Script  
Script 用于让 Agent Skill 运行代码实现自动化操作。例如创建 upload.py 脚本用于上传文件，在 skill.md 中添加上传规则描述，当用户提到上传、同步或发送到服务器等字眼时，运行该脚本将总结内容上传到服务器。使用时，Claude Code 会在处理请求过程中，根据规则请求执行对应的脚本，同意后便会执行并反馈结果。Script 仅被执行，不占用模型上下文 token，Claude Code 只关注脚本的运行方法和结果。
渐进式披露机制  
Agent Skill 的渐进式披露结构分为三层，加载机制不同。第一层是元数据层，包含所有 Agent Skill 的名称和描述，始终加载，相当于目录，大模型每次回答前会查看以匹配用户问题与 Agent Skill。第二层是指令层，即 skill.md 文件中除元数据外的部分，按需加载，只有当大模型发现用户问题与某个 Agent Skill 匹配时才加载。第三层是资源层，包含 Reference 和 Script，是按需中的按需加载。Reference 被读取并加载内容到上下文，会消耗 token；Script 被执行，不占用上下文。
Agent Skill 与 MCP 的区别和联系  
官方观点：MCP connects Claude to data，Skills teach Claude what to do with that data。即 MCP 为大模型供给数据，如查询销售记录、获取物流状态等；Skill 教会大模型处理数据，如规定会议总结格式、汇报文档包含内容等。Agent Skill 虽能写代码连接数据，但在代码执行的安全性和稳定性上不及 MCP，更适合处理轻量脚本和简单逻辑，实际场景中可结合使用。
