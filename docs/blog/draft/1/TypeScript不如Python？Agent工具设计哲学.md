# TypeScript 不如 Python？Agent 工具设计哲学

## 一个思想实验

> 假如你是老板，面前有两个员工：
> - 对 A 说：用这十种工具完成工作
> - 对 B 说：把事搞定，别超预算，别违规
>
> 谁更像聪明人？你更愿意把任务交给谁？

这个问题的答案，就是 Agent 工具发展的分水岭。

## 当前 Agent 工具的问题

- 为了让 Agent 发挥期望的效果，不断增加机制与约束
- 本质不是利用大模型的**创造力**，而是基于工程师认知的高阶辅助工具
- **设计得越精密的 Agent 工具，可能越在扼杀大模型的创造力**

## 两种设计哲学

|  | 约束型 | 放手型 |
|------|--------|--------|
| 做法 | 精密约束、状态机、接口说明 | 只给目标 + 安全边界 |
| 结果 | AI 解决问题的难度随工具数量线性增长 | AI 自己决定怎么达成 |
| Token 消耗 | 较低 | 较高，但集中在高价值推理 |

## 核心问题

> 工程惯性 vs 产品思考

- 我们用 TypeScript、Python、Rust 层层约束 AI——这是对的吗？
- 精心设计的工具链条可能在下一次模型迭代中直接被解决
- 如果 AI 的终极目标是 AGI，那么所有 AI 工具届时都会被淘汰

## 作者的答案：克制地放手

- 告诉 Agent **做什么** 和 **不能做什么**，而不是**怎么做**
- 权限、配额、文件隔离用状态机焊死
- 子 Agent 之间只交换必要数据，坚决不污染干涉
- 上下文一定要做物理清理

> 未来模型越来越强，你越想强行约束它，它反而越笨。学会克制、学会松绑，才能让硅基智能真正释放。
