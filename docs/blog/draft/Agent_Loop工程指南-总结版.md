# Agent Loop 工程指南

&gt; 2026 年下半年 AI 圈最重要的概念：从写 Prompt 转向设计 Loop

---

## 为什么是现在？

### 背景
- **6月7日**：Open Cloud 创始人 Pat Stenberg 说：「我们所有人都应该停止写 Prompt，而应该转向为你的编辑设计 Loop」（220万浏览量）
- **6月2日**：Cloud Code 创始人 Boris 说：「我现在根本就不写 Prompt，我的工作就是写 Loops」

### 为什么现在爆发？
- 一年前：要自己维护一堆脚本
- 现在：主流产品（Cursor、Cloud Code）都内置了这些能力
- **工具不再是障碍，设计变成障碍**

### 本质
- **瓶颈转移**：原来瓶颈是模型，现在瓶颈是循环中的人
- Agent Loop 把人从循环里剥离出来

---

## Agent Loop 的五大构建 + 记忆层

| 构建 | 说明 |
|------|------|
| **Automation** | 自动化 |
| **Work Trees** | 工作树 |
| **Skills** | 技能：一组指令 + 元数据 + 可选脚本/资源 |
| **Plugins/Tools** | 插件/工具层 |
| **Subagents** | 自代理层 |
| **Memory** | 记忆层：Markdown 文件，每轮对话都会加载 |

### 为什么 Skill 重要？
- 没有 Skill：每轮从零开始，浪费 token
- 有 Skill：像不断积累经验的团队，难题沉淀为 Skill，下次直接调用
- 缩短运行时间和次数，节省成本

### 为什么需要 Subagent？
- Agent 对自己写的代码打分比较宽容（高估自己）
- 需要另一个 Agent（Subagent）来评分和检查
- **Maker 和 Checker 要分开**，通过 Subagent 实现

## 设计 Loop 的五步法

### 1. 定义「Done」（最重要！）
- 停止条件必须**清晰、可写为代码形式**，不是模糊想法
- 常见停止条件：
  - 所有测试通过
  - 输出结构匹配预设结构
  - 文档评分超过阈值（如 8 分）

### 2. 构建/组装 Context
- 每轮 Context 不同
- 根据当前状态自动组装，反映现在的状态

### 3. 执行并捕获
- 执行任务，捕获结果

### 4. 用反馈闭合循环
- 把上一轮的失败信息整合到下一轮的提示词中
- **失败信息是下一轮的弹药，丢了就断了反馈链**

### 5. 设置 Guardrails（护栏）
- **最大迭代次数**：通常 15-50 次
- **无进展检测**：卡住时结束
- **Token 上限**：防止跑一晚上烧掉几千美金

## 六种多 Agent 编排拓扑

单 Agent 的限制：上下文饱和、创新瓶颈、恢复脆弱

| 拓扑 | 适用场景 |
|------|----------|
| 简单流水线 | 线性任务 |
| 协调者-工作者模式 | 上层协调，下层干活 |
| 并行合并 | 审计 200 个文件、查询多个数据源（各干各的，无需交互） |
| 并行通信 | 五 Agent 搜数据，发现点同步给其他四个优化搜索路径 |

### 五大失败模式
- 上下文污染
- 级联失败
- 范围蔓延
- ...

## Token 成本优化

### 导致 Token 消耗增长的六大问题
1. 没有检查和终止条件（永远不会停）
2. 没有整合上一轮失败信息到下一轮
3. ...

### 优化策略
- **协调者只路由不推理**：输出只是路由方向（12345、东南西北中），非常短
- **输出比输入更贵**：节省输出是四两拨千斤，效果比节省输入更好

## 从 Prompt Engineer 到 Loop Engineer

### 能力要求上移
- 原来：写一条好的指令
- 现在：设计一个系统
  - 熟悉系统的每个构件
  - 测试系统的稳定性和有效性
  - 稳定有效运行的同时还要低成本
  - 能在生产环境使用
