# 面试被问到 Claude 的检索是怎么做的？

这是前几天一个球友（目前在某大厂做 RAG 算法）去某头部 AI 初创公司面试遇到的真题，面试官问：**“你看过 Anthropic 官方发的那篇关于 Claude 检索架构升级的文章吗？能聊聊他们是怎么做上下文检索（Contextual Retrieval）的吗？相比于传统的 RAG，他们到底优化了什么？”**

这个球友当时虽然知道 RAG（检索增强生成），也知道一些常规的优化手段（如子块检索、重排等），但对 Anthropic 提出的这个 **Contextual Retrieval（上下文检索）** 的具体实现细节和核心痛点抓得不够准，导致回答得有些流于表面。

今天我们就来彻底拆解一下 Anthropic 的这篇技术分享，看看这家顶尖的 AI 公司是如何解决传统 RAG 的核心痛点，并把检索准确率大幅提升的。这不仅是面试的高频考点，更是我们在实际做 RAG 系统时可以直接复用的高阶方案。

## 1. 传统 RAG 的“致命死穴”：上下文碎片化

在聊 Claude 的方案之前，我们先要想明白：**传统的 RAG 到底有什么问题？**

传统的 RAG 做法非常简单粗暴：

1. 拿来一篇长文档（比如 100 页的财报或者技术手册）。
2. 按照固定大小（比如 512 个 token）把文档切成一个个的文本块（Chunk）。
3. 把这些 Chunk 通过 Embedding 模型转化成向量，存入向量数据库。
4. 用户提问时，计算用户问题的向量与库里 Chunk 的向量相似度，召回 Top-K 个 Chunk 喂给 LLM。

这个流程看似完美，但在实际工业界应用中，会遇到一个极其致命的痛点：**上下文碎片化（Contextual Fragmentation）**。

什么意思呢？因为你是硬生生把文档切开的，**单个 Chunk 往往会丢失它在整篇文档中所处的背景信息（Context）**。

### 举个通俗的例子：

假设有一份关于某公司财务状况的文档，其中有一个 Chunk 的内容是：

> “由于研发投入加大，该季度的运营利润下降了 10%。”

如果用户提问：**“2024 年第二季度 Google 的运营利润变化如何？”**

当向量检索去匹配这个 Chunk 时，它很难拿到高分。为什么？因为这个 Chunk 里根本没有包含“2024 年”、“第二季度”、“Google”这三个核心的上下文实体！它只是孤零零的一句话。

这就是传统 RAG 的死穴：**信息在切块时被孤立了。** 向量模型只能看到这 512 个 token 内部的字面语义，看不到整本文档的大背景。

## 2. Claude 的解法：Contextual Retrieval（上下文检索）

Anthropic 的思路非常直接且优雅：**既然 Chunk 缺上下文，那我们在把它变成向量之前，人工给它“补”齐上下文不就行了？**

具体的架构设计分为三个核心步骤：

### 步骤一：利用大模型为每个 Chunk 生成上下文提示（Contextual Prompting）

在文档切块（Chunking）之后，不直接去做 Embedding，而是把“整篇文档（Full Document）”**和**“当前的单个文本块（Individual Chunk）”同时输入给一个强大的大语言模型（比如 Claude 3.5 Sonnet），让大模型去思考：**这个 Chunk 在整篇文档中扮演了什么角色？它补充了哪些背景信息？**

然后，让大模型生成一段简短的、总结性的上下文（通常在 50-100 个 token 左右）。

### 步骤二：前置拼接（Pre-pending）

把大模型生成的这段“上下文背景”，直接拼接在原始 Chunk 的最前面，形成一个新的“增强版 Chunk”。

### 步骤三：向量化与存储

最后，把这个“增强版 Chunk”送去计算 Embedding 向量，并存入向量数据库。同时，BM25 关键词索引也是基于这个增强后的文本来建立。

### 我们来看看对比：

- **传统 Chunk**：

  > “由于研发投入加大，该季度的运营利润下降了 10%。”

- **Claude 增强后的 Chunk**：

  > **[由 Claude 生成的上下文]**：这份文档是关于 **Google 公司 2024 年第二季度的财务业绩报告**。本段讨论的是该季度的盈利表现和成本变化。
  >
  > **[原始文本]**：由于研发投入加大，该季度的运营利润下降了 10%。

这时候，如果用户再搜索 **“2024 年第二季度 Google 的运营利润”**，由于增强后的 Chunk 里包含了这些密集的关键词和实体语义，向量检索和关键词检索的准确率就会呈指数级上升！

## 3. Anthropic 官方给出的惊人实验数据

Anthropic 在文章中公布了非常详尽的测试结果，他们对比了不同组合下的检索错误率（MFR, Missed Retrieval Rate，即没有召回正确信息的概率，越低越好）：

| **检索方案**                     | **传统 RAG 错误率** | **上下文检索（Contextual）错误率** | **性能提升**     |
| -------------------------------- | ------------------- | ---------------------------------- | ---------------- |
| **仅向量检索（Embedding Only）** | 27%                 | 17%                                | 相对提升 **37%** |
| **混合检索（Embedding + BM25）** | 12.3%               | 5.3%                               | 相对提升 **57%** |

### 核心结论：

1. **单项提升巨大**：仅仅加上了“上下文拼接”（Contextual），向量检索的错误率就从 27% 降到了 17%。
2. **混合检索+上下文检索 = 王炸**：当把 **Embedding + BM25 混合检索** 与 **Contextual Retrieval** 结合起来时，错误率直接降到了 **5.3%**。这意味着检索的失败率降低了一半以上！

## 4. 工业界落地：成本与工程的考量

看完这个方案，作为算法工程师，你脑子里一定会跳出两个字：**成本**。

如果一篇文章有 1000 个 Chunk，意味着我们要调用 1000 次大模型来为每个 Chunk 生成上下文。这在以前的 API 价格下是不可承受之重。

但 Anthropic 在文章里专门提到了两个工程落地利器，这也是面试时可以彰显你具备实际工程落地经验的加分项：

### 1. 提示词缓存（Prompt Caching）

在为同一个文档的不同 Chunk 生成上下文时，**“整篇文档”的内容是完全相同的**，改变的只是当前的 Chunk。

通过使用 Claude 的 Prompt Caching 功能，你可以把整篇文档缓存起来。这样，大模型在处理后续的 Chunk 时，不需要重复计算整篇文档的 token，**读取缓存的成本可以降低 90%，延迟也会大幅降低**。

### 2. 低成本小模型（Claude 3.5 Haiku）

生成这 50-100 个字背景信息的任务，并不一定非要用最贵的 3.5 Sonnet。使用速度极快、成本极低的 **Claude 3.5 Haiku** 同样能取得非常不错的效果。

通过 **“Prompt Caching + 小模型蒸馏/微调”** 的组合拳，可以把这项技术的工程落地成本压到极低，完全达到商业化实用的标准。

## 5. 面试通关总结：如何惊艳面试官？

如果在面试中被问到这个题目，或者让你设计一个高准确率的 RAG 系统，你可以按照以下逻辑层层递进地回答：

1. **指出痛点**：先说传统 RAG 按照固定长度切块，导致 Chunk 丢失了文章全局的“实体、时间、主语”等上下文大背景（Contextual Fragmentation）。
2. **阐述原理**：详细介绍 Claude 的 **Contextual Retrieval** 方案——利用轻量级 LLM 结合整篇文档，为每个 Chunk 提炼一段 50-100 字的背景描述，前置拼接（Pre-pending）后再做 Embedding 和 BM25。
3. **结合混合检索**：强调配合 **Embedding + BM25 混合检索**，并利用 **Rerank（重排）** 模型进行后处理，能将检索错误率从 20%+ 压低到 5% 左右。
4. **大谈工程落地与成本优化（亮点）**：主动聊到在生产环境中如何控成本。提到利用 **Prompt Caching（提示词缓存）** 缓存全局文档，降低 90% 的输入成本；以及采用小模型（如 Haiku）来生成上下文，平衡效果与 ROI。

# 面试被问到 Claude 的检索是怎么做的？

最近在社区看到，有个求职者面试字节的时候，聊到了一些 RAG 相关问题，正好这个求职者就说自己用过 Claude 写代码，面试官就问他：**“那你知道 Claude Code 检索代码用的是什么方式吗？”** 他说是 RAG 吧，现在不都这样做吗。后续面试官就没接着聊 RAG 相关的了。

## 1. 先说说那个最流行的误解

很多人一听 Claude Code 用 Grep 不用 RAG，第一反应就是：“啊，这么简答，还可以这样啊，难道是为了省钱吗？”

这也不能说全错吧，但这部分小伙伴只看到了最外面那层。调用 Embedding 模型和向量数据库确实要花一些钱，而且哈，Grep 这种系统自带的工具，成本几乎可以忽略不计。虽然这么想有点道理，但是却是把那些工程师想得太简单了，人家用这个 Grep 肯定也是有自己的考量的，而且这部分人这么说也根本没碰到问题的核心。

## 2. Claude Code 检索方式的演进真相

实际上，据我了解哈，Claude Code 一开始并不是奔着 Grep 去的。**Boris Cherny** 就是 Claude Code 的创造者，也是 Anthropic 的主任工程师，他在 Hacker News 上直接说过：**早期版本确实用了 RAG 加本地向量数据库，但很快就发现 Agentic Search（也就是基于 grep/glob 的主动检索）效果更好。** 后来在 Latent Space 播客里他又补了一句，说测试结果“好很多，幅度大到团队自己也觉得意外”。你看，人家不是没做，人家是一开始想到的也是 RAG 的做法，但是效果不好啊，那，服务广大用户肯定是要用效果好的啊。

所以这根本不是一开始就没有理由的选择 Grep，而是跑过实验之后得出的结论，所以，实验才是验证真理的唯一标准啊。有看我之前内容的小伙伴应该知道，我在 Claude Code 源码中，对其中用到的一些工程技术进行了深入的分析且发现很多东西就是很简单，但是简答的好啊。

## 3. 为什么在代码库中，传统 Grep 往往优于 RAG？

### 核心痛点 01：时效性 (Latency) —— 索引建成即过时！

RAG 的思路大家都肯定知道的吧，就是提前建索引，怎么做呢，先把代码库切块、向量化、存进数据库，查的时候再去库里找。

听着挺合理的吧，**可代码库是活的啊**，一个正常迭代的项目，一天下来可能改动几十次。你的索引刚建完了，第一个文件一改，它就过时了。要想保持实时同步，就得搭一套增量更新、重切块、重 Embedding 的流水线，这套东西的**维护成本，比省下来的那几次 API 调用贵太多了**。

### 核心痛点 02：精确性 (Precision) —— 语义相似 $\neq$ 字符正确

再一个就是精确性的问题，这个很多小伙伴肯定没有想到，而且这问题更根本哦，面试的时候一定要答上来。代码检索和文档检索的需求不一样：你搜的是函数名、类名、API 名称，这些都是精确的字符串，不是模糊的“概念”或者啥文本哦。

语义搜索本来是为了处理“你想说的和文档里写的不是同一个词但意思差不多”的情况，但在代码里，你要找的东西只有唯一正确的写法，**语义相似反而会给你捞出一堆完全不相关的干扰项**。所以，就是要精确地找到，就是要 Grep 去搜。

### 核心痛点 03：安全隐私 (Security) —— 代码是企业最高资产！

还有安全和隐私的考量，这个能答上来就最好了，**因为索引文件本身就是一个攻击面**；代码的 Embedding 向量也不是绝对安全的，有研究表明在某些条件下可以从向量里部分还原出原始文本。对于企业用户来说，代码是最核心的资产，多一套外部系统就多一重风险。

## 4. Agentic Search 的逻辑与工程细节

说到底，争论 Grep 还是 RAG，绕不开一个更根本的问题：**代码检索到底是一次性的"查库"行为，还是一个需要来回确认的过程呢？**

RAG 默认选的是前者，也就是你问问题，它找代码，然后就完事了。但实际写代码时候，你很难一次就把问题描述清楚。大多数时候，第一次搜出来的结果只是个线索，引出下一个问题。**Agentic Search 恰恰顺应了这个规律：每一轮 grep 的结果，都成为下一轮搜索的判断依据，不断收窄范围，直到锁定目标。**

这有点像一个资深程序员排查陌生 Bug 时的工作方式——不会上来就把所有文件扫一遍，而是先 grep 一个关键词，看到结果之后判断往哪个方向走，再接着往下挖。Claude Code 的 Agentic Search，本质上是在用工具复现这种有目的的、有反馈的探索节奏。

### 🎯 关键加分项细节：隔离噪音

这里还有个细节，很多人注意到了但没说透，答出来肯定是画画的加分项。Claude Code 在处理复杂的检索时候，**会拉起一个独立的子进程来专门跑 grep/glob/read 这些操作，底层用的是响应速度最快的 Haiku 模型**。

这个子进程把混乱的搜索原始结果捋一遍，然后只给主模型递一份摘要。为什么要这么设计？因为 grep 的原始输出往往又长又杂，如果每次搜索结果都直接塞进主模型的上下文，跑几轮下来上下文窗口就满了。这个架构的核心其实是在做一件事：**隔离噪音**。

## 5. 对手的选择：Cursor 的标准 RAG 路线

当然，那肯定不是所有工具都做了同样的决定，这个分歧本身也很值得看的。

Cursor 从一开始就选了**代码库索引路线**：把仓库切块、向量化，需要上下文时语义检索。这是 RAG 在代码场景里的标准实现。有人评测说，Cursor 在处理大型陌生代码库时确实更“聪明”，找到的上下文更紧凑、冗余更少。

但这条路也有代价。路径屏蔽要做加密处理的，权限边界要管的，向量泄露要防的，索引同步要维护的，这基本上是在原有系统之外另起了一套。

Claude Code 有意回避了这条路，Boris Cherny 自己说过，**Claude Code 的定位是“一个 Unix 工具，不是一个产品”**，设计原则就是先做最简单的能跑通的东西，复杂度等到真正必要时再引入。所以这不是谁比谁高明，而是两种价值取向的不同选择：

- 一个愿意投资基础设施换语义能力；
- 另一个选择保持极简换可靠性和可控性。

## 6. Grep 也不是没有短板

说了这么多 Agentic Search 的优势，也得说说它的问题，不然就不公平了。

反对声音里最有力的一条来自 **Milvus**，他认为 grep 没有语义理解，搜出来的东西里有大量不相关的匹配，模型要从这堆结果里找有用的信息，本身就在消耗大量 Token。他们基于这个痛点做了一个开源 MCP 插件叫 Claude Context，引入向量检索之后声称 Token 消耗能降低 40%。

其次哈，规模也是个边界条件。对于 Google、Meta 这种量级的超大型 Monorepo，几百万个文件、几十年积累下来命名惯例混乱，纯靠 grep 确实会遇到天花板。在这种场景下，**提前建索引做语义预筛，再配合 grep 精确确认，反而是更合理的混合方案**。

## 7. 总结

所以更准确的说法不是 Grep 胜过 RAG，而是：**在大多数日常开发场景里，Agentic Search 的性价比和可靠性更好，但它没有解决所有问题，边界条件是真实存在的。**

聊了这么多技术细节，最后说点感受层面的东西。

这场争论里，最有意思的不是谁赢了，而是为什么那么多工程师第一反应是 RAG 肯定更好。大家太习惯用技术的新旧程度来判断优劣了——新的就该更好，用旧的就是保守。Anthropic 这个决定之所以值得反复想，不是因为 Grep 有多厉害，而是因为**他们真的跑了很多的实验、看了很多的数据，然后做出了一个反直觉但有效的选择**。
