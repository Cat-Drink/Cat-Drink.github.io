# Loop Engineering（循环工程）完整技术文档

## 封面主题

标题：我不再提示 AI 了，我在写让 AI 自己跑起来的系统
副标题：从 Prompt 到 Loop 的四次跃迁，Loop Engineering 六大核心要素

## 一、行业范式变革背景

### 1\.1 旧模式痛点：人工驱动 AI 的局限

传统 AI 编程工作流：人工逐轮输入 Prompt、等待返回、人工判断结果、再次输入指令，循环往复。
在这套模式中，开发者本质是 “AI 流水线操作员”，AI 的处理速度被人类打字、思考、输入的速度硬性限制，无法实现持续自主执行。

### 1\.2 行业权威观点引爆新范式

1. 2026 年 6 月初，Anthropic Claude Code 负责人 Boris Cherny 公开表态：

> 我不再提示 Claude 了。我有一套 Loop 在运行，它们负责提示 Claude 并决定下一步做什么。我的工作是编写 Loop。
> 
> 

2. 开源项目 OpenClaw 作者 Peter Steinberger 发文呼吁：开发者应当放弃手动提示 AI 编程助手，转而设计能自主驱动自身的 Loop 系统。

3. Google 工程师 Addy Osmani 将这套全新工程实践正式命名为 **Loop Engineering（循环工程）**。

### 1\.3 核心定义

Loop Engineering 是一套系统级工程方法论，核心转变：
从「手动逐条提示 Agent 的操作员」升级为「设计自主驱动 Agent 循环系统的架构师」。
核心特征：搭建外部自动化机制代替人工持续输入指令，让 AI 自主完成任务发现、执行、校验、迭代全流程，人类仅负责顶层规则设计与高价值决策。

## 二、AI 开发工程的四次范式跃迁

AI 开发的核心关注点逐层升维，四层范式为叠加关系，并非互相替代：

|阶段|工程名称|核心问题|核心工作逻辑|局限性|
|---|---|---|---|---|
|一代|Prompt Engineering<br>提示词工程|怎么问|打磨单次指令，追求模型单次输出合格结果|完全依赖人工逐轮输入，人力成本极高|
|二代|Context Engineering<br>上下文工程|给 AI 什么信息|构建知识库、历史记录、约束条件，补齐项目完整背景|仅解决信息缺失，仍需人工持续引导步骤|
|三代|Harness Engineering<br>编排工程|如何调用能力|串联模型、工具、数据库，搭建多步骤工作流|仅实现工具串联，无法自主循环迭代任务|
|四代|Loop Engineering<br>循环工程|如何让 AI 持续、自主完成目标|构建闭环循环系统，自动调度、执行、校验、重试、记录状态|系统设计门槛更高，需配套校验机制规避风险|

核心层级关系：Loop Engineering 位于 Prompt Engineering 上层，是更高维度的系统设计；Prompt 质量会直接影响 Loop 产出效率，劣质 Prompt 会让 Loop 快速批量产出错误结果。

## 三、Loop 完整骨架：六大核心构成要素

一套可稳定无人值守运行的 Loop，由 5 个功能组件 \+ 1 个贯穿全流程的持久记忆系统组成，六要素完整释义：

### 要素 1：自动触发器 ——Loop 的心跳

Loop 的调度启动核心，替代人工手动唤醒 Agent；支持定时触发、事件触发（CI 失败、工单新增等），实现无人值守持续运行，对应 Claude Code `/loop` `/schedule` 调度指令。

### 要素 2：Worktree 并行隔离层 —— 多 Agent 安全并行底座

为多个并行运行的 Agent 分配独立代码工作区，自动隔离文件修改冲突，支撑多任务同时处理。
⚠️ 关键瓶颈：Worktree 仅解决文件冲突，**人工代码审查带宽才是并行 Agent 数量的真实上限**。

### 要素 3：技能文件（Skills）——Agent 长期记忆卡片

载体为项目内`SKILL.md`文件夹，固化全局项目知识：技术框架规范、目录结构、构建部署流程、历史事故约束、禁止修改逻辑等。
解决**意图债务（Intent Debt）**：避免每次会话 AI 从零猜测项目规则，减少自主推断带来的逻辑偏差，实现一次编写、全会话复用。

### 要素 4：连接器（基于 MCP 模型上下文协议）—— 打通真实业务环境

仅读取本地文件的 Agent 能力存在极大局限，MCP 统一协议连接器打通外部业务系统，赋予 Loop 实操能力：

1. 研发侧：读取 Issue 工单、自动创建 PR、关联工单、查询数据库、调用第三方 API

2. 协作侧：联动 Slack / 企业微信推送通知、对接 CI/CD 流水线
核心价值：让 Agent 不止 “描述修复方案”，而是在真实业务环境完成端到端交付（修复 bug→提交代码→关联工单→CI 校验通过后通知团队）。

### 要素 5：子 Agent（Sub\-agent）—— 制作者与检查者分离（Maker\-Checker 制衡架构）

#### 底层痛点

同一模型同时完成代码编写 \+ 自检时，会天然自我宽容，仅靠 Prompt 优化无法根治结构性缺陷。

#### 标准分工模式

1. 二元基础分工：制作 Agent（实现代码 / 方案）\+ 检查 Agent（独立校验规范、缺陷、功能），两者使用完全隔离的指令集；

2. 三角可靠分工：探索 Agent（定位问题）\+ 实现 Agent（开发修复）\+ 验证 Agent（按项目规范验收结果）。

#### 设计必要性

Loop 可脱离人工监督运行，独立校验子 Agent 是系统可信的基础，多重分离 Agent 可捕捉单一模型自圆其说而忽略的漏洞。

### 要素 6：持久记忆 —— 跨对话状态存储（Loop 的大脑皮层）

#### 底层限制

大模型单次对话结束后会完全遗忘上下文，若无外部持久存储，每次 Loop 触发都将从零启动，丢失任务进度、历史执行记录、待办事项。

#### 轻量化落地方案

将任务状态写入代码仓库文件（`TODO.md`）或第三方看板（Linear），永久记录：已完成事项、合并记录、未关闭任务、执行失败日志；仓库持久存储，不受模型会话重置影响。

### 六要素一句话总结

自动触发让循环转起来，Worktree 让并行安全，技能让 Agent 不迷路，连接器让 Agent 能动手，子 Agent 让质量有保障，记忆让循环能持续。

## 四、双层循环核心架构：内循环与外循环

区分两层循环是理解 Loop Engineering 的关键，二者各司其职、层级分明：

### 4\.1 内循环（Inner Loop）：Agent 内置原生逻辑

无需开发者额外设计，是 Agent 执行任务的本能流程，固定遵循「感知→推理→行动→观察」闭环：
读取文件 → 修改代码 → 运行测试 → 读取报错 → 迭代修改
Prompt Engineering 优化的就是内循环每一步输出的内容质量。

### 4\.2 外循环（Outer Loop）：开发者需要设计的顶层系统

Loop Engineering 的核心设计对象，相当于调度多个工程师的项目经理，负责全生命周期管理：
按计划发现任务 → 分派子 Agent 执行 → 汇总验证执行结果 → 持久存储任务状态 → 开启下一轮循环

### 两层架构核心区别

- Prompt 工程：优化单次执行的内容质量，作用于内循环；

- Loop 工程：设计整体调度、并行、校验、持久化系统，作用于外循环；
两者不冲突，属于叠加关系，内循环质量直接决定外循环产出的可靠性。

## 五、Loop 工程的风险与边界：三大不可规避的核心问题

Loop 仅改变工作模式，三类风险不会随自动化消失，反而会被系统加速放大：

### 5\.1 最终验证责任仍归属人类

无人值守的 Loop 同样会无人值守生成错误。
子 Agent 校验仅能降低风险，“校验通过” 只是系统声明，不等于逻辑无漏洞；无论 Loop 自动化程度多高，代码合并上线前的人工终审不可省略。

### 5\.2 理解债（Comprehension Debt）会加速积累

Loop 产出代码速度越快，开发者通读、理解代码的比例越低，形成「理解债」。
这并非 AI 编程独有问题，但自动化循环会大幅放大债务积累速度；唯一解决方案：必须坚持阅读 Loop 生成的所有代码，不能因系统运行流畅放弃深度理解。

### 5\.3 认知投降：最隐性的长期风险

自动化运行时，直接全盘接受 Loop 返回结果是人类最省力的选择，会形成认知惰性，即 “认知投降”。
同样一套 Loop，两种完全不同的使用结果：

1. 正向用法：基于自身深度理解，借助 Loop 加速落地；

2. 负向用法：依靠 Loop 回避业务、代码深度理解工作。
Loop 无法区分两种使用方式，判断与约束只能依靠开发者自身。

### 补充：Loop 工程的设计难点

工作难度并未降低，核心工作重心发生转移：
从「如何写好单条指令」转变为「如何设计一套值得信任、风险可控的自主循环系统」。

## 六、落地实践指南：从最小 Loop 起步

### 6\.1 起步原则：不追求一步全自动

新手落地优先选择**只读、低风险**的轻量化 Loop，逐步提升自主操作权限，每增加一层自动化，同步提升校验覆盖度与校验精度。

#### 最小可用入门 Loop（零代码修改风险）

每日定时执行，仅做信息整理，无任何写操作：

1. 每日读取 CI 失败日志、项目 Issue 工单列表；

2. 按业务优先级整理任务清单，写入`TODO.md`状态文件；

3. 不修改代码、不发起 PR、不合并任何变更；

4. 开发者每日查看清单，自主安排执行。

### 6\.2 自动化程度迭代路线

1. 一阶（只读）：任务分类、优先级梳理（入门）

2. 二阶（轻度写）：自动起草问题修复方案

3. 三阶（中度操作）：自动创建 Draft 草稿 PR

4. 四阶（高度自动化）：双 / 三子 Agent 校验 \+ 符合规则后自动合并

### 核心落地口诀（仅记一条）

Loop Engineering 的核心转变：从「我来驾驶 AI」变成「我来设计驾驶 AI 的规则」。
Loop 会无差别放大所有设计决策，无论优劣，规则设计必须深思熟虑。

### 全文一句话总结

Loop Engineering 将工程师的工作从「逐轮手动对话 AI」升维到「自主循环系统架构设计」；系统自动化程度越高，对人类判断力、业务理解力的要求反而越高。
