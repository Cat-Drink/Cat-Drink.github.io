# Vibe Coding 反复 Bug 的解决经验

## 问题现象

- 同一个问题反复修改，改 A 坏 B，问题复现
- 让 AI 修 bug，AI 说已修复，刷新一看还是错的，再修还是错的
- 来来回回四五轮，每次 AI 都觉得自己改对了，每次都是错的
- 修好了 A，切换其他面板修改，问题又复现

## 根本原因

- **没有防 Bug 的规则**
- **没有给 AI 划定责任范围**
- **没有约束代码质量的架构规范**

## 解决方案

### 一、立四条防 Bug 规则

1. **编码前确认**：AI 必须先输出实现计划（改哪些文件、大致逻辑、边界条件），你确认后再动手。不确定的地方列出来，不许自己猜
2. **编码时自带防护**：生成代码必须加错误处理，不能静默失败；对外部数据做空值检查和类型转换；改代码时保留旧代码注释，说明修改原因
3. **交付前自检**：每个新功能交付时同时给出手动测试步骤；知道可能有副作用必须提前告知
4. **修 Bug 流程**：先复现再修，不要直接猜原因；最小修改原则，只改该改的行，不要顺手重构别的；修完告诉你要测试哪些相关功能

### 二、建立架构宪章

创建 `ARCHITECTURE.md` 文件——项目的"宪法"：
- 项目分几层（前端展示层、业务逻辑层、数据访问层）
- 每层做什么、不能做什么
- 文件名怎么起、错误怎么处理、目录结构怎么摆

**所有代码的最高准则**——每次改代码，AI 必须对照这份文件做一次代码健康检查。

### 三、给每个模块生成说明书

每个模块一份 `README.md`：
- 这个模块是干什么的
- 它依赖谁
- 对外暴露了什么
- 最简单的用法是什么
- 想改某个功能要去改哪个文件

## 核心转变

> **不需要会写代码，只需要会定规则、会验收。**
