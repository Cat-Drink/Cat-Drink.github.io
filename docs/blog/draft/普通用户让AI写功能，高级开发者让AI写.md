# 普通用户让AI写功能，高级开发者让AI写架构

面了个三年前端，简历上写着熟练使用AI辅助编程，有Cursor Cloud Code的深度使用经验。

我随口问了一个真实场景：**你让AI帮你写一个主题切换功能，light、dark、blue三个选项，保存用户偏好，刷新不丢。**

你向AI描述需求，代码哗啦啦就出来了。运行代码，控制台疯狂输出日志，页面渲染了十几次状态，在三个地方显示着不同的值，但神奇的是——功能居然能用。你点了切换按钮，页面确实变了，看起来没问题。

**你觉得这个功能能上线吗？**

他愣了一下，然后自信的说："写的代码质量挺高的，测试也过了，用户能切主题应该没问题吧。"

我摇摇头，笑着打开它生成的代码，指出了关键问题："看这段代码，你告诉我现在用户点击切换主题会发生什么？"

"点击切换，设置本地主题，触发use effect，设置全局主题，主题状态更新，更新local storage。"

我点点头继续逼问：**如果用户迅速点击切换三次 light → dark → blue，你的local storage会写几次？状态会更新几次？所有订阅主题状态的组件会重新渲染几次？**

"你认为AI生成的这段优雅代码能在你的线上大促页面里存活多久？"

它开始支支吾吾...面试到这也结束了？

**No，这正是AI代码的架构与工程化陷阱。**

你描述需求，AI进行一系列看似合理的推理，但AI犯了一个根本性错误：**它把每个最佳实践当做独立模块来组合，而不是设计一个统一的架构。**

为什么连AI都很难驾驭这类问题？因为大模型的工作模式是模式匹配，而不是架构设计。AI能清楚知道Zustand API、useState语法、localStorage的用法，但它不知道：**同时维护多套状态意味着点击一次触发N次更新，多个组件互相监听，形成状态同步风暴，每次变更都会激起一连串的级联更新。**

在开发环境，功能看起来能用，但在生产环境高并发快速点击时性能雪崩。

---

一个能真正驾驭状态管理的高级前端，必须建立三层防御：

## 第一层：分辨AI代码的功能正确与架构健康

功能确实能跑，但系统已失控。用React DevTools分析无效渲染，点击一次就渲染几十次，这就是潜在性能地雷。追踪级联状态更新链，多少个useEffect被触发，这些副作用又在驱动多少无谓的全局刷新。在Network Throttling和CPU减速模式下，模拟真实用户快速操作，**辨别能演示与能抗压**。

## 第二层：掌握跨系统协调的工程化思维

当需求需要整合多个状态时，必须设计统一的数据流，而不是依赖分离的多个状态各自为政。复杂事件处理应该引入状态机，用有限状态语义明确锁定当前唯一状态，从源头消灭魔法布尔值导致的非法组合。

## 第三层：重构能力

不少工程问题需要从核心架构解决问题，AI仅能缝补，开发者需善用静态分析工具，追踪不纯的useEffect，利用Chrome DevTools的Performance面板量化抓取原本隐藏的重复渲染瓶颈，手动精简状态同步链。

---

所以这道题真正考的是什么？

**不是你能不能用AI生成功能，而是你有没有从"我会让功能跑通"升级，到"我能评估AI代码的架构质量，并对系统做长期演进加固"。**
