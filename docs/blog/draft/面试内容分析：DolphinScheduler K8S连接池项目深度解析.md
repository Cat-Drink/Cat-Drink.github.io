#### 核心面试问答

**问：详细说一下 DolphinScheduler 与 K8S 连接池项目，做了什么、解决了什么问题，遇到过哪些困难？**
答：这是一个开源项目，我和导师对接完成。
原有问题是：DolphinScheduler 每执行一个 K8S 任务就创建一个连接实例，任务结束后直接销毁，造成资源浪费。负责人希望任务执行完后，K8S 连接可以保持活跃、被复用，因此提出了构建连接池的设计思路。
我的实现是：通过 PoolConfig 标记每个连接池的唯一性，基于 K8sUtils 连接工厂创建对应 K8S 集群的连接；任务调度时从工厂获取对应集群的连接实例，使用完之后释放资源、放回连接池复用。
遇到的主要问题是沟通偏差：一开始我以为要在连接池里创建多个独立连接实例，后来负责人才说明，是要基于 DolphinScheduler 底层原有的连接创建逻辑做复用，而非创建多个孤立连接。

**问：连接是如何实现复用的？请求过来怎么从连接池取连接？通过什么机制返回连接给调用者？**
答：具体实现细节我不太记得了，我负责开发 K8S 集群的 Client Pool，通过 K8sUtils 调用工厂创建连接；任务执行完成断开连接时，应该是通过 try\-catch 把连接放回连接池。

**问：多个线程同时取连接、同时回收，怎么保证连接正确分配？会不会有线程冲突、线程安全问题？**
答：线程安全方面，我记得是用了异步线程的 key 队列做连接池管理，基于集群配置文件生成 hash code 做集群识别，多并发的情况我不太清楚。

**问：连接池的本质是什么？和每次新建连接有什么区别？为什么要用连接池？**
答：好处是省去多次创建连接实例的资源浪费，连接会一直保存在池里保持活跃，只有 K8S 的认证配置文件过期时，才会尝试重连。如果连接失败，就更新配置，废弃旧连接、用新配置创建连接。
如果是非证书原因导致的连接失效，也会废弃连接，后续有新请求时再重新构建。
连接池的本质就是一个容器，存放常用连接，用的时候取、不用的时候放回去，省去每次任务都创建 \- 销毁连接的开销。

**问：你说的过期是什么意思？**
答：一开始我没考虑过期问题，是导师提醒 K8S 的 kubeconfig 配置有一年的认证有效期，要解决认证过期后旧配置无法连接集群的问题。
我的设计思路是：如果配置文件发生变更，先保留旧连接一段时间；如果连接多次失败，就废弃旧连接。

**问：如果取到的连接已经挂掉了，后续怎么维护？比如拿到连接出现 broken pipe 异常怎么办？K8S 集群重启后连接失效，除了重启程序还有什么办法？**
答：集群重启的情况我好像确实没考虑到。
我现在的思路是，增加一个旁路线程对连接池里的连接做心跳检测，判断 K8S 集群是否活跃，同时给管理员发告警，让工作人员去后台检查集群状态、做重启处理。

**问：我的意思是，K8S 只是重启一次，连接失效了，系统怎么自愈？连接是长连接，集群重启后肯定断了，怎么自动恢复？**
答：我能想到的是，如果连接失效导致任务执行失败，就让 DolphinScheduler 重新跑一次任务；重跑的时候就会重新建立连接，再把新连接放入连接池，相当于让连接恢复活跃。

**问：怎么重新建立连接？用什么机制感知失效？客户端拿到连接只会报错，不会自动重连，你怎么处理？**
答：自动处理我想不出更好的方法，我更倾向于人工重跑任务，重跑就会重新走创建连接、放入连接池、执行任务的流程。

**问：连接池里已经有失效的连接了，不是空的，不会触发新建连接啊。**
答：确实没什么好的想法。

（面试官提示：常规方案是两种，一是旁路线程巡检销毁重建，二是取连接时先做可用性检测，失效就销毁重建，数据库连接池都是这个逻辑。候选人表示之前没考虑过、了解较浅。）

**问：你这个改造按这个思路，开源社区肯定不会接受合并不进去对吧？你们导师没有审核这个问题吗？**
