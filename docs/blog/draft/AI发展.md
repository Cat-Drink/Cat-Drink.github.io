旨在清晰梳理 “Skills” 与 “MCP”“Workflow”“Command”“Prompt” 等概念的关系，以下是详细内容总结：

### 一、背景：AI 领域概念堆砌的现象

视频开篇指出，近年 AI 领域存在用 “Command”“Rule Prompt（提示词工程）”“上下文工程 RAG”“Agent”“MCP” 等概念堆砌，掩饰技术能力匮乏的情况，进而引出近期热门新词 “Skills”，并提出要探究其本质及与其他概念的关联。

### 二、核心概念逐个解析

#### 1. Prompt（提示词）

- 基本定义：让 AI 干活的简单指令，例如 “帮我写个简历”。
- 结构化提示词：为提升效果，在提示词中加入诸多规则（如简历个人信息居中、教育经历表格化、项目经历用 STAR 法则拆解），这类详细规定步骤的提示词即为结构化提示词。

#### 2. Command

因长提示词手动输入太麻烦，很多 AI 客户端（如 ClaudeCode）支持将常用提示词固化为文件，通过类似快捷键的命令唤起，这种 “用短命令替换固定内容” 的功能就是 Command。

#### 3. System Prompt（系统提示词）

- 优先级：系统提示词优先级高于用户提示词（User Prompt），将要求放入系统提示词，大模型的指令遵循效果更好。
- 实现方式：通常 AI 客户端会在后台发送系统提示词，部分软件（如 Cursor、ClaudeCode）提供入口，允许用户以文件形式自定义系统提示词（如写入 “请用中文回复我”）。

#### 4. Metadata（元数据）

- 解决的问题：不同场景（写简历、周报、邮件）对提示词要求不同，若全塞进一个系统提示词文件会不合适。
- 定义与作用：在拆分后的各场景提示词文件开头，添加一段描述信息，说明文件用途和调用场景，这些描述信息就是 Metadata。它能让 AI 客户端仅发送少量 Metadata 给大模型判断场景，再加载对应场景的完整系统提示词，既保证效果又节省 token（大模型处理的单位，影响成本）。

#### 5. References 和 Scripts

- References（参考资料）：当需求进一步细化（如写简历分产品、开发、算法岗），可将拆分出的细分文件放入 “references” 文件夹，大模型可按需参考其中材料。
- Scripts（脚本）：AI 客户端可调用系统命令执行代码脚本（如 Python 脚本将文本转 Word 再转 PDF），这些脚本文件放在 “Scripts” 文件夹，在提示词中说明执行场景，实现纯文本聊天外的功能。

#### 6. Skills

- 本质：是外化为文件夹形式、可动态加载的系统提示词集合。它整合了用户提示词、系统提示词、Metadata、References 和 Scripts，实现按场景和条件加载数据，减少 token 消耗。
- 工作流程：将 Skills 文件夹放入 AI 客户端（如 ClaudeCode）的指定目录后，用户发送需求（如 “帮我写个简历 pdf”），客户端先发送各 Skills 的 Metadata 给大模型，大模型判断后，客户端加载对应 Skill 的系统提示词、参考资料，甚至执行脚本，最终由大模型输出结果给用户。

### 三、概念区别辨析

#### 1. Skills 与 MCP

- MCP：是让大模型（像大脑）操控外部工具的协议，相当于 “手”，MCP 插件是 “手上的工具”。
- Skills：是操作经验，规定在什么场景下，按什么顺序组合使用哪些工具（工具可以是 MCP 插件或本地 Script 脚本）。

#### 2. Skills 与 Workflow

- Workflow：通过规则配置，将多个步骤编排调度的流程（如低代码工具 n8n 的拖拉拽流水线），流程结构在设计阶段就确定。
- Skills：本质也是逻辑编排，但执行流程由大模型驱动，灵活性更高。可简单理解为 “大模型驱动的 Workflow”。

#### 一、Agent Skills 的起源与发展

- 最初是 Claude 的小功能模块，因实用性强，Codex、Cursor、OpenCode 等 AI 编程工具陆续支持。
- 2025 年 12 月 18 日，Anthropic 将其发布为**开放标准**，朝着**通用、跨平台规范**方向发展。

#### 二、核心原理：“带目录的说明书” 式提示词管理

Agent Skills 是一种**渐进式披露提示词的机制**，将提示词分为三层：

- **元数据（Metadata）**：必定加载，类比 “书本目录”，仅包含 Skill 名称、调用时机等核心信息，Token 消耗极低（约 100 Token）。
- **指令（Instruction）**：按需加载，类比 “书本正文”，包含具体任务逻辑和细节。
- **资源（Resource）**：按需加载，类比 “书本附录”，可包含脚本、文档、图片等辅助文件。

这种设计大幅降低了提示词复杂度和 Token 消耗，让 AI 能 “按需查阅” 信息。

#### 三、实战操作：从 Claude Code 到跨平台应用

##### 1. Claude Code 实战

- **安装配置**：通过官网命令安装，若无 Claude 账号，可修改配置文件接入智谱 GLM 等第三方模型。
- **创建 Skill**：以 “字幕转 markdown” Skill 为例，需建立`./claude/skills/技能名/SKILL.md`目录结构，在 SKILL.md 中编写元数据（定义 Skill 名称、调用时机）和指令（详细任务逻辑，如保留所有字幕文字、插入截图占位符）。
- **调用测试**：拖拽 srt 字幕文件到 Claude Code，AI 自动识别并调用 Skill，生成带截图占位符的 markdown 笔记。

还演示了 “来点选题” Skill，帮助视频创作者生成选题灵感，进一步解释了 AI 调用 Skill 的流程：扫描本地 SKILL.md→提取元数据生成技能列表→AI 选择是否调用→按需加载指令和资源→生成最终结果。

##### 2. 跨平台与开源生态

- **迁移到 Codex**：只需将路径中的`.claude`改为`.codex`，即可将 Skill 迁移到 Codex 中使用。
- **开源 Skill 包**：GitHub 上有成熟的 Skill 库（如`Awesome Claude Skills`），可直接拖拽文件夹使用，例如 “域名头脑风暴” Skill 能为博主推荐个性化域名。

#### 四、进阶用法：资源层的深度应用

Skill 文件夹可添加三类资源文件，拓展功能边界：

- **scripts（可执行脚本）**：如用 Python+ffmpeg 编写截图脚本，完善 “字幕转 markdown” Skill，将截图占位符替换为真实视频截图。
- **references（补充文档）**：如在 “帮我写作” Skill 中放入范文，让 AI 学习行文风格，生成风格一致的深度文案。
- **assets（图片 / 模板）**：可存放辅助资源，进一步丰富 Skill 能力。

#### 五、与 MCP 的对比与配合

| 维度       | Agent Skills                       | MCP                           |
| ---------- | ---------------------------------- | ----------------------------- |
| 侧重点     | 提示词管理（渐进式披露）           | 工具调用（标准化工具箱）      |
| 类比       | 带目录的说明书                     | 标准化工具箱                  |
| Token 消耗 | 低（仅加载元数据，按需加载其他层） | 高（一次性塞入所有提示词）    |
| 核心主体   | Markdown 文件                      | 软件包（Node.js/Python 编写） |
| 编写难度   | 低（只需写 Markdown）              | 高（需搭建开发环境）、        |

**配合场景**：Skill 负责提示词的渐进式披露，MCP 负责具体工具调用。例如，“帮我写作” Skill 生成文案后，调用 GitHub MCP Server 自动创建仓库并上传文案，实现 “创作 + 存储” 的完整流程。

**“给我完整 code 并解释”**：被归为 “夯” 级。它是新手和懒人常用的提示词，只需无脑复制粘贴题目让 AI 生成代码和解释，像开盲盒一样等结果，但会陷入漫长的无效循环，用过一次就容易依赖。

**“只修改我让你修改的部分”**：属于 “顶级”。能给 AI 上紧箍咒，避免 AI 乱改测试用例、作业模板或他人代码，大多数情况下效果不错。

**“别自作聪明”**：列为 “NPC” 级。专用于 AI 过度引申、瞎改需求或 “上课” 时，能把 AI 打回原形，但用多了会让 AI 变得畏首畏尾，解决不了根本问题。

**“还是不对 再看看”**：归到 “NPC” 级。是比较礼貌的回复，多数时候 AI 只会说 “非常抱歉” 然后换个随机种子再猜，属于无效沟通，会和 AI 一起陷入循环。

**“用中文回答”**：定为 “夯” 级。因为满页外文会让人头皮发麻，要求 AI 用中文回答是和 AI 交流的基础，但有时需要多次提醒，不然有的模型会忘记。

**“不是，你觉得对吗？”**：属于 “人上人” 级。是给 AI 的反向图灵测试，能测出 AI 的 “墙头草” 属性，虽然双方会一起抓瞎和稀泥，但有其特殊作用。

**“你再仔细读一下题目”**：被评为 “拉完了” 级。说出这句话时人已经开始着急，是把锅甩给 AI，最后能不能蒙对答案全看运气。







视频从大语言模型（LLM）讲起，说明其本质是文字接龙，通过将对话划分为提示词（Prompt）、上下文（Context）、记忆（Memory）等部分，实现了可追问的对话能力。为解决大模型缺乏实时信息的问题，引入智能体（Agent），它可通过检索增强生成（RAG）联网搜索或调用本地工具，其中工具调用的格式约定为 Function Calling，Agent 与工具服务的交互协议为 MCP（模型上下文协议）。

对于复杂任务流程，衍生出 Langchain（编程框架）、workflow（低代码工作流），而 Skill 则是将提示词和流程说明封装成技能，子智能体（SubAgent）用于隔离复杂任务的上下文。

视频指出这些概念多是技术发展的中间产物，还对比了 Function Calling 与 MCP、Skill 与 MCP 的区别，从刚性到柔性梳理了 Langchain、workflow、Skill、Agent 的特性，最后强调这些技术本质是围绕大模型，通过各种方式扩展其能力，未来更便捷的智能体产品会逐渐淘汰复杂配置，聚焦为用户提供便利。
