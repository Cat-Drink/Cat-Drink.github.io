## 1. 腾讯后端岗面试

### 整体感受
腾讯后端面试非常硬核，极其看重底层原理、计算机基础（系统级）以及大规模分布式系统的架构设计能力。八股文不仅要会背，更要懂底层源码和实际应用场景。

### 核心考点

#### 架构与分布式
- 说一下你的架构设计，如果是高并发高可用的架构，该怎么设计？
- Redis 缓存和 MySQL 数据库如何保证一致性？具体有哪些方案？
- 分布式锁的实现方案有哪些？你更倾向于哪种？为什么？
- 如果 Redis 缓存发生穿透、击穿、雪崩，该怎么处理？
- 微服务架构中，服务注册与发现的原理是什么？Consul / Nacos / Eureka 的区别？
- 分布式事务的解决方案有哪些（2PC、3PC、TCC、本地消息表、Seata）？
- 消息队列（如 Kafka、RabbitMQ）如何保证消息不丢失？如何处理重复消费？
- 什么是全链路追踪？它的核心原理是什么？

#### JVM 与 Java
- 了解哪些垃圾回收器？CMS 和 G1 的区别是什么？
- 说一下 JVM 的内存区域划分。哪些是线程私有的？哪些是线程共享的？
- 类加载机制的过程是怎样的？什么是双亲委派模型？为什么要设计双亲委派？
- 强引用、软引用、弱引用、虚引用的区别和应用场景是什么？
- 说一下你常用的 JVM 调优参数有哪些？发生 OOM 怎么排查？

#### 计算机网络
- 输入一个 URL 到页面加载出来的全过程是怎样的？
- TCP 三次握手和四次挥手的流程是怎样的？为什么挥手需要四次？
- TCP 是如何保证可靠传输的？（滑动窗口、拥塞控制等）
- HTTPS 的加密机制是怎样的？对称加密和非对称加密分别用在什么阶段？
- HTTP/1.1、HTTP/2.0 和 HTTP/3.0 有什么区别？

#### 算法与数据结构
- 说一下常用的排序算法，快排和归并排序的时间复杂度和空间复杂度是多少？
- **手写算法题**：最长回文子串（要求给出最优解法，并解释时间复杂度和空间复杂度）
- 反转链表（手写代码）
- LRU 缓存淘汰算法的设计思路是什么？如果要你实现，你会用什么数据结构？
- 动态规划和贪心算法的区别是什么？

#### Linux 网络编程与内核
- `epoll` 与 `select/poll` 的本质区别
- `epoll` 的底层实现（红黑树 + 双向链表），触发模式：水平触发（LT）与边缘触发（ET）
- 零拷贝技术（Zero-Copy）：`mmap`、`sendfile` 的工作原理

#### 高并发架构设计
- 海量数据存储与高并发读取场景下的分库分表策略（垂直拆分、水平拆分、Hash取模、Range范围划分）
- 分库分表带来的分布式ID生成问题（雪花算法 Snowflake、UUID、数据库自增步长、Redis分布式ID）以及跨库Join的解决方案

#### 项目相关
- 说说你在项目中遇到的最大的技术挑战是什么？你是怎么解决的？

## 3. 核心技术专题：滑动窗口
**来源**：程序员小星

### 1. 核心定义与作用
**滑动窗口** 是一种常用于算法优化（如双指针法解决子串/子数组问题）以及网络流控（如TCP流量控制）的核心机制。

### 2. 面试高频考察方向
- **算法层面**：
  - 如何在 $O(n)$ 时间复杂度内求出满足条件的最长/最短子数组、子串
  - 经典题目：无重复字符的最长子串、最小覆盖子串、长度最小的子数组
- **工程与架构层面**：
  - **微服务限流机制**：滑动窗口限流算法（对比固定窗口限流，它如何平滑突发流量、消除临界窗口的流量两倍冲击问题）
  - **TCP协议中的滑动窗口**：
    - 接收端与发送端的窗口同步机制
    - 流量控制（Flow Control）与拥塞控制（Congestion Control：慢启动、拥塞避免、快重传、快恢复）的区别与联系

## 4. Java 后端面试真题精选
**来源**：-搁浅_-

### 1. Java 基础与集合框架
- **HashMap 源码级深挖**：
  - HashMap 的底层数据结构是什么？（数组 + 链表 + 红黑树）
  - 为什么在JDK 1.8中引入红黑树？触发链表转红黑树的条件是什么？（链表长度大于8且数组长度大于等于64）
  - 为什么红黑树在节点数变少时（小于等于6）会退化为链表？
  - 负载因子（Load Factor）为什么默认是 0.75？
  - 扩容机制（Resize）的实现原理，JDK 1.7 的头插法导致死循环的原因，以及 JDK 1.8 的尾插法优化

### 2. 线程池与并发编程
- **线程池核心参数**：核心线程数、最大线程数、空闲线程存活时间、时间单位、任务队列、线程工厂、拒绝策略
- **拒绝策略**：
  - `AbortPolicy`（抛出异常，默认）
  - `CallerRunsPolicy`（调用者运行，谁提交谁执行）
  - `DiscardPolicy`（直接丢弃）
  - `DiscardOldestPolicy`（丢弃队列中最老的任务）
- **工作流程**：核心线程 -&gt; 阻塞队列 -&gt; 非核心线程 -&gt; 拒绝策略

### 3. JVM（Java虚拟机）
- **内存区域划分**：堆、方法区（元空间）、虚拟机栈、本地方法栈、程序计数器。哪些是线程共享的？哪些是线程私有的？
- **垃圾回收（GC）机制**：
  - 如何判断对象是否可以回收？（引用计数法、可达性分析算法）
  - GC 垃圾回收算法：标记-清除、标记-复制、标记-整理
  - 常见的垃圾收集器（CMS、G1）的特点和工作原理

## 5. 某大厂后端开发多轮面经
**来源**：-怎么又饿了-

### 【第一阶段：基础扎实度考察】
- **计算机网络**：
  - TCP 建立连接的三次握手与断开连接的四次挥手详细状态机变化（SYN, ACK, FIN, TIME_WAIT）
  - 为什么需要 TIME_WAIT 状态？为什么是 2MSL？
  - HTTPS 的加密原理：对称加密、非对称加密、CA证书、数字签名以及 SSL/TLS 握手全过程
- **操作系统**：
  - 进程与线程的区别；线程间通信与进程间通信（IPC）的方式
  - 什么是虚拟内存？分页与分段管理机制，页面置换算法（LRU, FIFO）
  - 什么是死锁？死锁产生的四个必要条件是什么？如何避免死锁？
  - 进程、线程和协程的区别是什么？
  - 并发和并行的区别是什么？

### 【第二阶段：分布式与中间件架构】
- **Redis 高并发缓存**：
  - Redis 的数据类型及其底层编码（SDS、ZipList、SkipList、HashTable）
  - 缓存三剑客：缓存雪崩、缓存击穿、缓存穿透的定义、发生场景及彻底解决方案（布隆过滤器、热点数据永不过期、随机过期时间）
  - Redis 持久化机制：RDB 与 AOF 的优缺点及混合持久化
- **分布式事务与一致性**：
  - CAP定理与BASE理论的深层含义
  - 常见的分布式事务解决方案（2PC、3PC、TCC、本地消息表、RocketMQ事务消息）

### 【第三阶段：综合场景设计与面试官对抗】
- **架构设计题**：如何设计一个秒杀系统？
  - 如何防止超卖（Redis预扣减库存 + Lua脚本原子性）
  - 流量削峰（引入消息队列 RocketMQ/RabbitMQ）
  - 静态化与CDN加速、多级缓存架构、恶意请求拦截（限流防刷）
- **面试总结**：面试官非常注重底层原理的连贯性，从一个基础 API 能一路问到操作系统内核或底层数据结构的内存布局，必须具备系统化的知识网络

## 7. 面试问答技巧：常见问题回答策略

这篇文章整理了面试中6个高频问题的回答技巧，帮你在面试中脱颖而出，给面试官留下深刻印象。

---

### 一、你的核心优势是什么？

#### 常见错误回答

大部分人的回答都是这样的：
- "我的优势是学习能力强、适应性强"
- "我做事情比较认真负责，团队合作精神比较好，抗压能力强"

这些回答**每个人都这么说，放在任何人身上都成立，没有区分度**。面试官听完脑子里完全没有画面感，对你没有印象，而没有印象就等于这个人不存在，你自然无法脱颖而出。

就像买手机，店员跟你说"这个手机性能好、续航好、拍照也好"，你会买吗？不会。因为每个手机都这么说，这款手机跟别的手机没有区别。

但是如果他说"这款手机逆光拍照也很清晰，充电20分钟能用一整天，开最高画质也不卡"——给你描述具体的使用场景和细节，你就很容易记住了。

**面试其实也是一样的，要让面试官脑子里出现一个属于你的具体画面，不然你说再多也是苍白无力的。**

#### 为什么会犯这个错误？

根本原因是：大部分人根本就没有思考过自己的核心优势是什么，临到面试了，觉得应该说点积极向上的词，就把几个形容词往上一堆。但是这几个词是你的真实优势，还是你希望自己有这个优势，你自己可能都说不清楚。

**核心优势它不是一个形容词，它是一件你能反复做到的事情。**

你说学习能力强，那学习能力强这件事，就需要一个具体可以讲出来的证明。如果有，就把那件事说出来；如果没有，那这个词可能就不是你的优势，只是你的期望而已。

#### 正确回答公式

一个标签 + 一个证据 + 一个结果，就三件事：

1. **标签**：要具体，不能用"执行力强"这种万能词
2. **证据**：一件真实发生过的事情，说清楚你在什么情况下做了什么
3. **结果**：这件事情给你带来了什么，最好有数据

**例子：**
> "我的核心优势是能在信息很碎的情况下快速抓住重点。我在实习的时候负责过竞品分析，要在两天内整理出十几款竞品的核心差异。我自己搭建了一套分析框架，最后的报告直接被团队沿用了三个月。"

这个回答就描述得很具体，你做了什么事情，中间有数据，最后有结果。整个回答里边没有一个形容词，全是动词和名词。面试官听完之后，脑子里就有一个具体的画面，知道你能做什么。

### 二、你还投了哪些公司？你现在手上有offer吗？

这个问题一般在面试结尾，面试官对你有兴趣的时候才会问。

#### 两种常见错误回答

1. **清单式回答**：把投递的头部公司全部列出来，阿里、腾讯、美团挨个报一遍
2. **讨好型回答**："我只投了你们一家，我对你们最感兴趣"

#### 错误分析

**第一种清单式：**
面试官不会觉得你很抢手，他会想到：你投这么散，对我们这个岗位到底了解多少，你有多认真考虑过？你把自己变成了一个广撒网的候选人，不是一个有规划的人。对于企业来讲，听到这个信息之后就觉得自己没有被重视到，感受不好。

**第二种讨好型：**
面试官不会觉得你很专一，他会判断两件事：
- 要么你不自信，觉得自己没有竞争力，只敢投一家
- 要么你在说谎，正常求职者不可能只投一家

这两种判断对你都没有好处。

#### 面试官真正在问什么？

这道题背后考察三个问题：
1. 你的求职方向有没有规划，还是随便乱投？
2. 这个岗位你的意愿度到底有多强？
3. 我需不需要加快流程，你手上有没有其他在推进的机会？

**这是动机类考察、稳定性考察和意愿度考察，不是简单让你回报投递清单。**

#### 正确回答思路

**说方向，不说清单，点出意向，用进展增加筹码。**

- 说你投递的方向，不说具体公司名单
- 明确表达出这家是你的优先选项，给出一个真实具体的理由
- 如果手上有进展，合理透露出来，增加竞争筹码

**例子：**
> "我主要是在看互联网方向的产品运营岗，主要集中在几家头部公司。目前有两家在推进流程中，我对这边的意向是最高的，因为你们在这个业务方向的布局我是比较看好的——（这里说一个你真实了解过的具体点，而不是客套话）"

- 如果**手上有offer**，这个时候说出来最有价值，说明你有市场竞争力，也给对方一个加快流程的理由
- 如果**手上没有offer，但有几家在推进**，说有几家在推进就够了，不需要说具体到哪个阶段了
- 如果**真的什么都没有**，就说"我投得比较集中，主要看这几个方向"，不要胡编乱造

#### 追问：那你更看好哪一家？

很多面试官都会问这个问题，测试你说的意向是不是真的。

**两种错误回答：**
1. 立刻说"当然是你们"——太明显的讨好，面试官不会信
2. "还在比较中，说不准"——等于告诉对方你还没有特别倾向，印象分直接减分

**正确回答：**

给出一个具体的理由，而且这个理由要跟你的职业方向挂钩，不是泛泛地说公司大不大、名气高不高。

**例子：**
> "目前来说这边是我更倾向的。主要是因为这个岗位的业务方向跟我深耕的方向是一致的，我在这边看到的成长路径对我来说更加清晰，对我的未来职业规划会更有帮助。"

这个回答：第一，表达了倾向；第二，有具体的理由；而且理由关乎你的职业规划，不是单纯在夸对方公司有多好。面试官听完会觉得你是认真考虑过的，不是随口一说。

**一定要具体！**比如你面景观园林方向，公司做市政项目，你可以这样说：
> "目前地产行业形势不是那么景气，对景观园林需求量也没那么大。但是我看到咱们公司做的是某市重点市政园林工程项目，我觉得这个方向会比较有发展前景，而且和我的专业方向也比较对口，所以特别看好。"

这样既表达了强烈意愿，又说出了具体理由，还能体现出你对行业是有洞察的。

### 三、你为什么从上一家公司离职？

这是社招面试必问题，应届生一般不会遇到。

#### 网上流传的错误建议

网上说"离职原因要往正面说，不要说上家公司不好，要说追求更大的发展空间，希望挑战自己，想在新平台成长"——这是HR最烦的一类回答。

每个人都可以说"寻求发展空间"，这句话**没有任何信息含量**，HR一听就知道你在回避真实原因。HR心里会给你打上标签：你不够坦诚，这个标签很难洗掉。

#### HR真正想确认什么？

HR问离职原因时，真正想确认两件事：
1. 你离开上家公司的原因，会不会在我们这里重演？（比如你因为不喜欢加班离职，我们这里加班更多，将来你会不会也走？）
2. 你这个人对过去的经历是什么样的态度？是客观理性的，还是动不动就抱怨？看你的工作心态。

#### 回答原则

让HR对这两件事都放心，同时把离职原因跟你投递这个岗位的动机连起来，让整件事有逻辑。

#### 不同情况的回答思路

##### 1. 实习结束，正常离职（应届生）
直接说："实习合同到期了，我正在寻找正式的工作机会。"
不需要任何包装，这是事实。

##### 2. 业务调整/项目被砍
这其实是加分项，因为是外部原因，不是你的问题。
> "上家公司去年做了业务收缩，我负责的这个方向整个就缩减了。我认真考虑了一下自己想做什么，发现自己更想往深耕这个方向，所以看到这个岗位就过来了。"

有真实原因，有主动思考，而且把求职动机和目标岗位连起来了，逻辑非常顺。

##### 3. 薪资太低
不要直接说，可以侧面表达：
> "上家公司的薪资体系比较固定，成长空间有限。我觉得到了这个阶段想看看市场的机会。"

这是真实的，你没有抱怨，而且合情合理。注意不要说"钱给得太少"，那样听起来你好像只是为了钱，不是因为这个岗位。同一件事情，说法不同，感受完全不同。

##### 4. 和上级领导处不好
这个绝对不能直接说！哪怕上司真的有问题，面试官听到你说跟上司关系不好，第一反应不是同情你，而是在想：这个人来到我们这里之后，如果和上级处不好，会不会也这样说我们？

转化成客观原因：
> "那家公司的管理风格跟我的工作方式契合度不是很高，我在那个环境里发挥空间比较有限，所以我想找一个更适合我的团队。"

说"契合度"，不说对错，也不评价对方好坏。

##### 5. 工作太清闲，没有成长性
这种说法要有层次：
> "上家公司业务发展到了一个比较平稳的阶段，我在那边的工作偏重复，感觉自己的成长放缓了。我是一个希望能够持续在有挑战性的工作环境下工作的人，所以我主动出来看新的机会。"

这个回答传递出来的信号是：你是一个有上进心的人，不是一个坐不住的人。

#### 最重要的原则

**说真实的，但说的方式要客观，不带情绪，不评价对方的好坏。**

很多人第一次说离职原因，都会忍不住抱怨上家公司的某个问题。但面试官记得不是那个公司有多差，而是你这个人说话没有分寸。

### 四、面试前我们要做哪些准备？

很多人准备面试，都只准备自己：英文自我介绍、证书、专业问题——你都在准备自己，但是"知己知彼，百战不殆"，你是不是还要了解对方？

#### 常见准备都停留在表面

很多人说："我看一遍官网，读一遍JD，准备一下自我介绍"——这三件事都没错，但都停留在表面。

- 官网是公司想让你看到的东西，不是你真正需要的信息
- JD读了一遍，但你没有把每个要求对应到自己的某段经历上
- 自我介绍准备好了，但是不是每家公司都说同一个版本，没有针对岗位调整？

#### 三层准备法

##### 第一层：真正了解这家公司和业务

不是看官网简介，是**用他们的产品，带着观察去用**，找到你自己的真实感受和问题：你觉得哪个体验好，哪个体验不好，为什么？这些东西说出来，面试官会知道你是真实用过的，不是临时背的。

比如面试一个做轻食的产品岗，你一定要去他们的体验店体验一下产品，跟店员聊聊，了解产品功能和竞品差异。

如果是**to B公司**，你没有办法用他们的产品怎么办？
- 去36氪、虎嗅网搜这家公司名字，看最近有什么样的报道——这些内容都是真实行业信息，不是公关稿，能让你了解这家公司最近在做什么，遇到什么问题，市场怎么看它
- 看BOSS直聘上这家公司的职位页面，不是光看薪资，要看他们最近在招什么岗位、招多少人——你能判断出来这个公司是在扩张还是收缩，哪个业务线在发力

这些信息用在面试里，面试官会觉得你对公司有真实了解。

##### 第二层：把JD拆开来用

JD不是用来读的，是要来拆的：
- 把JD里每一个单独需求单独列出来
- 找你自己经历里对应的例子，一个一个准备案例
- 注意JD里要求的顺序，同一个岗位在不同公司的侧重不一样——有的把执行力放第一位，有的把用户洞察放第一位，顺序反映这家公司真正在乎什么，你准备的时候要把他们最在乎的点放在前面说

还可以去小红书搜这家公司氛围、加班、真实待遇，看近期离职员工的真实评价，帮你判断这家公司是不是你能接受的。**注意看发布时间，半年以上的信息可能就不适用了，要以近期为准。**

##### 第三层：准备你要问他们的问题

面试最后，面试官都会问"你还有什么想问我们的"——这是你展示是否认真考虑过这个机会的最后窗口。

很多人会说"没有了谢谢"，或者问"公司文化怎么样，有没有晋升机会"——这些问题每家公司都能问，说明你没有针对这家公司做过任何思考。**问不好会减分，问得好会加分。**

准备一个**跟这个公司具体业务相关的真实问题**：
- 可以说你用产品时发现的一个疑问
- 可以说你在报道里看到某个业务方向，想了解更多
- 可以问"这个具体岗位在团队里承担什么样的角色"

这样的问题问出来，面试官会觉得你是真正在考虑这个机会，而不是在走流程。

**细节决定差距：**有一个学员，面试前整理了公司最近的行业报道和创始人访谈，面试中说到了创始人在一次采访里提过的一个判断——面试官当场就停下来问他是在哪里看到的，这个细节就让他在这轮面试里和其他候选人完全不一样。这不是运气，是准备的深度不一样。

### 五、你最快什么时候能到岗？

面试官问这个问题，其实已经表达出了强烈的录取意愿。

#### 两种错误回答

1. **随时可以到岗**：随时可以来意味着你没有任何安排，没有其他在推进的机会，你把底牌全亮出去了，HR知道你随时可以来，后续谈判你就没有任何主动权了
2. **需要两三周，但不给理由**：HR不知道你这两三周要干什么，容易觉得你在拖、在等别的公司

#### 正确回答结构

**给出一个具体的时间 + 一个合理的理由 + 表达意向**

**标准说法：**
> "我这边还有一些交接和准备工作要处理，预期可能需要一周左右的时间。如果这边流程推进比较快，我可以尽量配合，因为我对这个岗位还是很有意向的。"

- 一周时间合理，不会让HR觉得你在拖
- 同时给自己争取了一个观察期
- "交接和准备工作"这个理由任何人都能理解
- 最后表达意向，让对方知道你是认真考虑这个机会的

#### 不同情况应对

1. **你手上还有其他公司在推进**：不要说"我在等另一家"，就用"还有一些准备工作要处理"，给自己争取一周时间，足够等大部分公司的结果了

2. **对方说着急，能不能三天内到岗**：不要立即答应，说"我需要确认一下能不能安排好，今天给你答复"——给自己几个小时想清楚，再给对方一个明确的时间点，让对方觉得你对这件事情是慎重的，不是随口一说那么草率

3. **你真的没有其他安排，随时可以去**：也不要说"随时"，就说"大概三天内/一周内可以到岗"就够了。给自己留几天做入职前准备（体检、办银行卡等）是非常合理的，同时也没有把底牌完全亮出去

4. **对方直接问：你还在等其他公司的结果吗？**：可以直接说"有一两家还在推进，不过如果这边条件合适，我会优先确认"——不撒谎，同时表达意向，让对方知道你是认真的

---

### 六、你经历的最大的挑战是什么？你是怎么解决的？

#### 常见问题

1. **选的事情太日常**：比如"有一次deadline很紧，我加班把它做完了"——这在任何岗位上都是正常工作内容，不是挑战
2. **说了很多背景，但不知道你做了什么**：全都是"当时情况很复杂，压力很大，我们团队一起努力"，但你个人做了什么，完全体现不出来
3. **有结果但没有量化**："最终效果还不错，大家都比较满意"——没有数字的结果等于没有结果，面试官感受不到区别

#### 正确回答思路

**STAR法则 + 学到一件事**

- **S (Situation) 背景**：一两句话交代清楚
- **T (Task) 任务**：说清楚重点是什么
- **A (Action) 行动**：要说你的判断和决策，不仅仅是动作
- **R (Result) 结果**：最好有数字，如果实在没有数字，描述清楚达到了什么样的状态
- **最后加一个Learning**：你从这个挑战里学到了什么具体的东西

**最后这一层很多人会忽略，但它是整个回答含金量最高的部分。**你要告诉面试官，你不仅仅能解决问题，还能从经历里提炼出东西，这才是面试官最想看到的。

#### 例子

学生案例：
> （背景）我之前在技工学校做招生工作，一开始老师和家长沟通能力很欠缺。
> （行动）然后我坚持每天和5-10个家长聊天邀约。
> （结果）慢慢锻炼出来了沟通能力，也学会了为人处世。
> （学到了）我发现沟通能力不是天生的，是可以通过刻意练习练出来的。这件事让我明白，面对不熟悉的领域，只要愿意沉下心来持续练习，就能不断进步。这对我今后做任何工作都会有帮助。

这样说，就让面试官觉得你是一个**自我迭代能力很强**的人，愿意不断进步，这会是很大的加分项。

面试短短的几十分钟，能不能让面试官记住你，印象深刻，和回答话术有很大关系。记住这个核心原则：

**给面试官一个具体的画面，用事实和数据说话，而不是堆形容词。**

### 8.1 算法与数据结构

#### 海量 QQ 去重

**【腾讯】面试题**
40 亿个 QQ 号如何去重？仅 1GB 内存。

**解析**：一个整数占 4 个字节，那么 40 亿个 QQ 号需要 16GB 内存，远远超过了 1GB 的限制。我们可以使用位图法来解决，每一位代表一个数字，也即 40 亿 ÷ 8 = 500MB，这样就可以在 1GB 内存内完成去重。

#### N 的阶乘末尾有几个0？

**解析**：判断这个阶乘中会出现多少个5（包括把所有5的倍数拆解得到的5），就有多少个0。

**代码实现**：
```java
public int trailingZeroes(int n) {
    int count = 0;
    while (n > 0) {
        n = n / 5;
        count += n;
    }
    return count;
}
```

#### 设计快速插入和查找的数据结构，你会用什么结构？

**解析**：

如果需要同时满足快速插入和快速查找，根据不同场景可以选择：

1. **平衡二叉搜索树（如红黑树、AVL树）**：
   - 插入和查找都是 O(log n)
   - 支持有序遍历

2. **哈希表（散列表）**：
   - 平均情况下插入和查找都是 O(1)
   - 不支持有序性，存在哈希冲突问题

3. **跳表**：
   - 插入和查找都是 O(log n)
   - 实现相对简单，支持有序遍历，在 Redis 中广泛使用

4. **B树/B+树**：
   - 适合磁盘存储，多路平衡查找树
   - 减少磁盘IO次数，数据库索引用B+树

**场景选择**：
- 内存中存储：跳表或哈希表
- 需要有序性：跳表或红黑树
- 磁盘存储：B+树

#### 进程是分配资源的基本单位，那么这个资源指的是什么？

**解析**：进程是操作系统进行资源分配和调度的基本单位，这里的资源主要包括：

1. **内存空间**：每个进程有独立的地址空间
2. **文件描述符表**：打开的文件、套接字等资源
3. **CPU上下文**：程序计数器、寄存器等
4. **I/O设备**：分配的I/O设备
5. **页表**：用于虚拟地址到物理地址的映射

关键理解：进程之间的资源是相互隔离的，一个进程崩溃不会影响其他进程。线程共享进程的资源，线程是CPU调度的基本单位。

### 8.2 数据库

#### 从 MySQL 千万条数据中搜索到指定数据

**【字节】面试题**

**解析**：首先我们先创建一张千万级数据的表 `test_data`。

**创建表**：
```sql
CREATE TABLE test_data (
    id INT AUTO_INCREMENT PRIMARY KEY,
    random_str VARCHAR(20),
    create_date DATETIME,
    num_value INT
);
```

**生成测试数据**：
```sql
-- 创建存储过程（示例）
DELIMITER $$
CREATE PROCEDURE generate_data(IN total INT)
BEGIN
    DECLARE i INT DEFAULT 0;
    START TRANSACTION;
    WHILE i < total DO
        INSERT INTO test_data (random_str, create_date, num_value)
        VALUES (
            CONCAT('DATA_', LPAD(FLOOR(RAND()*1000000), 6, '0')),
            NOW() - INTERVAL FLOOR(RAND()*3650) DAY,
            FLOOR(RAND()*10000)
        );
        SET i = i + 1;
        IF i % 1000 = 0 THEN
            COMMIT;
            START TRANSACTION;
        END IF;
    END WHILE;
    COMMIT;
END$$
DELIMITER ;
```

**优化查询**：
- 建立合适的索引
- 避免 SELECT *，只查询需要的列
- 使用分页避免一次性查询大量数据
- 考虑分区表

#### 百万级别数据的 Excel 如何快速导入到数据库中

**【阿里】面试题**

**解析**：从 Excel 中读取百万级数据并将其插入数据库时可能遇到的问题：

##### 数据正确性问题
读取和导入过程中会有各种潜在的问题，我们需要妥善处理各类异常情况，在插入数据前，我们需要对数据进行校验，确保数据的正确性。然后在插入过程中处理异常情况，如果单条数据插入失败，我们可以二次尝试，二次尝试仍然失败的话，将数据记录到日志中，方便后续对数据进行二次插入。导入数据过程中还需要注意数据可能出现重复的情况，我们需要采取设定唯一约束的方式来处理，同时记录日志。

##### 内存溢出问题
我们不能一次性将数据读取到内存，需要分批处理数据。我们可以使用 Apache POI 的事件模型来处理 Excel，这种方式适合处理大文件，因为它不会将整个文件加载到内存中，它会按行读取数据，就大大释放了内存的压力。

##### 性能瓶颈问题
针对百万级数据的处理，单线程显然效率很低，所以我们采取多线程处理数据。我们可以将多线程分为两种场景：一种是多线程读取文件操作，另一个是多线程插入数据。当然在数据插入的时候，除了使用多线程技术外，还可以使用批量插入数据的功能更进一步提升速度，可以通过 `values(),(),()` 的方式一次性插入多条数据。

#### Hive 中的排序函数

**解析**：Hive 中的排序分为四种：

- `ORDER BY`：全局排序，只有一个 Reducer
- `SORT BY`：分区内有序，不是全局排序
- `DISTRIBUTE BY`：按照指定字段分组，配合 SORT BY 使用
- `CLUSTER BY`：当分区字段和排序字段相同时，可以替代 DISTRIBUTE BY + SORT BY

**ORDER BY 和 SORT BY 的区别**：
- ORDER BY：全局排序，数据量大会很慢
- SORT BY：每个 Reducer 内排序，只保证区内有序

#### left join 和 right join 的区别

**解析**：
- `LEFT JOIN`（左连接）：保留左表所有记录，右表匹配不到的字段显示 NULL
- `RIGHT JOIN`（右连接）：保留右表所有记录，左表匹配不到的字段显示 NULL

### 8.3 架构设计

#### 为什么复杂的架构一定要做分层设计？

**【腾讯】面试题**

**解析**：分层思想在软件工程中是一种常见的设计方式，它是将整体系统拆分成 N 个层次，每个层次都有自己的职责，多个层次协同提供完整的功能。软件架构整体上会被分为三层：UI 层、业务逻辑层、数据访问层。

- **UI 层**：进行数据展示以及和用户进行交互
- **业务逻辑层**：负责业务逻辑的实现，可能会综合调度多个数据访问层
- **数据访问层**：主要和数据库进行通信

**分层架构的优点**：
1. **职能分离，隔离关注点**，每个人关注自己所在分层的任务即可
2. **解耦合，方便系统扩展**。当项目分层后，每层都仅能和自己相关的层做数据交互，不能跨层调度，这样就做到了解耦，也方便系统的扩展。我们可以将数据访问和业务逻辑层独立部署。

#### 设计一个分布式缓存系统需考虑哪些点?

设计一个分布式缓存系统需要考虑以下几个方面：

1. **数据分区和一致性**
   - 如何将数据均匀分布到多个节点上，确保每个节点负载均衡
   - 采用何种一致性模型（如强一致性、最终一致性）来保证数据的一致性

2. **高可用性和容错性**
   - 如何设计系统的冗余机制，确保在部分节点故障时，系统仍能提供服务
   - 采用何种故障检测和恢复机制，确保系统的可用性

3. **数据过期和淘汰策略**
   - 如何设置数据的过期时间，确保缓存数据的时效性
   - 采用何种淘汰策略（如LRU、LFU）来管理缓存空间，确保缓存的高效利用

4. **并发控制和锁机制**

#### 分布式ID生成器怎么实现？

**解析**：常见的分布式ID生成方案有几种：

1.** UUID **
   - 优点：简单，本地生成，无网络延迟
   - 缺点：无序，占空间大，信息泄露风险

2.** 数据库自增ID **
   - 基于数据库的自增主键，不同节点设置不同步长
   - 优点：有序，实现简单
   - 缺点：依赖数据库，存在单点问题，扩容麻烦

3.** 雪花算法（Snowflake）**
   - 时间戳 + 工作机器ID + 序列号，64位
   - 优点：高性能，低延迟，有序
   - 缺点：依赖时钟同步，回拨问题

4.** Redis 生成 **
   - 利用 Redis 的 INCR 命令生成递增ID
   - 优点：性能好，有序
   - 缺点：依赖 Redis，需要持久化保证不重复

5.** 美团 Leaf **、** 百度 UIDGenerator **等开源方案**
   - 分段缓存策略，解决性能和扩展性问题

**选型建议**：
- 对性能要求高：雪花算法
- 需要有序：数据库或Redis
- 简单场景：UUID

#### 怎么理解幂等性，有遇到过实际场景么？

**解析**：**幂等性**就是多次执行同一个操作和执行一次产生的效果是相同的。

**常见场景**：

1. **支付场景**：用户支付一次不能扣两次钱
2. **订单创建**：不能重复创建同一个订单
3. **接口重试**：网络超时重试时不能产生重复数据
4. **MQ消息消费**：消息重复消费不能产生副作用

**实现方式**：

1. **唯一索引**：防止新增重复数据
2. **token机制**：操作前获取token，操作时验证token
3. **乐观锁**：基于版本号实现
4. **悲观锁**：操作前加锁
5. **状态机**：根据状态判断，只有特定状态才能流转

**实际例子**：
- 支付系统：同一笔订单号，多次支付只扣一次钱
- 创建订单：前端防重 + 后端唯一索引，避免重复创建

### 8.4 网络

#### 什么是网络四元组

**【京东】面试题**

网络四元组是 TCP 协议中用于唯一标识一个连接的核心概念，它由源 IP 地址、目标 IP 地址、源端口号、目标端口号四个要素组成。

1. **源 IP 地址**：发起连接的客户端 IP 地址
2. **目标 IP 地址**：服务端的 IP 地址
3. **源端口号**：客户端随机分配的端口（范围：1024-65535）
4. **目标端口号**：服务端监听的端口

五元组是在四元组基础上增加了传输层协议（TCP/UDP）。

#### HTTP 1.1 -> 2.0 -> 3.0 演进

**HTTP/1.1**：
- 基于TCP，文本协议
- 队头阻塞问题，同一连接只能处理一个请求
- 支持长连接（Keep-Alive）
- 缓存机制（Cache-Control、ETag）

**HTTP/2**：
- 二进制分帧，多路复用
- 头部压缩（HPACK）
- 服务器推送
- 解决了队头阻塞问题（应用层）

**HTTP/3**：
- 基于QUIC协议，UDP之上
- 解决了TCP队头阻塞问题
- 更好的拥塞控制
- 连接迁移，0-RTT握手

#### HTTPS 状态码

常见状态码分类：
- **1xx**：信息性状态码
- **2xx**：成功（200 OK，201 Created，204 No Content）
- **3xx**：重定向（301永久重定向，302临时重定向，304未修改）
- **4xx**：客户端错误（400错误请求，401未授权，403禁止访问，404未找到，405方法不允许）
- **5xx**：服务器错误（500服务器错误，502网关错误，503服务不可用，504网关超时）

### 8.5 Java 相关

#### IOC、AOP、@Autowired 和 @Resource 注解的区别

**IOC（控制反转）**：将对象创建和依赖关系管理交给Spring容器，不是开发者自己new对象。

**AOP（面向切面）**：面向切面编程，不修改原有代码的基础上增加额外功能，如日志、事务、权限校验。

**@Autowired vs @Resource**：
- `@Autowired`：Spring提供的注解，默认按类型装配，结合`@Qualifier`按名称装配
- `@Resource`：JSR-250标准注解，默认按名称装配，找不到名称再按类型装配
- `@Autowired` 依赖注入是通过AutowiredAnnotationBeanPostProcessor处理的，而`@Resource`是通过CommonAnnotationBeanPostProcessor处理

#### Spring Boot 自定义 Starter

自定义Starter需要包含两个部分：
1. **autoconfigure模块**：包含自动配置代码
2. **starter模块**：只用来依赖管理，引入autoconfigure

关键步骤：
- 定义配置属性类
- 编写自动配置类
- 在`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`中注册自动配置
- 提供默认配置条件（`@ConditionalOnClass`、`@ConditionalOnProperty`等）

#### 数据倾斜 —— 如何发现、判断、解决？

**阿里判断规则**：
- 通过 Spark UI 观察各个Task执行时间和处理数据量
- 如果某个Task处理的数据量远远大于其他Task，就是数据倾斜

**发现方法**：
- 看日志：某个Task执行特别慢，一直卡住
- 看监控：大部分Task很快完成，少数几个跑很久
- 看Shuffle数据分布：某个分区数据量过大

**解决方法**：
1. **加盐优化**：对key增加随机前缀，分散到不同Reduce
2. **两阶段聚合**：先局部聚合，再全局聚合
3. **过滤空key**：如果是null值导致倾斜，提前过滤
4. **增大并行度**：增加Reduce数量，每个分区数据量变少
5. **优化Join**：将小表广播（broadcast join），避免大表Shuffle

#### maxwell 在项目里是怎么运行的？

Maxwell是一个CDC工具，用于监听MySQL的binlog，将数据变更输出到Kafka等消息队列。

**运行流程**：
1. 连接MySQL，开启binlog监听
2. 捕获数据的增删改操作
3. 将变更数据转换成JSON格式
4. 发送到Kafka
5. 下游Flink消费Kafka消息，进行实时数据处理

#### 了解实时数仓吗，Flink 大概是做什么的？

实时数仓是对数据进行实时处理和分析的数据仓库，满足低延迟的数据分析需求。

**Flink**：是一个分布式流处理引擎，支持高吞吐量、低延迟的流数据处理，可以做：
- 实时ETL
- 实时聚合统计
- 事件驱动应用
- 窗口计算

- 升级硬件

**缓存**：

**优化**：

- 减少不必要的查询
