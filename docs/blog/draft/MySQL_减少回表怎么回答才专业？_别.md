# MySQL 减少回表怎么回答才专业？别只会说加索引

面试官如果问你 MySQL 如何减少回表，你千万不要只说一句“加索引”，这个回答太浅了。

## 什么是回表？

在 **InnoDB** 里面，**主键索引**存的是整行数据，但是**普通索引**（也就是二级索引）叶子节点里面存的不是整行数据，而是**索引字段 + 主键值**。

比如有一张用户表，主键是用户 ID，普通索引是手机号。当你根据手机号查用户信息时，MySQL 会先走手机号索引，找到对应的用户 ID。但如果你要查的字段不在手机号索引里面，MySQL 还得拿着这个用户 ID 再去**主键索引**里面查一次完整数据。

这第二次查主键索引的过程，就叫**回表**。

---

## 为什么要减少回表？

因为回表本质上就是**多查一次索引树**。一次查询如果只命中几条数据，问题不大；但如果一次查几千条、几万条数据，每一条都要回表，那成本就会非常高。

尤其是**分页查询、范围查询、列表查询**里面，回表次数一多，性能就会明显下降。

所以优化目标很简单：**让查询尽量在二级索引里就完成，少去主键索引拿整行数据**。

## 减少回表的五种方法

### 第一种：使用覆盖索引

这是最常见也是最核心的方式。

所谓**覆盖索引**，就是查询需要的字段都已经包含在索引里面。

比如你有一个联合索引：`手机号、用户名、状态`。如果你的 SQL 只查这几个字段（手机号、用户名、状态），那 MySQL 直接在这个索引里就能拿到结果，不需要再回表。

所以面试里可以这样说：如果业务只需要展示列表页的少量字段，就不要直接查整行数据，而是把高频查询字段设计进联合索引，让它形成覆盖索引。

### 第二种：不要查询全部字段

很多回表问题其实是自己写出来的。

比如明明页面只需要用户 ID、昵称、头像，你却直接 `SELECT *` 查询所有字段，这样就算命中了普通索引，也大概率要回表。

所以减少回表的一个基本原则就是：**需要什么字段就查什么字段**。不要为了省事儿把大字段、无关字段一起查出来，尤其是文本字段、备注字段、扩展字段，这些字段最好不要出现在高频列表查询里。

### 第三种：合理设计联合索引

不是说把所有字段都塞进索引。索引字段太多也会带来写入成本、存储成本和维护成本。

正确做法是**围绕高频查询场景设计联合索引**。

比如订单列表经常按用户 ID、状态、创建时间查询，并且只展示订单号、金额、状态、创建时间，那就可以考虑建立一个联合索引：`(用户 ID, 状态, 创建时间, 订单号, 金额)`。

这样查询时既能利用筛选条件，又能直接从索引里拿到展示字段，这就减少了大量回表。

### 第四种：先查主键，再回查详情

这个在**深分页**场景里特别常见。

比如你直接查第 10 万页的订单列表，如果直接查完整字段，MySQL 可能要扫描大量索引并且产生大量回表。

优化方式是：

1. **第一步**：先通过覆盖索引查出这一页的主键 ID；
2. **第二步**：再根据这些 ID 回表查完整数据。

这样回表次数就从大量无效回表，变成只对最终需要的几十条数据回表。这也是很多深分页优化的核心思路。

### 第五种：冷热字段拆分

有些表里面既有高频访问字段，也有低频访问的大字段。

比如用户表里面，昵称、头像、状态是高频字段；个人简介、配置 JSON、备注信息是低频字段。

如果这些字段都放在一张大表里，高频查询很容易被迫读取很多不必要的数据。这时候可以考虑做**垂直拆分**：主表放高频字段，扩展表放低频字段。这样列表查询可以更容易被索引覆盖，也能减少无意义的回表和数据读取。

---

## 一句话总结

减少回表的本质，就是让 MySQL **少做一次主键索引查询**。能在二级索引里解决的，就不要再回主键索引拿整行数据。
