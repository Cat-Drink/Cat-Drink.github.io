## 一、四大核心上下文工程技巧

### 技巧 1：写入（持久化上下文，跨会话复用）

所属系列：TRAE 如何结合 MCP 实现上下文工程（二）

#### 核心说明

TRAE\+MCP 落地上下文工程四大核心技巧包含：写入、选取、压缩、隔离。
「写入」核心：将用户偏好、项目配置、核心结论等高价值信息持久化，突破单次对话限制；依托 MCP 对接向量数据库、知识库完成上下文持久存储。

#### 操作步骤

1. 添加存储型 MCP Server：优先选用`Knowledge Graph Memory MCP`，可把上下文以知识图谱结构化存储，方便后续关联检索；

2. 新建自定义智能体：进入 TRAE 智能体管理页，点击「新建智能体」，绑定上述 MCP Server；

3. 配置写入规则：通过系统提示词划定持久化信息类型与存储逻辑，示例提示词：

> 你是项目助手，需将用户询问的核心技术点、项目配置信息通过 Knowledge Graph Memory MCP 以知识图谱形式写入存储；定义信息关联规则（如 “技术框架版本” 关联 “初始化方法”“适用场景”）；后续对话中，自动从知识图谱读取关联上下文。
> 
> 

#### 实践效果

用户询问项目技术框架版本，智能体通过图谱 MCP 绑定版本、项目名称、使用模块存入图谱；后续提问该框架初始化方式时，自动读取关联的初始化步骤、注意事项上下文，无需重复提问，信息完整性更高。

### 技巧 2：选取（动态筛选相关上下文，提升准确性）

所属系列：TRAE 如何结合 MCP 实现上下文工程（二）

#### 核心说明

「选取」是上下文工程核心环节；依靠 MCP 调用外部检索工具，动态抓取和当前任务强相关信息，屏蔽无关干扰；典型搭配 RAG 检索增强，MCP 对接知识库完成精准检索。

#### 操作步骤（场景：提取任务配套文档信息）

1. 前置部署`Sequential Thinking MCP Server`，用于分步拆解检索逻辑，提升匹配精度；

2. 切换内置 MCP 智能体：打开 TRAE AI 侧边栏，将`@builder`切换为`@builder with MCP`，自动加载全部已配置 MCP 服务；

3. 模型选型：侧边栏选用 Gemini 2\.5 Pro 等高能力大模型；

4. 输入标准化检索提示词，示例：

> 请完成某项目核心功能的开发方案撰写，先通过 Sequential Thinking MCP 拆解任务步骤，确保上下文与步骤精准匹配。
> 
> 

#### 实践效果

智能体自动拆解任务为四阶段：需求分析→方案框架搭建→核心功能设计→方案验证；每一步匹配对应文档上下文（需求步骤匹配需求文档、设计步骤匹配技术选型文档）；输出方案逻辑分层清晰，不会因混杂冗余上下文出现逻辑混乱。

### 技巧 3：压缩（精简上下文，降低 Token 消耗）

所属系列：TRAE 如何结合 MCP 实现上下文工程（二）

#### 核心说明

上下文文本过长时，借助 MCP 调用压缩工具、自定义精简规则，用更少 Token 承载完整核心信息；TRAE MCP 可对接专业文本压缩服务做轻量化处理。

#### 操作步骤

1. 双服务部署：同时添加**文本压缩 MCP Server** \+ `Sequential Thinking MCP Server`；用顺序思考 MCP 划定压缩优先级（优先保留 API 参数、步骤要点，再删减描述性文字）；

2. 配置压缩策略提示词，规则示例：

> 你需先通过 Sequential Thinking MCP 拆解上下文核心信息层级，再通过 MCP Server 对超过 500 字的历史对话、文档片段进行压缩；优先保留核心信息（如技术参数、步骤要点、知识关联关系），删除冗余描述；压缩后的上下文需满足模型 Token 限制，且保持信息层级清晰。
> 
> 

#### 实践效果

上传 1000 字项目文档片段，询问文档核心内容；智能体经 MCP 压缩至 300 字关键要点，上下文信息无丢失，大幅削减 Token 开销。

### 技巧 4：隔离（避免信息干扰，提升复杂任务效率）

所属系列：TRAE 如何结合 MCP 实现上下文工程（二）

#### 核心说明

「隔离」依托**多智能体架构 \+ MCP 资源隔离配置**；各子智能体只处理自身业务域信息，仅向主智能体上报核心结论；解决多模块项目跨域长上下文互相干扰、性能下滑问题；不同模块配置独立 MCP 知识库实现上下文隔离。

#### 操作步骤（场景：多模块项目协同管理）

1. 部署独立 MCP 资源：为每个业务模块搭建专属知识库 MCP Server，模块间资源完全独立；

2. 搭建分层多智能体：

    - 子智能体 1（模块 A 专用）：绑定模块 A 专属 MCP，提示词约束：仅处理模块 A 查询分析，只输出核心结论；

    - 子智能体 2（模块 B 专用）：绑定模块 B 专属 MCP，配置逻辑同子智能体 1，只聚焦 B 模块；

3. 配置总控主智能体：绑定全部子智能体，提示词约束：接收用户需求，按业务模块分发任务给对应子智能体，汇总各子智能体核心结论统一输出，杜绝模块上下文交叉干扰。

#### 实践效果

用户提问「模块 A 与模块 B 接口对接方案」；主智能体分别调度两个子智能体；子 A 提取 A 接口定义、子 B 提取 B 接口需求，二者仅返回参数、适配要点；主智能体整合输出完整对接方案，彻底隔绝两边冗余文档信息的互相干扰。

## 二、进阶优化：MCP \+ 智能体的上下文工程最佳实践

所属系列：TRAE 如何结合 MCP 实现上下文工程（三）

### 1\. 按需选型 MCP Server

|MCP 服务|适用场景|
|---|---|
|Sequential Thinking MCP（免费）|复杂任务分步拆解、上下文有序分层管理|
|Knowledge Graph Memory MCP|结构化信息存储、知识关联图谱搭建，提升上下文复用 \& 检索效率|

### 2\. 优化智能体系统提示词

在提示词内硬性约束 MCP 调用规范，示例约束项：

- 调用 Sequential Thinking MCP 必须明确任务拆解层级；

- 调用知识图谱 MCP 必须提前定义信息关联规则；

- MCP 调用失败时自动提示用户检查 Server 配置，减少无效重试。

### 3\. 监控 MCP 调用日志

进入 TRAE MCP 设置页面，查看各服务调用日志；分析上下文选取准确率、文本压缩效果，反向迭代优化 MCP 参数与智能体提示词。

## 三、常见问题排查

所属系列：TRAE 如何结合 MCP 实现上下文工程（三）

1. **MCP 调用失败**

    - 排查网络连通性、Server 内 API Key/Token 填写正确性；

    - 本地 Local 类型 Server：检查环境是否安装 NPX/UVX 运行依赖。

2. **上下文选取不精准**

    - 修改智能体提示词，明确限定检索关键词范围；

    - 替换精度更高的第三方 MCP 检索工具。

3. **Token 消耗过高**

    - 强化压缩规则，对接压缩效率更高的 MCP 文本服务；

    - 收紧选取逻辑，过滤掉低相关度上下文内容。
