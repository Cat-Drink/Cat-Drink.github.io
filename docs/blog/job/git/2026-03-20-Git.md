### 面试

面试官:

由于工程架构的变动，现在A仓库被废弃了，创建了一个与A 仓库内容完全一样的仓库B，但是有一个同事还不知道这件事，不小心往A仓库提交了好多代码，问现在如何

将他提交到废弃仓库A的commit

转移到新仓库B?

首先思考:

最容易想到的一定是“笨办法

把所有改动的实件复制粘贴过去

但这是初级程序员的做法!如果改动的文件超级多，那复制粘贴要搞到猴年马月去!

这种操作一定是需要通过 Git 操

作来完成的，才足够规范和便捷。

接着思考:

哪种 Git 操作可以完成将内容从

一个地方同步到另一个呢?

我们常常通过 merge 操作将一个分支的改动同步到另一个分支。很多程序员会选择用 rebase

操作代替 merge 操作，也能达到

类似的效果。

这个回答及格了吗:

根据以上思考过程，我们可以回答可以使用 merge 操作或者 rebase操作，将A仓库的commit 同步到

B 仓库。

人

问题在于:

① rebase 操作真的可以跨仓库执行吗?

操作的细节呢?有没有更完美的方案?

这个回答我最多给60分!

满分回答在这里:

我们的需求是在两个原本内容相同的仓库之间同步一些增量的改动，那么首先一定要把两个仓库关联起来，才能实现，这是大前提!如何关联?在B仓库使用:

git remote add arepo https://xxx/A

即可将A 仓库添加为B仓库的一个远程代码源!并命名为了 arepo

C

添加了远程源之后我们需要将A仓库代码拉取下来:

git fetch repo

此时在A仓库找到需要同步到B仓

库的 commit hash ，假如该哈希为

hashxxx，那么只需要执行:

git cherry-pick hashxxx

*每个 git commit 都会对应一个唯一的哈希值喔~可以通过此确定到某一次具体的提交!

此时如果没有任何冲突，那么就会

直接把目标 commit 同步过来啦!

如果有冲突，那就和 merge 操作

一样解决冲突就好啦!

最后一定要把之前添加的 A仓库的

远程源给移除，把B仓库恢复成最

开始的样子:

git remote remove arepo

为什么不用merge:

看起来添加 A 仓库为远程源之后用 merge操作貌似也能达到一样的目的诶!那么为什么满分回答一定是 cherry-pick 呢?

这是因为:它们的粒度不同!

① merge 操作是分支粒度的，只能将整个分支 merge 到另一个分支上;

② pick 操作是 commit 粒度的，我可以精确控制我要把哪些commit 搬过来，不想要的我可以不搬!

### 业务流程

1.在分支开发

2.将master同步给开发分支

3.合并冲突并解决

4.推送pr

日常工作中，我们可能在 Git 使用上频繁交互的流程大致会是这样的（不同规范下会有一些区别，但是大差不大）：

1. 来了一个新需求，我们会从 master checkout 一个新的 feature 分支出来进行开发。
2. 开发完某个功能点，我们会执行 `git add` 将代码提交到暂存区。
3. 执行 `git commit` 将代码提交到本地仓库
4. 执行 `git push` 将代码提交到远端分支
5. 当我们开发完所有需求之后，可能会设立一个专门的测试分支比如名叫 dev 的分支，那么我们就把代码合并到这个测试分支上，发布测试环境进行测试。
6. 测试完毕之后，我们就需要合代码了，这个时候，我们可以发起一个 merge request，将我们的代码走 CR 流程合并到 master 分支。
7. 在提交 MR 的过程中，我们一般需要先自己预先将 master 分支的代码合并到当前需要被合并的分支，提交并解决冲突。

### 概念

版本控制是一种记录文件内容修改记录的变化，从而让用户能够查看历史版本，方便版本切换。 更新最新的版本

**从**个人操作过渡到团队开发

**版本控制功能**

- **协同修改：**多人并行不悖的修改服务器端的同一个文件。
- **数据备份：**不仅保存目录和文件的当前状态，还能够保存每一个提交过的历史状态。
- **版本管理：**在保存每一个版本的文件信息的时候要做到不保存重复数据，以节约存储空间，提高运行效率。这方面 SVN 采用的是**增量式**管理的方式，而 Git 采取了文件系统**快照**的方式。
- **权限控制：**对团队中参与开发的人员进行权限控制。对团队外开发者贡献的代码进行审核——Git 独有。
- **历史记录：**查看修改人、修改时间、修改内容、日志信息。将本地文件恢复到某一个历史状态。
- **分支管理：**允许开发团队在工作过程中多条生产线同时推进任务，进一步提高效率。

### 类型概述

#### 集中式版本控制工具

```
cvs，svn，vss
```

- 有一个单一大的集中管理的服务器，保存所有文件的修订版本，而协同工作的人通过客户端连接到服务器，取出最新的文件或者提交更新
- 好处：观察到其他人的行为，管理员可以轻松掌控每个开发者的情况
- 缺陷：**中央服务器的****单点故障**：如果服务器宕机，那么所有人无法提交，也无法协同工作。

##### SVN(SubVersion)

SVN是**集中式版本控制系统**，版本库是集中放在中央服务器的.

工作流程如下:

1. 从中央服务器远程仓库下载代码
2. 修改后将代码提交到中央服务器远程仓库

- 优点: 简单,易操作
- 缺点:所有代码必须放在中央服务器
  - 服务器一旦宕机无法提交代码,即容错性较差
  - 离线无法提交代码,无法及时记录我们的提交行为

#### 分布式版本控制工具

```
git,mercurial,bazaar
```

- 客户端提取的不是最新的文件快照，而是把代码仓库完整地镜像下来（**本地库****`clone`**），这样任何一处协同工作的文件发生故障，事后都可以用其他客户端的本地仓库进行恢复。因为每个客户端的每一次文件提取操作，实际上都是一次对整个仓库的**完整备份**。

> 代码托管中心：远程库
>
> 客户端：本地库

好处：**解决了集中式版本控制系统的缺陷：不再有单点故障**

1. 服务器断网的情况下也可以进行开发（因为版本控制是在本地进行的）
2. 每个客户端保存的也都是整个完整的项目（包括历史记录，更加安全）

### git文件状态讲解

Git工作目录下的文件存在两种状态：

1. **untracked 未跟踪（未被纳入版本控制）** :  比如新建的文件(此时文件夹上没有图标或者有一个"问号")
2. **tracked 已跟踪（被纳入版本控制）**     
   1. **Staged 已暂存状态**            : 添加 但未提交状态(此时文件夹上有一个"加号")
   2. **Unmodified 未修改状态**        : 已提交(此时文件夹上有一个"对号")
   3. **Modified** **已修改状态**          : 修改了,但是还没有提交 (此时文件夹上有一个"红色感叹号")

这些文件的状态会随着我们执行Git的命令发生变化
