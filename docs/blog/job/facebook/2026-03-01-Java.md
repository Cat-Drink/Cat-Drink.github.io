## volatile的作用是什么?

**两个核心作用：**
- 保证**可见性**：一个线程改了volatile变量，另一个线程能立刻看到最新值，不会读到过期数据
- 禁止**指令重排序**：编译器和CPU不会乱 reordered指令，防止写操作还没完成就读取，典型就是单例模式的double-check

## 堆和栈的区别？

**一句话说清：**
- **栈**：运行的时候用，放局部变量、方法参数、返回地址，每个线程一个栈，自动分配释放，速度快，空间小
- **堆**：放对象实例和数组，所有线程共享，手动GC管理，空间大，速度慢点，容易内存泄漏

## Java的双亲委派机制解释一下？

**一句话：** 类加载的时候，先让爹（父加载器）找，爹找不到了自己才去找，保证Java核心库的类不会被替换
流程：一个类加载请求过来 → 先委托给父加载器 → 父加载器找不到，才自己尝试加载
好处：防止重复加载同一个类，防止核心API被篡改，比如你自己写个`java.lang.String`也不会覆盖JDK的

### 那像上面的JDBC由于是第三方库，双亲委派机制不生效，会用什么方法加载第三方类呢？

用**线程上下文类加载器** (`Thread.currentThread().getContextClassLoader()`)
因为JDBC驱动是第三方的，在classpath下，启动类加载器（负责加载JDK核心类）找不到，所以得用上下文类加载器去加载应用路径下的类
说白了就是破坏了双亲委派，反过来用，这就是SPI的做法

## spring循环依赖怎么解决？

一句话：用**三级缓存**解决，只解决**单例bean的setter/属性注入**循环依赖，构造器循环依赖解决不了

过程：

2. A注入B，B发现依赖A
3. B直接从三级缓存拿A（哪怕A还没完全创建好）
4. B创建完成，放一级缓存
5. A拿到B，创建完成，放一级缓存

## spring三级缓存的数据结构是什么？

都是ConcurrentHashMap：
- **一级缓存（singletonObjects）**：放完全创建好的成熟bean，key是beanName，value是bean实例
- **二级缓存（earlySingletonObjects）**：放提前暴露的半成品bean（已经实例化，但还没注入属性），解决循环依赖中的重复创建
- **三级缓存（singletonFactories）**：放bean工厂，key是beanName，value是ObjectFactory，用来生成早期引用，解决aop代理的循环依赖

## java 中方法重载和重写的区别

 ◼ 重载：就是让类以统一的方式处理不同类型数据的一种手段，调用方法 时通过传递给它们的不同个数和类型的参数来决定具体使用哪个方法， 重载是一个类中多态性的一种表现。  
 ◼ 重写：当子类需要修改父类的一些方法进行扩展时，这样的操作就称为 重写  
 ⚫ 区别：  
 ◼ 重载发生在同一个类中，而重写发生在有继承关系的子类中  
 ◼ 重载的方法参数列表不同（包括参数顺序、个数、类型），而重写的方 法参数列表（包括参数顺序、个数、类型）相同  ◼ 重载的方法返回值类型和访问修饰符都没有限制，而重写子类方法的返 回值类型必须和父类相同，对于访问修饰符，子类的访问范围要大于等 于父类的访问范围  
 ⚫ 拓展：  
 ◼ 对于重载而言，在方法调用之前，就已经确定所要调用的方法，也叫作 静态绑定（早绑定） 编译  
 ◼ 对于多态而言，在方法调用时，才会确定所要调用的具体方法，也叫作 动态绑定（晚绑定） 运行

Java异常分为Error和Exception两种，其中Error表示程序无法处理的错误， Exception 表示程序本身可以处理的异常。这两个类均继承自Throwable。Error 常见的有StackOverFlowError，OutOfMemoryError 等。Exception 可分为运行 时异常和非运行时异常。对于运行时异常比如RuntimeException，可以利用 try catch的方式进行处理，也可以不处理。对于非运行时异常比如IOException， 必须处理，不处理的话程序无法通过编译。

### 设计模式七大原则

1. **单一职责原则**：一个类只负责一项职责
2. **接口隔离原则**：使用多个隔离的接口，比使用单个接口要好

5. **开闭原则**：对扩展开放，对修改关闭
6. **迪米特法则**：一个类应当对其他类知道的越少越好，只和朋友通信
7. **合成复用原则**：尽量使用组合/聚合，而不是继承

### 单例模式

**定义**：保证整个软件系统中，对某个类只能存在一个对象实例，并且该类只提供一个获取对象实例的静态方法

**实现方式**：
- **饿汉式（静态常量）**：类加载时就创建实例，在类中定义静态常量，静态方法返回该实例
- **饿汉式（静态代码块）**：将实例创建放在静态代码块中，原理同上
- **懒汉式（线程不安全）**：第一次使用时才创建，`if(instance == null) instance = new Singleton()`，多线程下不安全
- **懒汉式（线程安全，同步方法）**：给`getInstance()`方法加上`synchronized`，保证线程安全，但效率低
- **懒汉式（线程安全，同步代码块）**：只在创建对象时加同步，并不能真正保证线程安全
- **双重检查**：两次判断`instance`是否为空，在第二次判断前加同步，效率高且安全，需要`volatile`禁止指令重排
- **静态内部类**：利用静态内部类延迟加载，外部类加载时不加载内部类，调用时才加载，线程安全
- **枚举**：使用枚举实现，天然线程安全，防止反序列化重新创建对象，推荐使用

**JDK源码例子**：`java.lang.Runtime` 就是经典的单例模式

## java 中==和 equals 的区别 
- 对于基本数据类型，==比较的是对应的值，对于引用数据类型，==比较的是 地址值
- equals 方法如果未被重写，其作用和==一致，但是通常会重写该方法，比如 String 类型，equals 方法可以用来比较String的变量值
※※为什么有equals方法还需要hashcode方法（为什么重写equals方法要重写hashcode方法）
在Java中，重写equals方法时必须同时重写hashCode方法，这是为了保证对象在基于哈希的集合（如HashMap、HashSet）中能正确工作
equals方法和hashcode方法通常是在集合插入元素时配合使用的。在插入对象的时候，先调用该对象的hashcode方法，得到哈希码值，如果table中不存在该哈希码 值，那么直接插入，但是如果table中存在该哈希码值，就会继续调用 equals 方法，判断两个对象是否真的相同，相同的就不存，不相同才存进去。
- Java规范中规定
  1. 如果两个对象通过equals方法比较相等，它们的hashCode必须相同
  2. 如果两个对象的hashCode相同，它们不一定相等，此时才可通过equals进一步比较确认
如果只重写了equals方法，那么当两个对象的属性值相等的时候会返回true（说明对象相同），但是因为没有重写hashcode方法，显然hashcode值就不一样，这样就会和规定产生矛盾。

## java 的深拷贝和浅拷贝的区别
对于基本类型，深拷贝和浅拷贝都是一样的，都是对原始数据的复制，修改原始数据，不会对复制数据产生影响。
两者的区别，在于对引用类型的复制
- 对于浅拷贝，只会复制引用，没有复制指向的对象，所以对原始对象的修改，会对复制对象产生影响
  - 实现方式：继承Cloneable接口，重写clone()，因为Object中的clone方 法就是浅拷贝，所以对于浅拷贝实现就是在clone()中直接调用 super.clone()即可
- 对于深拷贝，它是复制该引用指向的对象，所以修改原始对象，不会对复制对象产生影响
  - 实现方式：对于深拷贝，一般需要调用强制类型转换操作

1.监控内存使用情况:。使用系统监控工具(如top、htop)或编程语言提供的内存监控工具，实时监控程序的内存使用情况。
。观察内存使用量是否持续增长，如果是，可能存在内存泄漏或死循环2.生成堆转储文件:。在程序运行过程中，生成堆转储文件(如Java中的Heap Dump)，记录程序在某一时刻的内存快照。。可以使用编程语言提供的工具或第三方工具生成堆转储文件。3.分析堆转储文件:
。使用内存分析工具(如MAT、JProfler)分析堆转储文件，查找内存泄漏的原因。

。根据内存分析结果，定位到可能存在问题的代码段，进行代码审查，

## Hashmap 的底层实现原理

HashMap底层由数组和链表（JDK 1.8后引入红黑树）组成。数组称为哈希桶数组，每个元素是一个链表或红黑树的头节点，用来存储键值对。
通过对键对象调用hashCode方法获取哈希值，再对哈希值进行一系列位运算和取模运算，将其映射到哈希桶数组的特定索引位置。例如，在JDK 1.8中，会先对哈希值进行高位与低位的异或运算，以降低哈希冲突的概率，然后通过(n - 1) & hash计算索引，其中n是哈希桶数组的长度。
当要存储一个键值对时，先计算键的哈希值，确定其在哈希桶数组中的索引。如果该索引位置为空，直接将键值对存储在该位置。如果该位置已有元素，即发生哈希冲突，则将新的键值对以链表的形式插入到该位置。当链表长度超过阈值（JDK 1.8中为8）且哈希桶数组长度大于等于64时，链表会转化为红黑树，以提高查找效率。
查找时，先计算键的哈希值，找到对应的哈希桶数组索引位置。如果该位置是红黑树，通过树的查找算法查找键值对；如果是链表，则遍历链表查找。若找到键相等的节点，则返回对应的值。
当HashMap中的键值对数量达到一定比例（负载因子，默认0.75）时，会触发扩容操作。扩容会创建一个新的更大的哈希桶数组，然后将原数组中的元素重新哈希并复制到新数组中。这是为了保持哈希表的性能，避免哈希冲突过多导致查找、插入和删除操作的时间复杂度退化。

## 如何解决哈希冲突？

1. 开放定址法
  - 线性探测法：当发生哈希冲突时，按照一定的步长在哈希表中依次探测下一个空位置，直到找到空位置插入新元素。例如，若当前元素的哈希值对应的位置已被占用，就向后移动一个位置继续检查，若还被占用则继续向后移动，直到找到空位置。
  - 二次探测法：与线性探测法类似，但步长不是固定值，而是随着探测次数的增加以二次方的方式变化，这样可以减少哈希冲突的聚集，提高哈希表的性能。
2. 链地址法
  - 将哈希值相同的元素存储在同一个链表中。当插入新元素时，若发生哈希冲突，就将新元素插入到对应的链表中。在查找元素时，先通过哈希值找到对应的链表，然后在链表中逐个比较查找目标元素。JDK 1.8后的HashMap在链表长度超过一定阈值时会将链表转化为红黑树，以提高查找效率。

## 时间复杂度的定义

时间复杂度是用来**衡量算法运行时间随数据规模增长的趋势**，它不代表算法实际运行的具体时间，而是描述算法执行效率随输入规模增大时的变化规律。

常用大O符号表示法：`T(n) = O(f(n))`，其中：
- `n` 是数据规模
- `f(n)` 是算法执行次数的函数
- 只保留最高次项，忽略常数系数和低次项

常见时间复杂度（从小到大）：
- **O(1)**：常数时间，比如直接访问数组元素
- **O(log n)**：对数时间，比如二分查找
- **O(n)**：线性时间，比如遍历一遍数组
- **O(n log n)**：线性对数，比如快速排序、归并排序
- **O(n²)**：平方时间，比如冒泡排序、双重循环
- **O(2ⁿ)**：指数时间，暴力递归解法
- **O(n!)**：阶乘时间，全排列问题

## 什么情况下会发生缓存雪崩?如何解决?

**发生场景**：
缓存雪崩是指大量缓存同时失效，导致所有请求都打到数据库上，数据库压力骤增甚至宕机。

常见原因：
1. **同一时间大量缓存key同时过期**：比如热点数据集中在同一个时间点设置了过期
2. **缓存服务器宕机**：整个缓存服务不可用，所有请求都直接访问数据库

**解决方法**：

1. **错开过期时间**：给不同key的过期时间加上随机值，避免同时过期
   ```
   expire = 300 + random(0~100)
   ```

2. **设置热点key永不过期**：对于访问频率极高的热点数据，不设置过期时间，后台异步更新

3. **加锁排队**：缓存失效时，只允许一个线程去查询数据库并重建缓存，其他线程等待

4. **缓存高可用**：搭建缓存集群（Redis主从+哨兵/Cluster），避免单点故障

5. **服务降级**：缓存失效时返回默认值或降级数据，不让数据库直接承压

## 常见的IO模型有哪些？

主要有四种经典IO模型：

**1. 阻塞IO（Blocking IO）**
- 调用IO操作时，一直阻塞等待直到操作完成
- 应用线程被挂起，什么都干不了
- 优点：简单，缺点：一个线程只能处理一个连接，并发低

**2. 非阻塞IO（Non-blocking IO）**
- IO操作如果没准备好，立刻返回错误，不会阻塞线程
- 需要循环反复询问内核是否准备好（轮询）
- CPU利用率高，但用户态忙轮询浪费CPU

**3. 多路复用IO（IO Multiplexing）**
- 通过`select/poll/epoll/kqueue` 一个线程同时监听多个socket
- 哪个socket准备好了，就处理哪个socket
- 主流方案，比如NIO、Netty、Redis都用这个
- 优点：单线程可以处理多个连接，资源消耗少

**4. 异步IO（Asynchronous IO）**
- 用户线程发起IO请求后立刻返回，内核完成IO后通知用户线程
- 真正的异步，整个过程用户线程不阻塞
- 比如Linux的`aio`，Windows的`IOCP`

**总结对比**：
- 阻塞IO → 等，全程等
- 非阻塞IO → 问，一遍遍问
- 多路复用 → 监听，哪个好处理哪个
- 异步IO → 内核做完通知我
