# 数据仓库

## 5：数仓分层了解吗 

> ODS -> DWD -> DIM -> DWS -> ADS 从下往上 
标准五层架构，从上到下（从源到应用）依次是：
1. **操作数据层（ODS - Operational Data Store）**：存放从源系统抽取的原始数据，结构上与源系统保持一致，基本不做修改，是数据仓库的数据准备区。
2. **明细数据层（DWD - Data Warehouse Detail）**：对 ODS 层数据进行清洗、过滤、维度退化，按照业务过程建模，保存各业务过程最小粒度的明细事实数据。
3. **维度层（DIM - Dimension）**：基于维度建模理论构建，存放一致性维度表，保存维度对象的描述属性信息，供事实表关联使用。
4. **汇总数据层（DWS - Data Warehouse Service / Summary）**：基于上层应用需求，以主题对象为建模驱动，对 DWD 明细数据按照不同维度进行聚合汇总，生成公共统计粒度的汇总表，供上层ADS复用。
5. **应用数据层（ADS - Application Data Service）**：存放各业务方最终需要的统计指标结果，直接供外部应用（报表、大屏、接口）查询使用。
**数据流向**：ODS（原始数据）→ DWD（清洗明细）→ DIM（维度关联）→ DWS（公共汇总）→ ADS（应用指标）

## 2：数仓分层的目的（为什么要分层）

1. **将复杂的需求简单化。**可以将复杂的问题分解为多个步骤来完成，每一层只处理单一的步骤，比较容易和理解
2. **提高数据的复用性【以空间换取时间】。**通过预计算中间层（如DWS公共汇总表）实现高频查询秒级响应。
3. **减少重复开发。**规范数据分层，开发一些通用的中间层数据，能够减少极大的重复计算。 
4. **清晰数据结构。**每一层都有它的作用域，这样在使用表的时候能更方便地定位和理解。
5. **数据血缘追踪。**给业务呈现的业务表的来源有很多，一旦有一张来源表出问题了，就能够快速准确地定位到问题，并清楚它的危害范围。 
6. **数据故障隔离。**当数据出现问题之后，通过分层管理可以锁定数据位置，从而快速从有问题的步骤开始修复，也避免了对其他层数据表的污染
7. **架构弹性，屏蔽业务影响。**当上游业务变更时，只需调整对应ODS层接入逻辑，上层通过"逻辑视图"保持兼容，不必改一次业务就需要重新接入数据。而且便于维护数据的准确性。

## 5：数据仓库建模的方法有哪些?优缺点？

1. ER模型：符合3范式的，目的就是整合数据，将各个系统中的数据以整个企业角度按主题进行分类，但是不能直接用于分析决策。ER模型冗余更少，但是在大规模数据跨表分析中，会造成多表关联，这会大大降低执行效率。【ER模型的优化】Data Vault模型和Anchor模型
2. 维度模型：出发点就是**分析决策，为分析需求服务**，而现在多数的数仓的搭建都是基于维度模型进行搭建的
   1. **【维度模型的实例】**
      1. 星型模型：这是最常用的维度建模的方式，核心就是以事实表为中心，所有的维度表直接连接在事实表上，就和星星一样，所以叫做星型模型
      2. 雪花模型：这是星型模型衍生而来的，相对于它不同的是维度表可以再连接其他 维度表，有点类似于3NF模型
      3. 星座模型：这也是星型模型的衍生而来的，相对于它不同的是基于多张事实表并且共享维度信息，也就是维度表之间可能会连在一起。*一般来说，企业中不会只有一张事实表，所以大多数情况下都是星座模型进行维度建模的。*

## 2：维度建模中表的类型

1. 维度表：**一张维度表就表示对一个对象的一些描述信息**。每个维度表都包含单一的主键列，和一些对该主键的描述信息，通常维度表会很宽。比如乘客信息表，司机信息表，城市首都表
2. 事实表：**一张事实表就表示对业务过程的描述，比如播单，下单，支付。**每个事实表都包含若干维度外键，若干退化维度（维度属性存储到事实表中，减少关联），和数值型的度量值，通常事实表会比较大。

## 2：事实表的类型和设计过程

- 事实表有三种类型：事务事实表、周期快照事实表、累积快照事实表
- 一共有五步，分别是**选择业务过程，声明粒度，确定维度，确定事实，冗余维度**  

1. **选择业务过程。**就是对业务的整个生命周期进行分析，然后选择与需求有关的**业务过程（业务过程可以概括为一个个不可拆分的行为事件）**
   1. 比如滴滴呼单的整个流程：乘客呼单，平台播单，司机抢单，司机接驾，完成订单，（买家下单，买家付款，卖家发货，买家确认收货）。**选择业务过程**就是根据我们的需求在其中去选择对应的过程（每一步骤即整个流程中的一个过程）
2. **声明粒度。**粒度就是用于确定事实表中一行所表示的业务的细节层次。通常在设计事实表的时候，粒度定义的越细越好。**（精确定义表每行数据表示什么，尽可能选最细粒度）**
   1. 比如订单明细表的粒度就是 **订单级别**
3. **确定维度。**确认清楚业务过程所处环境的维度信息。
   1. 比如订单明细表中出发城市，到达城市，产品线，司机，订单状态等（支付事实表，买家，买家，商品，收 货人信息，业务类型，订单时间）等多个维度，可以理解为是**看待事实的角度**
4. **确认事实。**也就是度量值，主要包括 订单量，订单金额，接驾时长，接驾距离**（确定与事实表相关的维度信息有哪些）(确定业务的度量值)**
   1. 事实就是分析业务过程中的度量值，比如订单金额，订单次数等
5. **冗余维度(退化维度)。**在事实表中冗余一些下游用户需要使用的常用维度，减少多表之间的关联
   1. 交易域构建维度模型（事实表）
      1. 主要选择了和订单有关的业务过程：冒泡 发单 播单 完单 支付；其中发单和完单和支付放在一张事实表中，构成多事务事实表  （流量域：冒泡，打开，登陆，浏览， 点击，退出）
   2. 比如城市的经纬度信息，后续可能分析某一位置的订单量更多，就需要这一维度

## 1：ODS层存在的意义

1. **数据集成****与清洗** ODS层提供了一个统一的数据集，包含了多个数据源中的所有数据，解决了数据冲突问题。
2. **缓冲作用** ODS层作为数据仓库的前置层，起到数据缓冲的作用，减少对上层数据仓库的直接冲击，确保数据仓库的稳定性和性能。
3. **业务需求驱动** ODS层的数据结构与业务系统相对接近，能够更好地满足业务部门对数据灵活的需求，提升数据应用的效率

## 5：讲一下数据倾斜是什么以及发生的原因

- **数据倾斜的定义及影响**

数据倾斜是**分布式计算中数据分布不均匀**的现象，导致某些节点（Task/Partition）处理的数据量远高于其他节点。系统表现为大部分任务都很快完成，只有一个或者少数几个任务执行的很，从而造成多数节点空闲，少数节点长时间运行（长尾任务）**【性能瓶颈】**，集群利用率低【**资源浪费**】，甚至出现最终节点OOM（内存溢出）或超时，导致任务执行失败【**任务失败**】。

- **为什么会发生数据倾斜**
  - **map task数据倾斜**
    - 不可切分的压缩算法（如使用了gzip,zip等进行压缩的数据文件）
    - 数据文件大小差异大（如HDFS中某些文件特大）
    - 数据存在较多空值
  - **reduce task数据倾斜**
    - **shuffle + Join/GroupBy操作后key分布不均【主要原因】（这两个是一个整体）**
      1. **信息获取不全时被填充默认值。**
      2. **该业务本身存在热点（如特价或热销商品购买量大增）**
      3. **存在由****同一IP****的爬虫等产生的恶意数据**

## **5：如何判断并解决数据倾斜问题**

- **如何判断数据倾斜**
  - **监控任务运行状态** **长尾Task**：通过计算框架的DashBoard来分析Task的处理时间 **数据量差异**：检查各Task的Input Size/Records，倾斜节点的数据量可能是平均值的数十倍。
  - **日志分析** **GC频繁或OOM错误**：倾斜节点因处理数据量过大频繁垃圾回收或内存溢出。 **超时重试**：Task因处理过慢被标记为失败并重试。
  - **资源监控** **CPU/内存不均衡**：通过YARN或K8s监控，发现少数节点资源利用率接近100%，其他节点闲置。
  - **数据采样分析**
  - **分区大小检查**
  - **业务经验预判** 已知业务中存在**热点实体**（如电商大促中的爆款商品、社交网络的明星用户）。 数据源来自**特定分区**（如某天的日志量激增）。
- **数据倾斜怎么解决【Map】**
  - **选择可切分的压缩算法（如lzo，bzip2，snappy等）**【注意：lzo压缩文件是可切片的，但是它的可切片特性依赖于其索引，所以需要手动为lzo压缩文件创建索引】
  - **让每个数据文件大小基本一致**
  - **过滤空值，无效数据等异常数据**
  - **通过Map端Join（Broadcast Join）消除shuffle：**适用于**大表Join小表**【小表数据量远小于大表，小表可放入内存，小表满足广播条件（否则如果广播的数据很大，可能内存溢出）】（Spark中，将小表数据**封装为广播变量，**【数据压缩、高效的通信框架Netty、BT协议】广播给所有的executor节点， 然后利用**map算子在本地进行join**即可，无需将数据按Key分发到同一节点，彻底消除Shuffle阶段
- **数据倾斜怎么解决【Reduce】**
  - 增大reduce并行度：`hashCode(key)%reduce个数`。该方法实现简单但不一定有效
  - 加盐。给key添加随机数强行打散数据
    1. **通用。**在map阶段先将key加上前缀或者后缀，shuffle之后， 先对打上随机前后缀的key进行局部聚合，再将各个key 的随机前后缀去掉后进行全局聚合，就得到了最终的结果 
    2. 适用于由**少数 key 的数据量大导致join的时候发生数据倾斜的情况。**可以**为数据量特别大的key增加随机前后缀（如****`A_1`****,** **`A_2`****）**，使得这些 key 分散到不同的task中。再对小表中对应的原始Key（如`A`）**生成所有可能的变体**（如`A_1`, `A_2`），通过笛卡尔积确保Join时Key匹配。最后执行Join并移除前后缀还原Key
    3. 适用于**出现数据倾斜的key较多甚至无法拆分出倾斜数据的情况**。可以选择**大表加盐，小表扩容**【扩容就是将该表和前缀作笛卡尔积】，然后再按照方法2进行处理。

## 1：指标下沉怎么做的

指标下沉是指**将宏观层面**的指标逐步**细化**到**更小的层级**，以便进行更深入的分析。 比如：最近30天销售额，通过指标下沉，能够将其划分为以下几个层级            
生产线：不同产品线的月销售额。            
按地区：不同地区的月销售额。            
按销售平台：不同销售平台的月销售额

拓展其他下沉方式： 
**口径下沉：**指在定义和计算指标时，对数据口径进行进一步的细化和明确。目的是确保数据的**一致性**和**可比性**，以避免不同部门或不同场景下的口径不一致导致的数据混淆或误解。

Flink CDC的底层是怎么样的
CDC（Change Data Capture，变更数据捕获）的底层原理主要基于数据库事务日志的解析，其核心思想是实时捕获数据库中的变更操作（如INSERT/UPDATE/DELETE），并将这些变更以有序事件流的形式输出，供下游系统消费。以下是其技术原理的分步解释：

---
1. 数据库事务日志（Transaction Log）
- 作用：所有数据库（如MySQL、PostgreSQL、Oracle等）都会维护事务日志（如MySQL的binlog、PostgreSQL的WAL），记录每一次数据变更的详细信息。
- 日志内容：
  - 操作类型：INSERT、UPDATE、DELETE
  - 表名、行ID：标识变更的具体数据位置
  - 变更前后的值：记录字段修改前后的状态
  - 事务时间戳：确保事件的有序性

---
2. 日志解析与事件生成
- CDC工具（如Debezium、Canal、Maxwell）会：
  1. 监听日志：持续监控数据库的日志写入位置（如MySQL的binlog offset）。
  2. 解析日志：将二进制日志转换为结构化事件，例如：
json{
  "table": "users",
  "op": "UPDATE",
  "before": {"id": 1, "name": "Alice"},
  "after": {"id": 1, "name": "Alice Smith"},
  "timestamp": "2023-10-01T12:00:00Z"
}
  1. 事件排序：基于事务时间戳（LSN, Log Sequence Number）保证事件顺序。

---
3. 数据传输与消费
- 输出目标：
  - 消息队列（如Kafka、RabbitMQ）：供实时流处理引擎消费。
  - 数据仓库/湖（如Snowflake、S3）：用于离线分析。
  - 数据库（如PostgreSQL、Elasticsearch）：实现跨系统数据同步。
- 消费模式：
  - At-Least-Once：确保事件不丢失（可能重复）。
  - Exactly-Once：通过事务或幂等设计实现精确一次消费。

---
4. 关键挑战与优化
- 性能开销：
  - 日志解析可能增加数据库负载，需通过批量读取、并行解析优化。
- 一致性保证：
  - 需处理分布式事务（如跨分片更新）和日志回溯（checkpoint机制）。
